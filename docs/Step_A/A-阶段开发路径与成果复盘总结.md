# A-阶段开发路径与成果复盘总结

## 一、文档目的

这份文档不是替代 `Step_A` 里的各阶段详解，而是作为一份**阶段复盘索引**，回答下面几个问题：

- `Step_A` 到底是按什么顺序把项目搭起来的
- 每个阶段具体产出了什么
- 这些产出最终落到了哪些源码文件里
- 每一阶段解决了什么问题，又留下了哪些后续演进空间
- 项目结束后，我该如何沿着真实开发路径回看自己的成长过程

这份文档建议在项目收尾、面试准备、技术复盘时反复阅读。

---

## 二、Step_A 的整体开发主线

`Step_A` 的核心不是“堆模块”，而是按一条比较清晰的服务端成长路径推进：

1. 先补并发基础设施：线程池、阻塞队列
2. 再搭高性能网络骨架：muduo 风格 Reactor、`EventLoop`、`Channel`、`TcpConnection`
3. 再定义消息通信能力：协议、编解码、序列化、缓冲区
4. 再补业务服务层：登录、心跳、消息、过载保护
5. 再做整合与测试：`TcpServer`、主程序、配置、客户端、构建系统
6. 最后引入 Redis，让项目从单机内存态走向共享状态架构

换句话说，`Step_A` 实际完成的是一条非常典型的后端基础设施搭建路线：

`并发基础 -> 网络骨架 -> 消息协议 -> 业务服务 -> 工程整合 -> 分布式增强`

这条路线很适合你在项目结束后回顾，因为它天然对应了服务端开发者能力的成长顺序。

## 三、阶段拆解复盘

## 阶段一：线程池与阻塞队列

### 1. 阶段目标

第一阶段要解决的不是网络问题，而是**异步执行基础设施**问题。  
因为后面无论是 Redis 操作、业务任务、定时任务，最终都需要依赖一个稳定的任务执行载体。

### 2. 核心产出

- `src/base/blockingQueue.h`
- `src/base/threadpool.h`
- `src/base/threadpool.cpp`
- `docs/Step_A/A-阶段一：线程池模块的详细设计.md`

### 3. 本阶段实际完成了什么

- 实现线程安全阻塞队列
- 实现固定线程数线程池
- 支持 `submit()` 异步提交任务
- 支持 `future` 获取结果
- 支持基本的优雅关闭与异常隔离

### 4. 这一阶段的意义

这一阶段看起来像“工具层”，实际上很关键。  
它决定了整个项目后续是否有能力：

- 异步执行 Redis 操作
- 将耗时逻辑从 I/O 线程剥离
- 在面试中讲清楚任务调度、生产者消费者模型、线程安全边界

### 5. 本阶段留下的开发痕迹

- 源码痕迹：`src/base/`
- 文档痕迹：线程池详细设计文档
- 架构痕迹：后续 `HeartbeatService` 和 Redis 异步化直接依赖这套线程池

### 6. 复盘要点

- 我为什么没有直接使用第三方线程池
- `BlockingQueue` 为什么要支持阻塞和超时获取
- 当前 `ThreadPool::stop()` 的停止语义是否足够适合 IM 主链路

### 7. 后续遗留问题

- 线程池缺少任务队列长度指标
- 缺少拒绝策略
- 缺少 `drain/discard` 两种停止模式

## 阶段二：muduo 风格网络骨架

### 1. 阶段目标

这一阶段的任务是搭建整个系统的**网络核心骨架**。  
也就是让项目从“有线程池的普通程序”变成“有 Reactor 事件驱动能力的 TCP 服务器”。

### 2. 核心产出

- `src/net/nonCopyable.h`
- `src/net/timestamp.h/.cpp`
- `src/net/currentThread.h/.cpp`
- `src/net/inetAddress.h`
- `src/net/channel.h/.cpp`
- `src/net/epollPoller.h/.cpp`
- `src/net/eventLoop.h/.cpp`
- `src/net/tcpConnection.h/.cpp`
- `docs/Step_A/A-阶段二：muduo架构详解.md`

### 3. 本阶段实际完成了什么

- 建立 `Channel -> Poller -> EventLoop -> TcpConnection` 的事件驱动主链
- 基于 `epoll` 实现 I/O 多路复用
- 使用 `eventfd` 完成线程间唤醒
- 建立 one loop per thread 的基本模型
- 支持 `runInLoop()` / `queueInLoop()` 跨线程投递

### 4. 这一阶段的意义

这是整个项目最重要的架构性阶段。  
如果没有这一步，后面所有协议层、服务层、Redis 集成都没有稳定承载点。

这一阶段真正完成的不是“几个类”，而是：

- 事件驱动架构
- 线程归属模型
- 连接生命周期管理的基础能力

### 5. 本阶段留下的开发痕迹

- 源码痕迹：`src/net/`
- 设计痕迹：大量模块协作图、数据流图、Reactor 图
- 行为痕迹：当前 `EventLoop::runInLoop()` 仍是后续线程边界切换的核心基础

### 6. 复盘要点

- 为什么选择 muduo 风格，而不是直接上 Boost.Asio
- `EventLoop` 为什么必须强绑定线程
- 为什么 `eventfd` 是跨线程唤醒的关键
- `TcpConnection` 为什么必须独立管理输入输出缓冲区

### 7. 后续遗留问题

- 当前还没有系统化区分 LT/ET 的配置策略
- 对慢连接、高水位、背压的治理还不够完整
- 网络层虽然成型，但尚未完全服务于“可靠消息语义”

## 阶段三：协议、编解码与消息通信工具

### 1. 阶段目标

网络层能收发字节流之后，下一步必须解决的是：

- 如何定义应用层协议
- 如何解决粘包、半包
- 如何把业务结构体安全地变成网络可传输数据

### 2. 核心产出

- `src/codec/protocol.h/.cpp`
- `src/codec/codec.h/.cpp`
- `src/codec/messageSerializer.h/.cpp`
- `src/codec/buffer.h/.cpp`
- `docs/Step_A/A-阶段三：服务器消息通信工具详解.md`

### 3. 本阶段实际完成了什么

- 定义固定头协议
- 定义命令字枚举与业务消息结构
- 使用 CRC16 进行数据校验
- 实现状态机式的解包逻辑
- 使用 JSON 完成消息结构体的序列化 / 反序列化
- 建立 `Buffer -> Codec -> Serializer -> 业务消息` 的转换链路

### 4. 这一阶段的意义

这一阶段意味着项目正式拥有了“自定义应用层协议”。  
从这一刻开始，项目不再只是“TCP 套接字练习”，而是具备了：

- 自定义协议设计能力
- 编解码能力
- 粘包半包处理能力

### 5. 本阶段留下的开发痕迹

- 源码痕迹：`src/codec/`
- 协议痕迹：当前项目中仍然沿用了 16 字节头 + JSON 负载的主思路
- 结构痕迹：`protocol::Command`、`Codec::pack()`、`serializer::Serialize/Deserialize()`

### 6. 复盘要点

- 为什么先做“固定头 + 可变体”协议
- 为什么选择 JSON 作为 Step_A 阶段的序列化方案
- 为什么要在协议层做 CRC16，而不是只依赖 TCP
- 为什么 `Buffer` 采用 muduo 风格双指针模型

### 7. 后续遗留问题

- 协议头字段还不够丰富
- 缺少版本号、错误码、扩展位
- JSON 更适合作为过渡期方案，后续应演进到 Protobuf
- 当前协议对 ACK、重试、补拉的支撑还不足

## 阶段四：业务服务层

### 1. 阶段目标

协议层打通后，项目需要从“能收发消息”升级到“有 IM 业务语义”。  
也就是补上：

- 登录鉴权
- 心跳保活
- 点对点消息
- 广播消息
- 过载保护

### 2. 核心产出

- `src/service/authService.h/.cpp`
- `src/service/heartbeatService.h/.cpp`
- `src/service/messageService.h/.cpp`
- `src/service/overloadProtectService.h/.cpp`
- `docs/Step_A/A-阶段四：服务器聊天业务详解.md`

### 3. 本阶段实际完成了什么

- 建立用户上下文与会话管理
- 建立登录 / 登出流程
- 建立心跳检测和超时踢出
- 建立用户连接注册与消息路由
- 建立广播消息路径
- 建立连接数与 QPS 的过载保护机制

### 4. 这一阶段的意义

这是项目第一次真正接近“IM 服务器”的阶段。  
因为从这个阶段开始，系统已经不只是框架骨架，而是有了：

- 用户
- 会话
- 在线状态
- 消息投递
- 保护机制

### 5. 本阶段留下的开发痕迹

- 源码痕迹：`src/service/`
- 行为痕迹：`AuthService + MessageService + HeartbeatService` 的协同模式
- 配置痕迹：心跳超时、连接数、QPS 等配置参数开始具备实际含义

### 6. 复盘要点

- 为什么服务层要按职责拆为多个 Service
- 为什么 `MessageService` 依赖 `AuthService`
- 为什么心跳和业务消息不能混为同一类逻辑
- 过载保护为什么必须作为独立服务，而不是散落在各业务点里

### 7. 后续遗留问题

- 当前消息路由主要还是在线直发
- 还没有 ACK、重试、消息状态机
- 没有离线消息持久化
- `MessageService` 还不是完整意义上的消息投递中心

## 六、Step_A 的关键设计取舍

这部分是后续复盘时最值得反复看的，因为它记录的是“为什么这样做”，而不是“做了什么”。

### 1. 先做线程池，再做网络层

原因：

- 后续 Redis 异步化、业务异步化都需要任务执行基础设施
- 并发模型必须先想清楚，网络层才能有明确边界

### 2. 先用 JSON，再考虑 Protobuf

原因：

- Step_A 的重点是先跑通整个消息链路
- JSON 便于观察、调试、抓包和快速迭代
- 等主链路稳定后，再在 Step_B 升级 Protobuf 更合理

### 3. 先做在线直发，再考虑可靠消息

原因：

- 项目必须先具备最基本的可交互能力
- 可靠消息、ACK、补拉属于更高一层的系统能力

### 4. 先引入 Redis，再引入 MySQL

原因：

- Redis 更适合作为共享状态存储切入点
- 它能先把“单机状态外置”这个关键动作做完
- MySQL 的引入更适合在 Step_B 作为可靠持久化主线推进

## 七、Step_A 的不足与自然演进方向

如果项目到此为止，Step_A 已经足够作为一个很完整的阶段成果。  
但如果目标是继续升级为工业级 IM，Step_A 仍然留下了明确的升级空间。

### 1. 协议层不足

- 缺少版本号
- 缺少错误码体系
- 缺少扩展位
- 缺少 ACK / 补拉 / 序列号语义

### 2. 消息层不足

- 没有离线消息落盘
- 没有消息状态机
- 没有重试机制
- 没有去重窗口

### 3. 数据层不足

- Redis 已有，但 MySQL 还未接入
- 当前没有可靠持久化主链路

### 4. 稳定性与工程化不足

- 缺少系统化单元测试
- 缺少 Prometheus 指标
- 缺少结构化 Trace
- 缺少更细粒度的限流、背压、慢连接治理

这些不足并不是 Step_A 的失败，恰恰说明 Step_A 的定位是正确的：  
它完成的是“从零到一”的骨架搭建，而不是“一步到位工业级终局”。

- 为什么需要心跳
- 为什么需要 Redis

### 第四遍：按“Step_A -> Step_B”回看

- 哪些设计已经足够稳定，可以继续复用
- 哪些地方只是过渡方案，需要在 Step_B 推翻或升级

---

## 九、给未来自己的总结

`Step_A` 最有价值的地方，不只是你写出了一个能运行的服务器，而是你沿着一条真实的服务端开发路径，把一个项目一点点搭了起来。

你在这个阶段真正收获的是：

- 知道一个高性能服务端项目应该先搭什么
- 知道网络层、协议层、业务层、工具层如何逐步成型
- 知道哪些能力该先做，哪些能力应该留到下一阶段

如果未来项目结束、简历写完、面试结束，再回头看 `Step_A`，最值得记住的不是某个函数实现，而是这条开发路线本身：

**先把地基打稳，再把链路跑通，再把业务做出来，最后再去思考分布式和工业级升级。**

这就是 `Step_A` 最核心的学习成果。
