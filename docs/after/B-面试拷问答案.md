### 1. unique_ptr和shared_ptr的核心区别？循环引用如何解决？RAII还能管理哪些资源？

**unique_ptr和shared_ptr的核心区别：**

- **所有权**：
  - `unique_ptr`：独占所有权，同一时间只能有一个unique_ptr指向同一个对象
  - `shared_ptr`：共享所有权，多个shared_ptr可以指向同一个对象
- **性能开销**：
  - `unique_ptr`：几乎无额外开销（仅保存原始指针）
  - `shared_ptr`：需要维护引用计数，有一定开销
- **支持操作**：
  - `unique_ptr`：支持移动语义，不支持拷贝
  - `shared_ptr`：既支持移动，也支持拷贝
- **适用场景**：
  - `unique_ptr`：资源只需要一个所有者
  - `shared_ptr`：资源需要多个所有者共享

**shared_ptr循环引用问题：**

- **产生原因**：两个对象互相持有对方的shared_ptr，导致引用计数永远无法归零，内存泄漏
  ```cpp
  class A { shared_ptr<B> b; };
  class B { shared_ptr<A> a; };
  auto pa = make_shared<A>();
  auto pb = make_shared<B>();
  pa->b = pb;
  pb->a = pa;  // 循环引用！
  ```
- **解决方法**：使用 `weak_ptr`打破循环
  - `weak_ptr`：不增加引用计数，只作为观察者
  - 将其中一方的shared_ptr改为weak_ptr

**RAII除了内存，还能管理的资源：**

- 文件描述符（fd）
- 互斥锁（mutex）
- 网络连接（socket）
- 数据库连接
- 线程句柄
- 图形资源（纹理、缓冲区等）

### 2. 游戏实时排行榜用什么数据结构？为什么？Redis的zset底层是怎么实现的？

**实时排行榜数据结构选择：**

- **跳表（Skip List）**：
  - 优点：O(logN)插入、删除、查询，实现相对简单，支持范围查询
  - 适用：单机排行榜
- **平衡二叉树（红黑树）**：
  - 优点：O(logN)操作
  - 缺点：实现复杂，范围查询不如跳表方便
- **Redis zset**：
  - 优点：支持大规模并发，分布式友好，开箱即用
  - 适用：生产环境、分布式场景

**推荐方案**：

- 单机：跳表
- 生产环境：Redis zset

**Redis zset底层实现：**
Redis zset使用**跳表（Skip List）** + **哈希表（Hash Table）** 双结构：

- **跳表**：维护有序性，支持按分数范围查询
- **哈希表**：O(1)查找成员对应的分数
- **跳表结构**：
  - 多层链表结构，最下层是完整有序链表
  - 上层是下层的"快速通道"，随机生成
  - 查找时从顶层开始，快速定位
- **时间复杂度**：O(logN) 插入、删除、查询

### 3. TCP和UDP的本质区别？枪战游戏场景选择？epoll LT和ET差异？IM项目用哪种？

**TCP和UDP的本质区别：**

| 特性     | TCP                      | UDP                  |
| -------- | ------------------------ | -------------------- |
| 连接性   | 面向连接（三次握手）     | 无连接               |
| 可靠性   | 可靠（重传、确认、有序） | 不可靠               |
| 传输方式 | 字节流                   | 数据报               |
| 头部开销 | 大（20-60字节）          | 小（8字节）          |
| 拥塞控制 | 有                       | 无                   |
| 适用场景 | 要求可靠传输             | 要求低延迟、允许丢包 |

**枪战类游戏场景选择：**

- **必须用TCP**：
  - 登录、鉴权、充值
  - 聊天消息
  - 玩家数据保存
  - 匹配结果、结算数据
- **可以用UDP**：
  - 玩家位置同步
  - 射击、技能释放
  - 移动状态更新
  - （这些场景允许少量丢包，重传反而影响体验）

**epoll LT和ET核心差异：**

- **LT（水平触发，Level Triggered）**：
  - 只要缓冲区有数据，就会一直通知
  - 支持阻塞和非阻塞IO
  - 编程简单，不容易出错
- **ET（边缘触发，Edge Triggered）**：
  - 只在状态变化时通知一次
  - 必须使用非阻塞IO
  - 性能更高，但编程复杂
  - 需要一次性把数据读完

**IM项目选择：**

- **LT模式**
- **原因**：
  1. 编程简单，代码更稳健
  2. IM消息不追求极致性能，稳定性更重要
  3. 避免ET模式下漏读数据导致的bug

### 4. Linux排查CPU占用过高？查看端口占用？给进程抓包？

**排查CPU占用过高：**

1. **top**：查看哪个进程CPU占用高
   ```bash
   top -Hp <pid>  # 查看进程下的线程
   ```
2. **ps**：查看进程详细信息
   ```bash
   ps -ef | grep <进程名>
   ```
3. **perf**：性能分析工具，定位热点函数
   ```bash
   perf top -p <pid>
   perf record -g -p <pid>  # 记录调用栈
   perf report  # 查看报告
   ```
4. **pstack**：查看进程调用栈
   ```bash
   pstack <pid>
   ```

**查看端口被哪个进程占用：**

```bash
# 方法1：netstat
netstat -tulnp | grep <端口号>

# 方法2：ss（推荐，更现代）
ss -tulnp | grep <端口号>

# 方法3：lsof
lsof -i :<端口号>
```

**给正在运行的进程抓包：**

```bash
# 方法1：tcpdump（最常用）
tcpdump -i any -w capture.pcap port <端口号>
tcpdump -i eth0 host <IP地址> -w capture.pcap

# 方法2：tshark（Wireshark命令行版）
tshark -i any -w capture.pcap -f "port <端口号>"

# 查看抓包文件
tcpdump -r capture.pcap
wireshark capture.pcap  # 用Wireshark图形界面分析
```

### 5. Redis为什么快？RDB和AOF优缺点？Redis宕机怎么办？

**Redis为什么这么快：**

1. **纯内存操作**：所有数据存储在内存，避免磁盘IO
2. **单线程模型**：避免线程切换开销和锁竞争
3. **高效的数据结构**：跳表、压缩列表、整数集合等
4. **非阻塞IO**：使用epoll多路复用
5. **优化的协议**：RESP协议简单高效

**RDB和AOF持久化对比：**

| 特性                 | RDB                          | AOF                       |
| -------------------- | ---------------------------- | ------------------------- |
| **方式**       | 内存快照                     | 追加写命令                |
| **文件大小**   | 小（二进制压缩）             | 大（文本命令）            |
| **恢复速度**   | 快                           | 慢                        |
| **数据安全性** | 可能丢数据（取决于save策略） | 更安全（取决于fsync策略） |
| **适用场景**   | 备份、冷备                   | 主从复制、数据持久化      |

**RDB优缺点：**

- 优点：文件小、恢复快、适合备份
- 缺点：可能丢失最近的数据

**AOF优缺点：**

- 优点：数据更安全、可读性好
- 缺点：文件大、恢复慢

**Redis节点宕机解决方案：**

1. **主从复制（Master-Slave）**：
   - 配置一主多从
   - Master宕机后，手动或自动（Sentinel）将Slave提升为Master
2. **Redis Sentinel（哨兵）**：
   - 自动监控Master状态
   - Master故障时自动故障转移
   - 提供配置中心服务
3. **Redis Cluster（集群）**：
   - 数据分片存储
   - 每个分片一主一从
   - 自动故障转移
4. **数据持久化**：
   - 开启RDB+AOF混合持久化
   - 定期备份RDB文件到异地
5. **业务层处理**：
   - 服务降级（暂时不可用）
   - 数据回滚到最近快照

## 二、核心项目深挖（占面试60%以上时间，最能拉开差距）

### （一）Linux高性能即时通讯服务器

#### 1. 3分钟介绍IM服务器整体架构、核心模块、用户A发消息给B的完整流程

**整体架构（3分钟版本）：**

**架构概述：**
采用**单Reactor多线程**架构，基于epoll IO多路复用，实现One Loop Per Thread模型。

**核心模块划分：**

1. **EventLoop事件循环**：核心驱动，管理IO事件和定时器
2. **Channel事件分发器**：封装fd和事件回调
3. **TcpServer连接管理**：accept新连接、管理连接生命周期
4. **Buffer缓冲区**：muduo风格，解决TCP粘包
5. **Codec编解码器**：解析自定义协议
6. **AuthService鉴权服务**：用户登录、token验证
7. **MessageService消息服务**：消息转发、广播
8. **HeartbeatService心跳服务**：保活、超时踢人
9. **OverloadProtectService过载保护**：连接数和QPS限流
10. **Redis分布式存储**：会话和在线状态

**用户A给用户B发消息完整流程：**

1. **客户端A**：构造P2P消息 → 编码 → 发送TCP
2. **服务器接收**：
   - epoll_wait检测到可读事件
   - Channel回调读函数
   - 从socket读取数据到Buffer
   - Codec解析协议（处理粘包）
   - 提取完整消息、反序列化
3. **业务处理**：
   - AuthService验证A已登录
   - MessageService查B的在线状态
   - 情况1：B在线 → 通过B的连接发送
   - 情况2：B不在线 → 返回A发送失败
4. **客户端B**：
   收到消息 →
   Codec解析协议（处理粘包）→
   提取完整消息、反序列化→
   显示给用户

#### 2. Reactor模型+One Loop Per Thread原理图？为什么不选单线程或多进程？瓶颈和优化？

**Reactor模式+One Loop Per Thread工作原理图：**

```
        ┌────────────────────────────────────────────────────────┐
        │                     主线程 Main Thread                 │
        │  ┌─────────────────────────────────────────────────┐   │
        │  │           EventLoop (Main Reactor)              │   │
        │  │  ┌──────────────┐                               │   │
        │  │  │   Acceptor   │  accept() 新连接              │   │
        │  │  └──────┬───────┘                               │   │
        │  └─────────┼───────────────────────────────────────┘   │
        └────────────┼───────────────────────────────────────────┘
                     │
                     ├─────────────────────┬─────────────────────┐
                     │                     │                     │
        ┌────────────▼─────────-─┐ ┌──────▼──────────-┐ ┌───────▼─────────--─┐
        │   子线程 1             │ │   子线程 2       │ │   子线程 3         │
        │  ┌───────────────────┐ │ │ ┌─────────────-─┐│ │  ┌─────────────--─┐│
        │  │  EventLoop        │ │ │ │  EventLoop    ││ │  │  EventLoop     ││
        │  │ ┌─────-─┐ ┌──────┐│ │ │ |┌─────-─┐ ┌───┐|│ │  │ ┌────-──┐ ┌───┐│|
        │  │ │Channel│ │Timer ││ │ │ |│Channel│ │...│|│ │  │ │Channel│ │...││|
        │  │ └─────-─┘ └──────┘│ │ │ |└─────-─┘ └───┘|│ │  │ └────-──┘ └───┘│|
        │  │ ┌─────-─┐ ┌──────┐│ │ │ |┌─────-─┐ ┌───┐|│ │  │ ┌─────-─┐ ┌───┐│|
        │  │ │Channel│ │...   ││ │ │ |│Channel│ │...│|│ │  │ │Channel│ │...││|
        │  │ └────-──┘ └──────┘│ │ │ |└─────-─┘ └───┘|│ │  │ └──────-┘ └───┘│|
        │  └───────────────────┘ │ │ └──────────────-┘│ │  └─────────────--─┘│
        └────────────────────────┘ └──────────────────┘ └───────────────────-┘
                    │                          │                         │
              ┌─────▼────--─┐             ┌────▼─────-┐            ┌────▼─────┐
              │  连接池     │             │  业务线程 │            │  线程池  │
              │ (BlockQueue)│             │  池       │            │          │
              └───────────--┘             └─────────-─┘            └──────────┘
```

**为什么不选单线程Reactor：**

- 单线程无法充分利用多核CPU
- 业务逻辑耗时会阻塞IO处理
- 只适合连接数少、业务简单的场景

**为什么不选多进程模型：**

- 进程间通信（IPC）开销大
- 进程切换开销比线程大
- 共享数据麻烦
- 内存占用高

**瓶颈场景：**

1. **IO瓶颈**：单线程accept成为瓶颈
2. **业务瓶颈**：耗时业务占用IO线程
3. **锁竞争**：共享数据访问频繁
4. **网络带宽**：单机带宽上限

**优化方案：**

1. **多Reactor多线程**：主线程accept，子线程处理连接
2. **线程池**：耗时业务丢给线程池
3. **无锁设计**：尽可能用atomic、减少锁粒度
4. **读写分离**：用shared_mutex
5. **连接分散**：连接均匀分配到各个子Reactor

#### 3. 10000+并发怎么测的？用了什么工具？最大瓶颈？怎么定位解决？

**压测工具：**

- **wrk**：HTTP压测（如果有HTTP接口）
- **tcpcopy**：TCP流量复制
- **自定义压测客户端**：C++/Go编写，模拟大量连接

**10000+并发连接测试步骤：**

1. **修改系统限制**：
   ```bash

   echo "fs.file-max = 1000000" >> /etc/sysctl.conf

   - 每个连接定期发送心跳

   - CPU、内存、网络IO
   - 连接数、QPS、响应延迟

- **瓶颈1：文件描述符限制**
  - 现象：无法建立更多连接

  - 现象：在线用户表访问频繁，锁等待高
  - 解决：用读写锁、或分片（多个map）

**定位问题的方法：**

1. **top/htop**：看CPU使用情况
2. **perf**：看热点函数
3. **netstat/ss**：看连接状态
4. **日志**：添加关键路径日志

#### 4. 16字节协议头字段含义？为什么选固定头部？半包、粘包、异常包处理？

**16字节协议头结构：**

```
| 字段         | 长度 | 说明                                    |
|--------------|------|-----------------------------------------|
| 总长度       | 4B   | 整个消息长度（头16B + 体长度）         |
| 命令号       | 2B   | 标识消息类型（登录、心跳、消息等）      |
| 序列号       | 4B   | 请求响应配对、去重、排序                |
| CRC16        | 2B   | 整个消息的CRC校验，防止篡改             |
| 消息体长度   | 4B   | JSON消息体的长度（冗余，双重保险）      |
```

**为什么选择固定头部：**

- **对比分隔符**：
  - 分隔符问题：不能出现在消息体中、不支持二进制数据
  - 固定头：支持二进制、无此限制
- **对比变长头部**：
  - 变长头：解析复杂、需要状态机
  - 固定头：解析简单、固定16字节一读
- **本项目选择**：16字节固定头
  - 优点：简单、可靠、易调试
  - 缺点：头部有一定冗余（但可以接受）

**半包、粘包、异常包处理：**

**半包处理：**

- Buffer缓存未收齐的数据
- 等待下一次read事件
- 继续累积，直到凑齐完整包
- 状态机：
  - 状态1：等待协议头（需要16字节）
  - 状态2：等待消息体（需要 总长度-16 字节）

**粘包处理：**

- 读取数据到Buffer
- 循环解析：
  1. 检查是否够16字节头
  2. 解析头，获取总长度
  3. 检查Buffer可读数据是否≥总长度
  4. 够则取出完整消息，更新readerIndex
  5. 不够则break，等待下次
- 直到Buffer中不够一个完整包

**异常包处理：**

- **CRC校验失败**：
  - 丢弃整个Buffer
  - 记录警告日志
  - 可选：关闭连接
- **长度异常**（如总长度太小或太大）：
  - 丢弃Buffer
  - 记录日志
  - 关闭连接
- **命令号无效**：
  - 记录日志
  - 返回错误响应
  - 不关闭连接

#### 5. IM扩展成多节点集群，消息怎么路由？跨节点消息怎么转发？

**多节点集群架构：**

```
                    ┌──────────────┐
                    │   客户端     │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  负载均衡器   │
                    │  (Nginx/LVS) │
                    └──────┬───────┘
           ┌───────────────┼───────────────┐
           │               │               │
    ┌──────▼──────┐ ┌─────▼──────┐ ┌─────▼──────┐
    │  IM节点1    │ │  IM节点2    │ │  IM节点3    │
    └──────┬──────┘ └─────┬──────┘ └─────┬──────┘
           │               │               │
           └───────────────┼───────────────┘
                           │
                    ┌──────▼───────┐
                    │  Redis集群    │  ← 用户路由表
                    │  (userId→节点)│
                    └──────────────┘
                           │
                    ┌──────▼───────┐
                    │  消息队列     │  ← 跨节点消息转发
                    │  (Kafka/RMQ) │
                    └──────────────┘
```

**消息路由方案：**

1. **Redis存储路由表**：
   - Key: `user:route:{userId}`
   - Value: 节点ID/IP
   - 登录时写入，登出时删除
2. **消息转发流程**：
   - 用户A在节点1，用户B在节点2
   - A给B发消息：
     1. 节点1查Redis：B在节点2
     2. 情况1：本地节点 → 直接发送
     3. 情况2：其他节点 → 转发到节点2
3. **跨节点消息转发**：
   - 方案A：节点间直接建立TCP连接
   - 方案B：通过消息队列（Kafka/RabbitMQ）
   - 方案C：通过Redis Pub/Sub

**具体实现：**

```
用户A(节点1) → 节点1查Redis → B在节点2
                  ↓
            节点1发消息到MQ
                  ↓
            节点2订阅MQ收到消息
                  ↓
            节点2查本地连接表
                  ↓
            发送给用户B

#### 6. 30秒心跳、10秒检查间隔怎么确定？连丢2个心跳包会断开吗？过载保护怎么实现？过载优先拒绝什么请求？

**30秒心跳、10秒检查间隔的确定：**

- **心跳周期30秒**：
  - 太短：流量大、服务器压力大
  - 太长：死连接检测不及时
  - 30秒：平衡，业界常用值
- **检查间隔10秒**：
  - 比心跳周期短，保证及时检测
  - 10秒：适中
- **超时阈值90秒**（3个心跳周期）：
  - 允许丢1-2个心跳包（网络波动）
  - 90秒无心跳才认为连接死了

**连丢2个心跳包会直接断开吗？**

- **不会**
- **原因**：
  1. 网络可能暂时波动
  2. 客户端可能临时卡顿
  3. 给3次机会（90秒）更稳健
- **策略**：
  - 丢1个：记录，继续观察
  - 丢2个：记录警告，继续观察
  - 丢3个（90秒无心跳）：踢除连接

**过载保护实现：**

1. **连接数限制**：
   - atomic计数器
   - 超过阈值（如10000），新连接直接close
2. **QPS限制**：
   - 滑动窗口或令牌桶
   - 超过阈值（如10000 QPS），拒绝请求
3. **实现代码（简化）**：
   ```cpp
   class OverloadProtectService {
       std::atomic<uint64_t> currentConnCount_{0};
       std::atomic<uint64_t> qpsCounter_{0};
       const uint64_t maxConnCount_ = 10000;
       const uint64_t maxQps_ = 10000;
   public:
       bool canAcceptNewConn() {
           return currentConnCount_ < maxConnCount_;
       }
       bool canProcessRequest() {
           return qpsCounter_ < maxQps_;
       }
   };
   ```

**过载时优先拒绝的请求：**

1. **新连接**（最优先）：保护已在线用户
2. **非核心请求**：如获取历史消息、用户资料
3. **广播消息**：开销大，优先限流
4. **保留**：登录、心跳、私聊等核心功能

#### 7. 开发时遇到最难排查的bug是什么？怎么定位和解决的？

**最难排查的bug：偶发死锁**

**问题现象：**

- 服务器运行几小时后突然卡死
- CPU占用很低（都在等锁）
- 新连接无法建立，已有连接无响应
- 偶发，难以复现

**定位过程：**

1. **gdb attach 到进程**：
   ```bash
   gdb -p <pid>
   thread apply all bt  # 查看所有线程堆栈
   ```
2. **发现**：
   - 线程1：持有锁A，等待锁B
   - 线程2：持有锁B，等待锁A
   - 典型死锁！
3. **查代码**：
   ```cpp
   // 线程1
   void func1() {
       std::lock_guard<std::mutex> lockA(mutexA_);
       // ...
       std::lock_guard<std::mutex> lockB(mutexB_);  // 死锁！
   }
   // 线程2
   void func2() {
       std::lock_guard<std::mutex> lockB(mutexB_);
       // ...
       std::lock_guard<std::mutex> lockA(mutexA_);  // 死锁！
   }
   ```

**解决方案：**

1. **统一加锁顺序**：所有地方先锁A再锁B
2. **或用std::lock**：同时锁多个，避免死锁
   ```cpp
   std::lock(mutexA_, mutexB_);
   std::lock_guard<std::mutex> lockA(mutexA_, std::adopt_lock);
   std::lock_guard<std::mutex> lockB(mutexB_, std::adopt_lock);
   ```
3. **缩小锁粒度**：减少锁的持有时间

**经验总结：**

- 加锁顺序一定要统一
- 用工具（tsan、helgrind）提前检测
- 日志要记录加锁、解锁点

### （二）MiniRime轻量级桌面输入法

#### 1. WH_KEYBOARD_LL低级键盘钩子工作原理？和普通钩子对比优缺点？兼容性和性能问题？

**WH_KEYBOARD_LL工作原理：**

- **低级键盘钩子**：安装在系统层面，拦截全局键盘输入
- **工作流程**：
  1. 调用 `SetWindowsHookEx(WH_KEYBOARD_LL, ...)`安装钩子
  2. 用户按下键盘 → 系统先调用钩子回调
  3. 回调可以：放行、修改、拦截按键
  4. 调用 `CallNextHookEx`传给下一个钩子或目标窗口
- **特点**：不需要注入到其他进程，在自己进程内回调

**和普通键盘钩子对比：**

| 特性             | WH_KEYBOARD_LL（低级） | WH_KEYBOARD（普通）  |
| ---------------- | ---------------------- | -------------------- |
| **注入**   | 不需要注入             | 需要注入到目标进程   |
| **权限**   | 需要管理员权限         | 普通权限即可         |
| **范围**   | 全局                   | 全局或特定线程       |
| **稳定性** | 更稳定（不注入）       | 可能导致目标进程崩溃 |
| **兼容性** | 更好                   | 可能被杀毒软件拦截   |
| **性能**   | 稍差（系统调用）       | 稍好                 |

**优缺点：**

- **优点**：
  - 稳定，不会导致其他进程崩溃
  - 容易调试
  - 不易被杀毒软件误报
- **缺点**：
  - 需要管理员权限
  - 比普通钩子稍慢（但可接受）

**开发中遇到的问题：**

1. **兼容性问题**：
   - Windows 7/10/11行为略有差异
   - 解决：测试各版本，条件编译
2. **性能问题**：
   - 钩子回调要快，不能耗时
   - 解决：只做必要处理，耗时操作丢给队列
3. **焦点问题**：
   - 输入法不能抢焦点
   - 解决：悬浮窗设为WS_EX_NOACTIVATE、WS_EX_TOPMOST

#### 2. 为什么选命名管道而不是共享内存、消息队列、Socket？阻塞问题怎么处理？

**为什么选择命名管道：**

| IPC方式            | 优点                       | 缺点           | 本项目是否适合      |
| ------------------ | -------------------------- | -------------- | ------------------- |
| **命名管道** | 简单、可靠、流式、类似文件 | 单机           | ✅ 适合             |
| **共享内存** | 最快                       | 需要同步、复杂 | ❌ 不需要这么高性能 |
| **消息队列** | 消息边界                   | Windows下麻烦  | ❌ 不如管道简单     |
| **Socket**   | 跨机器                     | 本机开销大     | ❌ 不需要跨机器     |

**选择理由：**

1. **简单**：API类似文件读写，容易实现
2. **可靠**：Windows内核保证，不丢包
3. **流式**：适合传输序列化数据（JSON）
4. **够用**：输入法和服务端通信量不大

**命名管道阻塞问题：**

- **问题**：ReadFile/WriteFile默认阻塞
- **处理方式**：
  1. **异步IO（Overlapped）**：
     ```cpp
     CreateFile(..., FILE_FLAG_OVERLAPPED, ...);
     ReadFile(..., &overlapped);
     GetQueuedCompletionStatus(...);  // 或用事件等待
     ```
  2. **非阻塞模式**：
     ```cpp
     DWORD mode = PIPE_NOWAIT;
     SetNamedPipeHandleState(hPipe, &mode, NULL, NULL);
     ```
  3. **本项目方案**：
     - 用异步IO + 完成端口
     - 或单独开线程做阻塞读写

1. 全程用智能指针（unique_ptr、shared_ptr）
2. RAII管理所有资源
3. 避免裸new/delete
4. 异常安全设计

### 3. 游戏同步方案？状态同步vs帧同步优缺点？三角洲行动大战场用哪种？

**主流游戏同步方案：**

| 方案               | 原理                       | 优点                     | 缺点                   | 适用游戏     |
| ------------------ | -------------------------- | ------------------------ | ---------------------- | ------------ |
| **状态同步** | 服务器计算，广播结果       | 易开发、易反作弊、容灾好 | 流量大、延迟高         | MMORPG、MOBA |
| **帧同步**   | 广播输入，各端自己计算     | 流量小、同步精确         | 开发难、容错差、回放难 | FPS、RTS     |
| **混合方案** | 核心帧同步，非核心状态同步 | 兼顾两者                 | 复杂                   | 部分现代游戏 |

**状态同步详细说明：**

- **流程**：
  1. 客户端发送操作（如"移动"）
  2. 服务器接收，验证，计算新状态
  3. 服务器广播新状态给所有客户端
  4. 客户端显示
- **优点**：
  - 服务器权威，易反作弊
  - 客户端崩溃不影响
  - 开发简单
- **缺点**：
  - 流量大（传状态）
  - 延迟高（等服务器）

**帧同步详细说明：**

- **流程**：
  1. 客户端收集输入
  2. 上传服务器
  3. 服务器转发所有输入给所有客户端
  4. 每个客户端按相同逻辑计算
- **优点**：
  - 流量小（只传输入）
  - 同步精确
  - 可以做精彩回放
- **缺点**：
  - 要求所有客户端逻辑完全一致
  - 一个客户端卡，大家都卡
  - 反作弊难（客户端计算）

**三角洲行动大战场用哪种？**

- **推测：状态同步为主，关键操作混合帧同步**
- **理由**：
  1. 100人大战场，状态同步流量可控
  2. 射击、移动需要精确，可能用帧同步
  3. 状态同步易反作弊（FPS外挂多）
  4. 符合现代FPS趋势（如CS:GO用状态同步）

- **负载均衡**：Nginx/LVS，均匀分发
- **CDN**：静态资源走CDN
- **WAF**：防攻击、防刷

**3. 排队系统**：

- **登录排队**：进入游戏前排队，显示预估时间
- **虚拟大厅**：先进入大厅，再慢慢进服
- **分批开放**：按时间分批开放注册

**4. 服务层**：

- **限流**：令牌桶/滑动窗口，限制QPS
- **熔断**：依赖服务挂了，快速失败
- **降级**：关闭非核心功能（如排行榜、聊天）
- **异步**：耗时操作丢MQ，先返回成功

**5. 数据层**：

- **读写分离**：读走从库，写走主库
- **分库分表**：用户表按userId哈希分片
- **多级缓存**：本地缓存 + Redis，减少DB压力

**6. 监控与应急**：

- **全链路监控**：实时看各项指标
- **快速扩容**：支持弹性伸缩
- **回滚预案**：出问题快速回滚

**排查和处理流程：**

```
┌─────────────────────────────────────────────────────────────┐
│              大面积掉线排查处理流程                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │

│  ├─ 停止发布、停止变更                                      │

│                                                              │
│  【2. 初步排查（10分钟内）】                                 │
│  ├─ 看监控面板：                                             │

│  ├─ 看现象：                                                 │

│  │  ├─ 新玩家能登录吗？                                     │
│  │  └─ 是断开还是卡顿？                                     │
│  └─ 快速定位大类：网络/服务器/中间件/代码                   │
│                                                              │
│  【3. 深入排查（30分钟内）】                                 │
│  ├─ 查日志：ERROR/FATAL日志                                 │
│  ├─ 查堆栈：jstack/gdb看死锁/阻塞                           │
│  ├─ 查网络：telnet/ping/tcpdump                             │
│  ├─ 查中间件：数据库慢查询、Redis大key                      │
│  └─ 查变更：最近是否有发布、配置变更？                       │
│                                                              │
│  【4. 止损恢复（越快越好）】                                  │
│  ├─ 回滚：最近发布的先回滚                                   │
│  ├─ 重启：有问题的服务先重启                                 │
│  ├─ 扩容：负载高的扩容                                       │
│  ├─ 降级：关闭非核心功能                                     │
│  └─ 切流：有问题的机房/节点切走                             │
│                                                              │
│  【5. 事后复盘】                                             │
│  ├─ 根因分析：5Why找出根本原因                              │
│  ├─ 修复问题：永久修复，加监控加告警                         │
│  └─ 总结改进：完善流程，避免再犯                             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**常见原因和对应处理：**

| 可能原因             | 现象           | 处理                 |
| -------------------- | -------------- | -------------------- |
| **服务器崩溃** | 进程挂了       | 重启服务             |
| **数据库挂了** | 所有DB操作失败 | 切从库、重启DB       |
| **网络故障**   | ping不通、丢包 | 联系网络运维、切机房 |
| **死锁**       | CPU低但无响应  | gdb看堆栈、重启      |
| **OOM**        | 内存满了被kill | 加内存、优化内存     |
| **发布事故**   | 刚发布就出事   | 快速回滚             |
