# 一、60 秒项目自我介绍（流利、专业、面试官最爱版）

我做的项目是 **基于 Reactor 模型的高并发 TCP 长连接网关服务器** ，主要面向即时通讯场景，支持万级并发连接与十万级 QPS。

整体采用 **单 Reactor 多线程架构** ，基于 Epoll 实现 IO 多路复用，使用 muduo 风格的 Buffer 解决 TCP 粘包拆包问题，并设计了 **带长度字段与 CRC 校验的自定义应用层协议** ，保证消息可靠传输。

业务上实现了**登录鉴权、心跳保活、私聊与广播消息、过载保护**等完整功能。

服务器由 EventLoop 事件循环驱动，通过 Channel 封装 IO 事件，TcpServer 统一管理连接，配合线程池处理异步任务。

同时做了大量工程化优化：优雅退出、连接限流、QPS 限流、超时踢人、日志监控等。

整个项目从网络框架、协议设计到业务逻辑全部自主实现，对高并发、线程安全、IO 模型、长连接稳定性有比较深入的理解。

（语速正常 ≈ 55~65 秒，完美适配面试开场）

## 1. **Reactor 模型 + EventLoop 事件循环（重中之重）**

### 为什么要用 Reactor？

Reactor 模式是一种事件驱动的设计模式，核心思想是**将 IO 事件与业务逻辑分离**。

- 相比阻塞 IO，Reactor 可以用单线程管理大量连接，避免大量线程开销
- 相比多进程多线程，Reactor 无上下文切换开销，延迟更低
- 事件驱动模型更符合网络编程特点：连接多、活跃少

### 单 Reactor 多线程 vs 多 Reactor 多线程

- **单 Reactor 多线程**：
  - IO 事件由主线程统一处理，耗时业务交给线程池
  - 实现简单、延迟低，适合万级连接
  - 缺点：单线程 IO 成为瓶颈，无法充分利用多核
- **多 Reactor 多线程**：
  - 主线程 accept 新连接，分发给子 Reactor
  - 每个子 Reactor 独立处理自己的连接
  - 可以充分利用多核，支持更高并发
  - 实现复杂，需要处理负载均衡

### EventLoop 为什么要 one loop per thread？

one loop per thread（一个线程一个事件循环）是 Reactor 模式的核心设计：

- **串行化处理**：同一个线程内的所有事件串行执行，避免竞争
- **无锁设计**：IO 线程内不需要锁，天然线程安全
- **线程亲和性**：EventLoop 绑定到特定线程，避免线程切换
- **清晰的职责划分**：每个线程只负责自己的事件循环

### 怎么保证线程安全？为什么不锁？

- **IO 线程内**：所有回调都在同一个线程串行执行，不存在竞争，不需要锁
- **跨线程通信**：使用 EventLoop::runInLoop() 将要执行的函数投递到目标线程
- **共享数据**：
  - 简单计数器用 atomic（如连接数、QPS）
  - 复杂数据结构用 mutex（如连接 map、用户表）
- **为什么不锁**：锁会带来性能开销、死锁风险，能无锁就无锁

### 定时器怎么实现？（你的心跳就是靠它）

项目中基于 EventLoop 实现定时器：

- **数据结构**：使用 priority_queue（最小堆）管理定时器，按超时时间排序
- **触发方式**：每次 epoll_wait 返回后，检查堆顶定时器是否超时
- **心跳实现**：每个连接记录最后心跳时间，定时器定期遍历检查，超时则踢除
- **优化方向**：更高并发可改用时间轮算法，O(1) 插入删除

## 2. **Buffer 设计（muduo 风格缓冲区）**

### 为什么要 prependable 区域？

prependable 是前置预留区（通常 8 字节），作用是：

- **提前预留空间**：某些协议需要在消息前加头部（如长度字段），prependable 可以避免数据移动
- **高效 prepend**：直接在前置区写数据，不需要移动后面的可读数据
- **减少内存拷贝**：相比每次都要重新分配内存，prependable 可以复用空间

### 怎么解决 TCP 粘包拆包？

TCP 是流式协议，没有消息边界，需要应用层解决：

- **方案一：固定长度消息**：所有消息长度相同，简单但浪费空间
- **方案二：特殊分隔符**：用 \n 或 \0 分隔，不适合二进制数据
- **方案三：固定头 + 长度字段**（本项目采用）：
  1. 先读固定长度协议头（16 字节）
  2. 从头部获取消息体长度
  3. 根据长度读取完整消息
  4. Buffer 缓存半包，直到凑齐完整包
- **优势**：效率高、支持二进制、可靠

### makeSpace 内部逻辑（数据移动 vs 扩容）

makeSpace 确保有足够的可写空间：

```
1. 检查 writable 区域是否足够
   ├─ 足够 → 直接返回
   └─ 不足 → 继续
2. 检查 prependable + writable 是否足够
   ├─ 足够 → 将 readable 数据向前移动，合并出连续空间
   └─ 不足 → 扩容 vector（通常 2 倍或按需扩容）
3. 返回可写指针
```

- **数据移动**：空间足够但碎片化时，避免扩容
- **扩容**：空间真的不足时，才进行内存分配

### 为什么用 vector `<char>` 而不是自己 malloc？

- **自动内存管理**：RAII，自动释放，避免内存泄漏
- **连续内存**：vector 保证内存连续，适合网络 IO
- **方便扩容**：resize() 自动处理扩容逻辑
- **STL 兼容性**：可以使用各种 STL 算法
- **安全**：有边界检查（at()），调试更方便

### 读写指针如何移动？

Buffer 有三个指针/索引：

- **readerIndex**：可读数据起始位置
- **writerIndex**：可写数据起始位置
- **readable 区域**：[readerIndex, writerIndex)
- **writable 区域**：[writerIndex, capacity)

**移动规则**：

- **读取数据**：readerIndex += len（消费数据）
- **写入数据**：writerIndex += len（生产数据）
- **retrieveAll**：readerIndex = writerIndex = kCheapPrepend（清空）
- **确保不变式**：0 ≤ readerIndex ≤ writerIndex ≤ capacity

## 3. **TcpServer / TcpConnection 生命周期管理**

### 为什么要继承 enable_shared_from_this？

TcpConnection 需要在回调中获取自身的 shared_ptr：

- **异步回调安全**：回调执行时，对象可能已被销毁，用 shared_ptr 保证对象存活
- **避免悬空指针**：不能直接用 this，回调时对象可能已析构
- **延长生命周期**：shared_ptr 引用计数保证对象在回调执行期间不被销毁
- **使用方式**：在回调中调用 shared_from_this() 获取自身 shared_ptr

### 连接建立、消息到达、连接关闭的完整流程

**连接建立**：

1. 客户端 connect → 服务器 accept 新连接
2. 创建 TcpConnection 对象，封装 fd
3. 设置回调函数（读、写、关闭、错误）
4. 注册到 EventLoop，关注可读事件
5. 调用用户连接建立回调

**消息到达**：

1. epoll_wait 返回可读事件
2. Channel 调用读回调
3. 从 fd 读取数据到 Buffer
4. Codec 解析协议，处理半包
5. 提取完整消息，反序列化
6. 分发到业务逻辑处理

**连接关闭**：

1. 检测到对端关闭（read 返回 0）或错误
2. 调用关闭回调
3. 从 TcpServer 连接 map 中移除
4. 从各个服务注销（心跳、鉴权、消息）
5. 调用用户连接关闭回调
6. 关闭 fd，引用计数归零自动析构

### 优雅关闭 / 半关闭 / RST 问题

**close**：

- 直接关闭套接字，读写都不可用
- 如果接收缓冲区还有数据，可能发送 RST 导致数据丢失
- 不推荐用于主动关闭

**shutdown**：

- 半关闭，只关闭写端，读端仍然可用
- 可以继续接收对端发送的数据
- 更优雅，等待对端处理完数据

**RST 问题**：

- RST 是重置报文，表示连接异常关闭
- 发生场景：close 时接收缓冲区还有数据、对端崩溃、网络异常
- 后果：数据丢失、连接立即断开
- 避免：使用 shutdown + 优雅等待，不要直接 close

### 服务器如何管理所有连接？

TcpServer 内部维护连接表：

- **数据结构**：`unordered_map<string, shared_ptr<TcpConnection>>`
- **Key**：连接名称（如 "host:port#fd"），唯一标识连接
- **操作**：
  - 连接建立：insert 到 map
  - 连接关闭：erase 从 map
  - 查找连接：通过 key O(1) 查找
- **线程安全**：map 操作需要加 mutex 保护

### 线程安全怎么保证？

- **TcpConnection 自身**：所有回调在 IO 线程串行，不需要锁
- **连接表**：unordered_map 共享访问，用 mutex 保护
- **跨线程操作**：使用 runInLoop() 投递到 IO 线程执行
- **智能指针**：shared_ptr 保证多线程下对象生命周期安全
- **原子变量**：简单计数器用 atomic，避免锁

## 4. **自定义协议 + Codec 编解码（粘包杀手）**

### 协议头结构（长度、命令号、序号、CRC）

本项目协议头 16 字节：

```
| 总长度 (4B) | 命令号 (2B) | 序列号 (4B) | CRC16 (2B) | 消息体长度 (4B) |
```

- **总长度**：整个消息字节数（头 + 体），用于分包
- **命令号**：标识消息类型（登录、心跳、消息等）
- **序列号**：请求响应配对、去重、排序
- **CRC16**：校验消息完整性，防止篡改
- **消息体长度**：JSON 消息体长度

### 如何解决粘包？（固定头 + 长度字段）

**核心思路**：用长度字段明确消息边界

1. **读取固定头**：先读取 16 字节协议头
2. **解析长度**：从头部获取总长度
3. **判断完整性**：检查 Buffer 中可读数据是否 ≥ 总长度
4. **提取消息**：够则取出完整消息，更新 readerIndex
5. **缓存半包**：不够则等待下一次数据到达，继续累积

**Codec 状态机**：

- 状态 1：等待协议头（需要 16 字节）
- 状态 2：等待消息体（需要 总长度 - 16 字节）
- 循环往复，处理 Buffer 中所有完整包

### CRC 校验作用

CRC（循环冗余校验）作用：

- **完整性检查**：检测网络传输中是否发生比特错误
- **防篡改**：防止消息被恶意篡改（虽然不加密，但能检测）
- **快速失败**：校验失败直接丢弃，避免错误数据进入业务逻辑
- **本项目**：CRC16 校验整个消息（头 + 体）

**校验失败处理**：

- 丢弃当前数据包
- 清空 Buffer（避免错误数据污染后续解析）
- 记录警告日志
- 可选：关闭连接（严重错误）

### 半包、丢包、错包如何处理？

**半包**：

- Buffer 缓存未收齐的数据
- 等待下一次 read 事件
- 继续累积，直到凑齐完整包

**丢包**：

- TCP 保证可靠传输，应用层通常不处理
- 如果是 UDP，需要序列号 + 重传机制
- 本项目 TCP，无需应用层重传

**错包**：

- CRC 校验失败
- 丢弃数据包，清空 Buffer
- 记录日志，可选关闭连接
- 防止错误数据扩散

### 序列化为什么用 JSON？Protobuf 对比？

**JSON 优势**：

- 可读性好，调试方便
- 无需编译 .proto 文件，开发快
- 跨语言友好
- 适合小项目、原型开发

**Protobuf 优势**：

- 序列化后体积更小（二进制）
- 序列化/反序列化更快
- 强类型，编译时检查
- 向前向后兼容性好
- 适合高性能、大规模项目

**本项目选择 JSON**：

- 学习项目，可读性优先
- 消息量不大，性能足够
- 方便调试和观察数据

## 5. **心跳机制 + 超时踢人（长连接必备）**

### 为什么应用层要心跳？TCP keepalive 为什么不够？

**TCP keepalive 局限性**：

- 默认周期太长（通常 2 小时）
- 只能检测连接是否存活，无法检测应用层是否假死
- 不是所有系统都支持，配置复杂
- 无法自定义心跳内容

**应用层心跳优势**：

- 可自定义周期（如 30 秒）
- 可检测应用层状态（进程 hang 但 TCP 还在）
- 可携带业务数据（如负载信息）
- 跨平台，不依赖系统配置
- 超时后可以主动清理资源

### 定时器如何实现？（时间堆 / 时间轮 / 简单遍历）

**本项目实现**：简单遍历（适合万级连接）

- 数据结构：vector 或 map 保存所有连接
- 触发：每隔 N 秒遍历一次
- 检查：比较当前时间与最后心跳时间
- 踢除：超时则关闭连接

**时间堆**（优先队列）：

- 按超时时间排序，堆顶是最早超时的
- 每次只检查堆顶，O(logN) 插入删除
- 适合连接数多、精度要求高

**时间轮**：

- 环形数组，每个槽代表一个时间刻度
- O(1) 插入、删除、触发
- 适合超大规模连接（百万级）

### 超时怎么判断？

**心跳流程**：

1. 连接建立后，记录 lastHeartbeatTime = now
2. 每次收到心跳包，更新 lastHeartbeatTime = now
3. 定时器定期检查：
   ```
   if (now - lastHeartbeatTime > timeoutThreshold) {
       超时，踢除连接
   }
   ```
4. 超时阈值：通常是心跳周期的 2~3 倍（如心跳 30s，超时 90s）

**注意**：

- 收到任何消息都可以更新心跳时间（不仅是心跳包）
- 超时阈值要合理，避免误判网络抖动

### 踢人后资源如何清理？（Auth、MessageService 注销）

**完整清理流程**：

1. **从 TcpServer 移除**：从连接 map 中 erase
2. **从 AuthService 注销**：清除 userId ↔ conn 的映射
3. **从 MessageService 注销**：从在线用户表中移除
4. **从 HeartbeatService 移除**：不再监控心跳
5. **关闭 fd**：shutdown 或 close 套接字
6. **调用回调**：通知用户层连接关闭
7. **智能指针析构**：引用计数归零，自动释放内存

**关键**：清理要彻底，避免内存泄漏和悬空指针

## 6. **线程安全设计（atomic + mutex + 无锁设计）**

### OverloadProtectService 为什么用 atomic？

OverloadProtectService 管理连接数和 QPS 计数：

- **场景**：多线程频繁更新简单计数器
- **atomic 优势**：
  - 无锁，性能更高
  - 原子操作，硬件支持
  - 不会死锁
  - 适合简单的增减操作
- **mutex 劣势**：
  - 锁开销大
  - 可能死锁
  - 高并发下竞争激烈
- **结论**：简单计数用 atomic，复杂数据结构用 mutex

### MessageService 用户连接表为什么加锁？

MessageService 维护在线用户表 `unordered_map<userId, conn>`：

- **场景**：多个线程可能同时读写（登录、发消息、登出）
- **为什么要锁**：
  - unordered_map 不是线程安全的
  - 并发读写可能导致数据竞争、崩溃
  - 迭代器失效、内存 corruption
- **锁的选择**：
  - std::mutex：互斥锁，简单场景
  - std::shared_mutex：读写锁，读多写少（如本项目）
- **优化**：缩小锁粒度，用 fine-grained locking

### 哪些地方可以无锁？为什么？

**可以无锁的场景**：

1. **单线程内**：EventLoop 回调都在同一个线程，天然无竞争
2. **线程局部数据**：thread_local，每个线程独立副本
3. **原子操作**：simple counter，用 atomic
4. **无锁数据结构**：CAS 实现的队列、栈（复杂，不推荐自己写）

**为什么能无锁**：

- 无竞争：数据只被一个线程访问
- 硬件支持：atomic 利用 CPU 原子指令
- 性能更好：无锁开销，无死锁风险

**无锁原则**：能无锁就无锁，必须加锁才加锁

### 线程池任务队列为什么用阻塞队列？

线程池任务队列是生产者-消费者模型：

- **生产者**：IO 线程投递任务
- **消费者**：线程池线程执行任务

**阻塞队列优势**：

- **自动等待**：队列为空时，消费者线程自动阻塞（condition_variable）
- **自动唤醒**：有新任务时，自动唤醒一个消费者
- **无空轮询**：不消耗 CPU，避免 busy wait
- **线程安全**：内部用 mutex 保护，并发安全

**实现**：

```cpp
template <typename T>
class BlockingQueue {
    std::mutex mutex_;
    std::condition_variable notEmpty_;
    std::queue<T> queue_;
public:
    void put(T task) {
        std::lock_guard<std::mutex> lock(mutex_);
        queue_.push(task);
        notEmpty_.notify_one();
    }
    T take() {
        std::unique_lock<std::mutex> lock(mutex_);
        notEmpty_.wait(lock, [this] { return !queue_.empty(); });
        T task = queue_.front();
        queue_.pop();
        return task;
    }
};
```

## 7. **业务流程（登录 → 消息转发 → 广播）**

### 登录流程

**完整登录流程**：

1. **客户端发送登录请求**：

   - 构造 LoginReq 消息（userId、token）
   - 编码成二进制，发送给服务器
2. **服务器接收并解析**：

   - TCP 接收数据到 Buffer
   - Codec 解析协议，提取完整消息
   - 反序列化 JSON 为 LoginReq
3. **鉴权验证**：

   - 检查 userId 是否合法
   - 验证 token（本项目简化，实际项目查数据库/Redis）
   - 检查是否重复登录（可选：踢掉旧连接）
4. **绑定连接与用户**：

   - TcpConnection 绑定 userId
   - AuthService 记录 userId ↔ conn 映射
   - MessageService 注册到在线用户表
   - HeartbeatService 加入心跳监控
5. **返回登录响应**：

   - 构造 LoginResp（success=true/false）
   - 编码发送给客户端
6. **后续**：

   - 客户端可以开始收发消息
   - 心跳包定期更新 lastHeartbeatTime

### P2P 消息如何找到对应用户

**私聊消息流程**：

1. **发送方 A 发消息**：

   - 构造 P2PMsgReq（fromUserId、toUserId、content）
   - 发送给服务器
2. **服务器处理**：

   - 鉴权：检查 A 是否登录
   - 查在线表：MessageService 根据 toUserId 查找连接
   - 两种情况：
     - **用户在线**：
       - 构造 P2PMsgNotify（from、content、msgId）
       - 通过 B 的连接发送
       - 返回 A：发送成功
     - **用户不在线**：
       - 可选：存储离线消息（本项目未实现）
       - 返回 A：发送失败，用户不在线
3. **接收方 B 收消息**：

   - 收到 P2PMsgNotify
   - 显示给用户

**关键**：MessageService 维护 userId → conn 的 map，O(1) 查找

### 广播怎么实现

**广播消息流程**：

1. **发送方发广播请求**：

   - 构造 BroadcastReq（content）
   - 发送给服务器
2. **服务器处理**：

   - 鉴权：检查是否允许广播
   - 遍历所有在线连接：
     - 对每个连接，构造 BroadcastNotify
     - 编码发送
   - 返回发送方：广播成功

**实现细节**：

- **数据结构**：遍历 MessageService 的在线用户 map
- **性能**：
  - 万级连接以下，直接遍历足够
  - 更高并发：分片（每个 Reactor 负责一部分）、批量发送
- **注意**：
  - 不要在 IO 线程做耗时操作（广播耗时交给线程池）
  - 避免阻塞，一个连接慢不影响其他连接

### 消息 ID 如何唯一

**消息 ID 生成**：

- **方案**：全局原子自增整数
- **实现**：`std::atomic<uint64_t> globalMsgId{0}`
- **生成**：`uint64_t msgId = globalMsgId.fetch_add(1);`
- **优势**：
  - 单线程生成（或原子操作），保证唯一
  - 有序，方便排序和去重
  - 实现简单，性能高
- **适用场景**：单实例服务器
- **分布式场景**：需要 Snowflake 算法（时间戳 + 机器 ID + 序列号）

**本项目**：单实例，用 atomic 自增足够

### 异常处理（用户不在线、网络断开）

**用户不在线**：

- P2P 消息：返回发送失败，提示用户不在线
- 可选：存储离线消息，用户上线后推送

**网络断开**：

- 检测到连接关闭
- 从所有服务中注销（Auth、Message、Heartbeat）
- 清理资源
- 通知相关用户（可选）

**消息发送失败**：

- 写缓冲区满：等待可写事件再发
- 连接已关闭：丢弃消息，记录日志
- 超时：重试或丢弃

**通用原则**：

- 快速失败，不要阻塞
- 记录日志，便于排查
- 资源清理彻底，避免泄漏
- 给用户明确的错误提示

# 三、全套面试题标准答案（精简、好背、不踩坑）

## 【一、架构与 IO 模型】

### 1. 用的什么 IO 模型？为什么？

使用 **IO 多路复用 + Reactor 模式** ，Linux 下基于 epoll 实现。

相比阻塞 IO、多进程多线程，epoll 能在单线程内高效管理大量连接，无线程切换开销，适合高并发长连接场景。

### 2. 讲 Reactor 工作流程

1. 主线程 EventLoop 不断调用 epoll_wait 等待事件；
2. 事件就绪后由 Channel 分发到对应回调；
3. 读事件负责接收数据并交给业务层；
4. 写事件负责发送缓冲区数据；
5. 所有 IO 操作都在主线程串行，避免锁竞争。

### 3. EventLoop 是什么？one loop per thread 意义

EventLoop 是事件循环，负责驱动定时器、IO 事件、异步任务。

one loop per thread 保证 **IO 事件在单个线程内串行处理** ，无竞争、无锁、高效稳定。

### 4. 单 Reactor 还是多 Reactor？

项目采用 **单 Reactor 主循环** ，IO 事件统一由主线程处理，耗时任务交给线程池。

实现简单、延迟低，足够支撑万级连接；后续可扩展为多 Reactor 支持更高并发。

### 5. Channel 封装了什么？

Channel 封装 **fd + 事件类型 + 回调函数** ，将读、写、错误、关闭事件与处理逻辑绑定，是 EventLoop 与 IO 事件的桥梁。

### 6. LT 与 ET 区别？用哪种？

水平触发 LT：只要缓冲区有数据就一直通知，实现简单、不易丢数据。

边缘触发 ET：只有状态变化时通知，效率更高但需一次性读完。

项目使用 **LT 模式** ，更稳定、适合长连接网关。

### 7. 如何支撑高并发？

1. epoll 管理大量连接；
2. 非阻塞 IO 避免阻塞；
3. 无锁 IO 线程减少竞争；
4. Buffer 减少内存拷贝；
5. 过载保护防止雪崩。

## 【二、Buffer、协议、粘包拆包】

### 8. Buffer 结构？

分为三个区域：

* prepend 前置预留区（8 字节）
* readable 可读数据区
* writable 可写空闲区
  使用 vector `<char>` 做连续内存存储。

### 9. 如何解决粘包拆包？

采用 **固定长度协议头 + 消息体长度字段** ：

1. 先读 16 字节协议头，获取总长度；
2. 再根据长度读取完整消息；
3. Buffer 缓存半包，直到凑齐完整包再解析。

### 10. 协议头 16 字节设计？

* 4 字节：总长度
* 2 字节：命令号
* 4 字节：序列号
* 2 字节：CRC16 校验
* 4 字节：消息体长度
  用于分包、校验、路由、去重。

### 11. CRC 作用？失败怎么处理？

CRC 校验消息完整性，防止网络传输错误、篡改。

校验失败直接 **丢弃数据包、清空缓冲区** ，避免错误数据污染业务逻辑。

### 12. makeSpace 逻辑？

1. 剩余空间不足则 **扩容 vector** ；
2. 空间足够但碎片化，则 **将数据向前挪动** ，整理出连续可写区；
3. 避免频繁扩容与拷贝。

### 13. 应用层拆包逻辑？

1. 从 Buffer 读取数据；
2. 先判断是否够 16 字节头；
3. 解析长度，判断是否够完整包；
4. 够则取出并消费缓冲区；不够则等待下一次数据。

## 【三、连接管理与生命周期】

### 14. 为什么用 shared_ptr？

TcpConnection 生命周期不确定，可能在网络线程、业务线程、定时器中被引用，

用 shared_ptr 保证 **资源自动释放、避免悬空指针** 。

### 15. 为什么 enable_shared_from_this？

在回调中需要获取自身的 shared_ptr，避免直接裸指针不安全，

保证对象在异步调用中不被提前销毁。

### 16. 连接关闭完整流程？

1. 检测到关闭事件；
2. 从 TcpServer 连接 map 中移除；
3. 从心跳服务、鉴权服务、消息服务注销；
4. 关闭 fd，释放资源；
5. 引用计数归零自动析构。

### 17. shutdown 与 close 区别？

close 直接关闭套接字，可能发送 RST 导致数据丢失。

shutdown 是半关闭，只关闭写端，等待对端数据接收完毕，更优雅。

### 18. 如何管理在线连接？

TcpServer 内部维护一个 `unordered_map<string, shared_ptr<TcpConnection>>`，

以连接名为 key，O (1) 增删查。

## 【四、业务逻辑与高并发关键】

### 19. 登录流程？

1. 客户端发登录包；
2. 服务器鉴权验证；
3. 验证通过则绑定 conn 与 userId；
4. 注册到 MessageService；
5. 返回登录响应；
6. 加入心跳监控。

### 20. 私聊全链路？

1. A 发送 P2P 消息；
2. 服务器鉴权；
3. 根据 toUserId 查在线连接；
4. 构造通知包，编码发送；
5. 返回发送结果给 A。

### 21. 广播实现？

遍历所有在线连接，依次发送消息。

万级以下效率足够；更高并发可采用分片发送避免卡顿。

### 22. 为什么要应用层心跳？TCP keepalive 不行？

TCP keepalive 周期长、粒度粗、无法感知应用层假死。

应用层心跳可 **自定义周期、精准判断存活、及时清理死连接、释放资源** 。

### 23. 心跳超时与定时器？

基于 EventLoop 定时器，定时遍历所有连接，

比较当前时间与最后心跳时间，超过阈值则踢除。

### 24. 过载保护怎么做？

* 连接数限流：atomic 计数，超过阈值拒绝新连接；
* QPS 限流：按秒统计消息数，超阈值直接丢弃。

### 25. 消息 ID 唯一？

使用全局原子自增整数，单线程生成，保证唯一且有序。

---

## 【五、线程安全与并发】

### 26. 哪些地方用锁？

* 连接 map、用户在线表：共享数据结构用 mutex；
* 阻塞队列：生产者消费者用 mutex + condition_variable；
* 配置、日志单例：保护读写。

### 27. 哪些用 atomic？为什么不用锁？

当前连接数、QPS 计数、运行状态等简单计数器用 atomic，

无锁、原子操作，比 mutex 开销小、高并发下更快。

### 28. 线程池为什么用阻塞队列？

实现生产者消费者模型，任务队列为空时线程阻塞等待，

避免空轮询消耗 CPU。

### 29. IO 线程与业务线程划分？

IO 事件（读、写、连接）在主线程 EventLoop 处理；

数据库、加密、复杂计算等耗时任务提交到线程池，不阻塞主循环。

### 30. EventLoop 内部为什么不用锁？

所有 IO 回调都在**同一个线程**内串行执行，不存在竞争，天然线程安全。

---

## 【六、工程与优化】

### 31. 优雅退出？

捕获 SIGINT/SIGTERM 信号，通知 EventLoop 退出；

依次停止心跳、关闭服务器、关闭所有连接、停止线程池、释放资源。

### 32. 内存泄漏如何排查？

* 智能指针管理资源；
* 连接断开时完整清理；
* 使用 Valgrind、AddressSanitizer 检测。

### 33. 大量频繁建连断连怎么办？

* 快速回收 fd 与内存；
* 限制新建连接速率；
* 避免锁竞争；
* 日志异步化。

### 34. QPS 突增怎么处理？

* 过载保护直接限流；
* 业务逻辑异步化；
* 合并发送、减少系统调用；
* 优化锁粒度。

### 35. 支持 10 万连接改哪里？

* 改用多 Reactor 多线程；
* 启用 SO_REUSEPORT；
* 定时器改用时间堆 / 时间轮；
* 广播改用批量发送；
* 调整系统参数（fd 上限、tcp 参数）。
