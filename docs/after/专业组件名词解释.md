# 一、IO 模型 & 高并发核心

## 1. Reactor 模型

**直译：反应堆模式**

* 一种 **事件驱动的高并发 IO 设计模式** 。
* 由一个主线程循环监听事件（连接、读、写、异常），事件就绪后分发给对应回调处理。
* 优点：单线程处理大量连接、无阻塞、无过多线程切换，适合长连接高并发。
* 你的项目：**主 Reactor + 业务线程池**架构。

## 2. Epoll

Linux 下 **高性能 IO 多路复用机制** ，select/poll 的升级版。

* 支持百万级并发连接
* 内部使用红黑树管理 fd，事件通知效率高
* 是高并发服务器的基石
* 你的项目：用 epoll 作为 EventLoop 的底层驱动

## 3. LT 模式（Level Triggered 水平触发）

* 只要 fd 缓冲区有数据 / 可写，就 **持续触发通知** 。
* 实现简单、稳定、不易丢数据。
* 你的项目默认使用 LT。

## 4. ET 模式（Edge Triggered 边缘触发）

* 只有状态**发生变化**时才触发一次通知。
* 效率更高，但必须一次性读写完全部数据，实现复杂。
* 你的项目未使用，属于可扩展优化点。

## 5. One loop per thread

* 每个线程 **最多绑定一个 EventLoop** 。
* IO 事件在单个线程内串行执行， **无锁、无竞争、天然线程安全** 。
* 现代 C++ 高并发网络库标准设计（muduo /libevent/libev 都用这套）。

# 二、网络组件（你项目核心结构）

## 6. EventLoop

事件循环，服务器的 **心脏** 。

* 不断执行：epoll_wait → 分发事件 → 执行回调 → 处理定时任务
* 管理定时器、异步任务、IO 事件
* 保证所有 IO 操作在同一线程

## 7. Channel

对 **fd + 事件 + 回调** 的封装。

* 把 socket、可读 / 可写 / 错误 / 关闭事件绑定在一起
* 是 EventLoop 与具体 IO 事件的桥梁

## 8. TcpServer

服务器入口，负责 **监听端口、接受新连接** 。

* 创建监听 socket
* accept 新连接
* 管理所有 TcpConnection

## 9. TcpConnection

代表 **一条客户端连接** 。

* 封装一个客户端 socket 的生命周期
* 处理收发数据、连接关闭、异常
* 使用 shared_ptr 管理生命周期

## 10. muduo 风格

陈硕 muduo 网络库的经典设计风格：

* Reactor 模型
* one loop per thread
* vector`<char>` 实现自动扩容缓冲区
* 双指针读写 Buffer
* 回调机制解耦网络与业务
  你的 Buffer、EventLoop、TcpConnection 都属于 muduo 风格。

# 三、缓冲区 & 协议相关

## 11. Buffer（应用层缓冲区）

* 用 `vector<char>` 实现的自动扩容、读写分离的网络缓冲区。
* 解决 TCP 流式传输无边界问题。
* 结构：prependable（前置预留区）+ readable（可读区）+ writable（可写区）

## 12. TCP 粘包 / 拆包

* **粘包** ：多个小包被 TCP 一次性发过来，粘在一起。
* **拆包** ：一个大包被 TCP 分成多次发送，只收到一部分。
* 根本原因：TCP 是 **流协议** ，无消息边界。
* 你的解决方案： **自定义协议 + 长度字段 + CRC 校验** 。

## 13. 自定义协议

你设计的应用层数据包格式：

* 固定 16 字节协议头
* 包含：总长度、命令号、序列号、CRC16、体长度
* 用于消息路由、完整性校验、解决粘包

## 14. CRC 校验

循环冗余校验，一种 **数据校验算法** 。

* 发送方计算校验和
* 接收方重新计算比对
* 防止网络传输错误、数据篡改、脏数据
* 校验失败直接丢弃包

## 15. 序列化 / 反序列化

* 序列化：结构体 → JSON 字符串
* 反序列化：JSON 字符串 → 结构体
* 你的项目用 JSON 做跨平台消息传输

# 四、业务与性能指标

## 16. QPS

Queries Per Second， **每秒请求 / 消息数** 。

衡量服务器处理能力的核心指标。

## 17. 长连接

客户端连接后 **不断开** ，持续收发消息。

适合 IM、游戏、网关、物联网。

短连接：请求一次就断开（HTTP1.0）。

## 18. 心跳机制

* 客户端定时发心跳包
* 服务端记录最后活跃时间
* 超时未收到则判定客户端死亡并踢人
* 解决：断网、崩溃、假死连接无法感知问题

## 19. TCP Keepalive

操作系统自带的 TCP 保活机制。

* 周期太长（默认 2 小时）
* 应用层不可感知
* 不能替代应用层心跳

## 20. 过载保护

当连接数、QPS 超过阈值时：

* 拒绝新连接
* 限流消息
* 防止服务器雪崩、卡死、崩溃

# 五、网络异常与底层机制

## 21. RST 包

TCP 复位包，表示 **强制断开连接** 。

出现场景：

* 访问已关闭的端口
* close 粗暴关闭 socket
* 协议错误、异常断开
  会导致数据丢失，项目中尽量避免。

## 22. 半关闭 / SHUT_WR

调用 `shutdown(fd, SHUT_WR)` 只关闭 **发送方向** ，不关闭接收。

* 优雅关闭
* 保证数据发送完毕
* 避免 RST

## 23. 优雅退出

收到 SIGINT（Ctrl+C）后：

* 停止接收新连接
* 关闭已有连接
* 释放资源、停止线程池、退出事件循环
* 不崩溃、不泄漏、不丢数据

## 24. SO_REUSEADDR / SO_REUSEPORT

* 允许端口快速复用
* 支持多进程 / 多线程绑定同一个端口
* 提升并发、减少重启报错

## 25. Non-blocking IO（非阻塞 IO）

* read/write 不会阻塞线程
* 没数据直接返回 EAGAIN
* 配合 Epoll 实现高并发

# 六、并发与线程相关

## 26. 线程池

一组预先创建的线程，处理耗时任务。

* 不阻塞主线程（IO 线程）
* 避免频繁创建销毁线程
* 你的项目用于异步业务处理

## 27. BlockingQueue

线程安全的 **阻塞队列** ，用于生产者 — 消费者模型。

* 空时线程等待
* 有任务自动唤醒
* 无空轮询，省 CPU

## 28. Atomic（原子变量）

* 无锁线程安全变量
* 适合计数、开关状态
* 比 mutex 轻量高效
  你的项目：连接计数、QPS 限流、运行状态。

## 29. 共享指针 shared_ptr

自动管理对象生命周期，自动释放内存。

* 解决悬空指针、重复释放、内存泄漏
* TcpConnection 必须用它

## 30. enable_shared_from_this

在类成员函数中安全获取自身的 `shared_ptr`。

* 避免异步回调中对象被销毁
* 高并发网络库必备

# 七、一句话速记版（面试急救）

* **Reactor** ：事件驱动，主线程等事件，来了就处理。
* **Epoll** ：Linux 高并发 IO 多路复用。
* **muduo 风格** ：现代 C++ 高并发网络库标准设计。
* **Buffer** ：自动扩容读写缓冲区，解决粘包。
* **CRC** ：校验数据是否完整不被篡改。
* **QPS** ：每秒处理多少请求。
* **LT/ET** ：水平触发（稳定）/ 边缘触发（高效）。
* **RST** ：强制断开，粗暴关闭。
* **心跳** ：判断客户端是否活着，超时踢人。
* **长连接** ：一次连接，持续通信。
