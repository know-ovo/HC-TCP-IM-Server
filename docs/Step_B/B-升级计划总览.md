# Step_B 升级计划总览

## 1. 文档目的

本文档不是泛泛而谈的“架构畅想”，而是基于当前仓库已有实现，为项目从“可运行 Demo”升级到“可用于求职展示、并具备工业级设计深度”的 Linux 高性能 IM 服务器，给出一份可落地的升级计划。

本阶段目标有两个：

- 让项目的主线能力从“网络通信 Demo”升级为“具备可靠性、存储、分布式、可观测能力的 IM 服务器”
- 让每一个升级点都能在面试中讲清楚：为什么要做、为什么这样做、代价是什么、怎么逐步落地

---

## 2. 当前项目基线

从现有代码看，项目已经具备较好的基础骨架：

- `src/net/` 已具备 Reactor + `epoll` + `EventLoop` + `TcpServer` + `TcpConnection`
- `src/codec/` 已具备 `Buffer`、定长头协议、CRC16、JSON 序列化
- `src/service/` 已具备 `AuthService`、`MessageService`、`HeartbeatService`、`OverloadProtectService`
- `src/base/` 已具备 `ThreadPool`、`Config`、`Log`、`RedisClient`
- `server_full.cpp` 已将登录、心跳、P2P、广播、Redis、线程池拼装为完整入口

但与工业级可用 IM 相比，当前还有明显缺口：

- 协议仍以 JSON 为主，缺少版本、错误码、兼容策略
- 消息只有“发送成功/失败”概念，没有 ACK、补拉、去重、状态机
- 线程池已存在，但业务与 I/O 的职责边界还不够严格
- Redis 已接入，但更多是“能力存在”，还没有形成真正的分布式路由闭环
- MySQL、可观测性、自动化测试、限流熔断等还未形成主线能力

因此，本阶段升级不应推翻现有结构，而应遵循：

- **先稳骨架，再补主链路**
- **先做可靠性与工程闭环，再做极致性能炫技**
- **优先选择面试可讲、实现可控、价值明确的改造项**

---

## 3. 总体升级原则

## 3.1 原则一：渐进式重构，不大爆炸重写

当前项目已有较完整的模块边界，最忌讳“一次性重写”。  
更合理的方式是：

- 保留现有 `net/base/service/codec` 分层
- 在原有模块上演进接口和数据结构
- 通过“兼容旧协议 + 引入新能力”逐步切换

这样做的好处：

- 每个阶段都能保持可运行
- 更容易验证改造效果
- 文档、代码、测试、面试话术能同步积累

## 3.2 原则二：面试亮点与工程价值并重

本项目是求职项目，因此需要保留一部分“高亮技术点”，例如：

- `epoll ET`
- Protobuf
- Redis 分布式状态
- Prometheus 指标
- 限流 / 熔断

但这些亮点必须建立在真实主线之上。  
否则面试中很容易出现“名词会说，细节落不了地”的问题。

## 3.3 原则三：先形成 IM 主链路闭环

工业级 IM 的主链路是：

`连接建立 -> 登录认证 -> 建立会话 -> 发送消息 -> 服务端落盘/投递 -> 客户端 ACK -> 状态更新 -> 断线重连补拉`

如果这个闭环不成立，那么：

- 协程、零拷贝、对象池这些优化再多，也仍然只是“高性能通信 Demo”

所以本次升级，P0 的重点必须围绕这个闭环展开。

---

## 4. 分阶段实施路线

## P0：主链路升级，形成“能讲清的工业级 IM”

P0 要解决的是“主链路闭环”，包括：

- 网络层治理：非阻塞、LT/ET、Buffer、背压、慢连接
- 协议升级：Protobuf、协议头扩展、兼容策略
- 消息可靠性：ACK、去重、补拉、状态机、离线消息
- 并发模型：I/O 线程与业务线程池职责收敛
- 数据层：MySQL + Redis 的职责边界与落地结构

P0 做完后，项目应达到：

- 可以清楚讲出 IM 消息从接入到送达的完整流程
- 可以解释断线、重连、消息重复、消息丢失如何处理
- 可以解释 Redis 和 MySQL 分别在系统里承担什么职责

## P1：把单机 IM 演进为“有集群意识的系统”

P1 的重点不再是“消息能不能发”，而是“系统在压力和节点故障下如何保持可用”：

- Redis 状态中心
- 跨节点路由 / 转发
- 重连恢复
- 限流、熔断、降级
- 日志、Trace、Prometheus 指标
- 测试与工程化体系

P1 做完后，项目在面试中的定位会从：

- “我写了一个 IM 服务器”

升级为：

- “我设计并实现了一套具备单机高并发、可靠消息、状态共享、可观测能力的 IM 服务端骨架”

## 6. 当前阶段建议的交付顺序

建议严格按照下面顺序推进，而不是并行乱改：

1. 先完成协议头扩展与 Protobuf 引入
2. 再完成 ACK / 去重 / 补拉 / 消息状态机
3. 再落地 MySQL + Redis 数据层
4. 再整理线程池与 `runInLoop()` 的职责边界
5. 再做分布式状态、稳定性、可观测和工程化

原因很简单：

- 如果协议不先定，后面的 ACK、补拉、持久化表结构都会反复改
- 如果可靠性语义不先定，MySQL 表设计会失焦
- 如果主链路不先闭环，限流、监控、压测数据都缺少真实意义

---

## 7. 面试表达建议

这套升级计划最终不是为了“看起来做了很多”，而是为了沉淀一条你能稳定讲出来的主线。

建议你后续始终围绕下面这条主线来组织表达：

`Reactor 接入层 -> Protobuf 协议 -> ACK/去重/补拉 -> MySQL/Redis 状态分层 -> 线程池异步业务 -> 分布式状态中心 -> Prometheus/日志/Trace`

只要这条主线能讲顺，项目的说服力就会明显高于普通聊天室 Demo。
