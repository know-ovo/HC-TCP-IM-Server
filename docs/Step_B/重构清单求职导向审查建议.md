# 《重构清单》审查建议（求职导向版）

## 一、总体结论

这份清单整体方向是对的，而且**非常适合求职场景**。  
它的优点是：

- 主题鲜明，明确从“toy demo”往“工业级 IM”升级
- 选题覆盖高性能网络、协议设计、消息可靠性、分布式、工程化，面试可讲点很多
- 部分条目具备明显“亮点效应”，例如 `ET`、`Protobuf`、`Redis`、`Prometheus`、`Coroutines`

但如果站在“求职 + 能经得住深挖”的角度，这份清单还有三个问题：

- 有些点**适合讲，不一定适合优先做**，比如 `强制 ET`、`sendfile`、`C++20 Coroutines`
- 有些点**讲了一半**，容易被面试官继续追问，例如 ACK、去重、Redis 分布式、熔断
- 缺少几个真正能体现“工业级闭环”的内容，例如背压、重连补拉、优雅停机、数据分层存储、连接治理

因此，这份清单**不需要推翻**，而是应该做一次“求职友好化重排”：

- 保留高亮技术点，方便面试输出
- 补齐容易被追问的关键细节
- 将少数炫技项降级，避免投入过大、收益过低

## 二、哪些内容建议保留

下面这些项我认为都值得保留，而且对求职非常有帮助。

### 1. `ET + 非阻塞 I/O`

这项可以保留，因为它非常适合展示你对 `epoll` 的理解，属于高频面试题。  
但建议把“**强制开启 ET**”改成“**支持 LT/ET，并重点实现 ET 模式**”。

原因：

- 面试时你依然可以重点讲 ET
- 工程表达上更稳，不会显得结论太绝对
- 如果实现过程中 ET 细节没有完全打稳，也能保留 LT 作为稳定路径

建议补充你必须能回答的问题：

- 为什么 ET 必须配合非阻塞 I/O
- 为什么要持续读写直到 `EAGAIN`
- 如果只读一次会发生什么
- ET 下如何避免连接饥饿
- ET 与 LT 的优缺点分别是什么

### 2. `Buffer 优化`

这项也建议保留。  
因为 Buffer 是网络项目里非常适合展开细讲的模块，既能讲数据结构，也能讲性能问题。

建议表达方式从：

- 内存池或环形缓冲区二选一

改成：

- 第一阶段先做**环形缓冲区**
- 第二阶段再评估是否需要**对象池 / 内存池**

原因：

- 环形缓冲区更贴近你当前项目的实际收益
- 内存池更容易引入复杂度，求职阶段不一定要先做

### 3. `Protobuf 替换 JSON`

这项依然值得保留，因为求职里确实非常加分。  
但建议把表述从“全面扔掉 JSON”改成：

- **客户端长连接协议升级为 Protobuf**
- **调试接口 / 管理接口可继续保留 JSON**

这样更像真实工程方案，也更容易应对面试官追问“为什么不全部统一”。

你还应该补充：

- 协议版本号
- 请求 ID / 序列号
- 错误码
- 可扩展字段
- 向前 / 向后兼容策略

因为工业级协议的核心不只是“换成 Protobuf”，而是“**协议可演进**”。

### 4. `ACK + 去重 + 持久化`

这是你整份清单里最重要的一组，必须保留，而且建议强化。  
这部分是区分“聊天 demo”和“可用 IM 服务”的核心。

建议继续保留：

- ACK 机制
- sequence_id
- 去重窗口
- 离线消息持久化

但一定要补上：

- ACK 是谁回给谁：客户端 ACK 还是服务端 ACK
- 超时重试策略
- 重连后的补拉机制
- 消息状态定义：`sent`、`delivered`、`read`
- 幂等处理方案

### 5. `业务线程池`

这项必须保留，且很适合作为你项目架构升级的主线之一。

因为它能直接体现你理解了：

- I/O 线程不应该被阻塞
- 网络层与业务层解耦
- 线程间任务投递与回切
- 线程安全边界

这是求职时非常容易讲清楚、也容易获得认可的一项。

### 6. `Redis / 可观测性 / 限流熔断 / 工程化`

这几项都建议保留。

因为它们能分别覆盖下面几类高频问题：

- Redis：分布式、高可用、状态共享
- Prometheus + 日志：线上问题排查
- 限流 / 熔断：稳定性设计
- 测试 + clang-tidy + benchmark：工程化能力

这些内容组合起来，会让你的项目不像“只会写网络代码”，而是像“有完整后端工程意识”。

## 三、哪些内容建议补充

这部分是我认为你当前清单里最缺的内容。  
这些点未必都像 ET、Protobuf 那样“耀眼”，但它们非常重要，而且能显著提升你的回答可信度。

### 1. 背压（Backpressure）与慢连接治理

这是高性能长连接服务里非常关键的一环，你现在清单里没有体现。

建议补充：

- 发送缓冲区高水位 / 低水位
- 写队列积压时的处理策略
- 慢客户端识别与踢除
- 单连接最大缓存限制
- 单包大小限制

为什么必须补：

- 面试官很可能会问：如果客户端读得很慢怎么办
- 如果你没有背压策略，系统在高峰期很容易内存暴涨

### 2. 重连补拉与会话恢复

你写了 ACK 和去重，但没有写“断线重连后怎么办”。  
IM 系统这是必问项。

建议补充：

- 客户端带上上次收到的最大 `seq`
- 服务端根据 `seq` 拉取未确认消息
- 恢复未读状态或补发离线消息
- 多端登录时的同步策略

### 3. 数据分层存储

你现在写的是 `SQLite 或 RocksDB`，但如果目标是求职中的“工业级 IM”，建议改成更符合主流后端认知的方案。

建议补成：

- `MySQL`：用户、好友、群组、离线消息元数据、ACK 状态
- `Redis`：在线状态、路由表、短期缓存、分布式 session
- `RocksDB / 本地 WAL`：作为加分探索项，而不是主线

这样更贴近企业实际，也和你后续接 MySQL 的计划一致。

### 4. 连接生命周期治理

建议补充：

- 连接建立、认证、活跃、超时、关闭的完整状态流转
- 登录前后权限边界
- 未认证连接超时踢除
- 异常断开后的清理动作

这能体现你不是只关注“收发消息”，而是理解“服务端连接管理”的完整生命周期。

### 5. 优雅停机与配置热更新

工业级项目经常会被问：

- 服务升级时如何不中断用户
- 配置改了是否必须重启

建议补充：

- graceful shutdown
- 停止接收新连接
- 等待已有请求处理完成
- 配置热加载（日志级别、限流阈值、心跳参数）

### 6. 安全性补全

你目前安全性部分偏少，只写了密码哈希和 TLS。  
建议补：

- token 过期与刷新
- 登录限速
- 重放攻击防护
- 敏感日志脱敏
- 鉴权失败审计日志

这样在面试里会显得更完整。

## 四、哪些内容建议降级或改写

这部分不是说不能做，而是说**不建议作为当前阶段的高优先级主线**。

### 1. `sendfile`

建议从 `P0` 降到 `P2` 或“扩展项”。

原因：

- IM 核心场景主要是小包消息，不是大文件传输
- 如果你当前还没有文件传输子系统，这项落地价值有限
- 但它依然可以保留作为“性能优化探索项”，面试时也能讲

更合适的表述：

- 如果后续增加文件传输能力，可引入 `sendfile` 进行零拷贝优化

### 2. `强制 ET`

建议改写，不建议继续使用“强制”这个词。

原因：

- 面试里可以高调，文档里最好稳一点
- ET 容易引入细节 bug，如果还没完全打磨好，表述太绝对反而危险

建议改成：

- 支持 LT/ET 两种模式，重点实现 ET 路径，并总结其适用场景

### 3. `全面替换 JSON`

建议改成“面向核心链路的协议升级”，而不是“一刀切替换”。

原因：

- 管理接口、调试接口继续用 JSON 很合理
- 工程上分层使用不同协议并不丢分，反而更真实

### 4. `C++20 Coroutines 重写异步逻辑`

建议降到 `P2` 甚至 `P3`。

原因：

- 协程确实能加分，但实现成本很高
- 你当前项目更需要先把消息可靠性、持久化、线程模型、连接治理做扎实
- 如果协程只做了一半，面试里很容易被深入追问到挂

更推荐的写法：

- 预留基于 C++20 协程重构异步逻辑的演进方向，当前优先完成线程池 + Reactor 的稳定实现

## 五、推荐的重排版本

下面是我认为更适合你当前“求职导向”目标的优先级方案。

## P0：最值得优先完成的部分

### 1. 网络层

- 支持非阻塞 I/O
- 支持 LT/ET 两种事件模式，重点实现 ET
- Buffer 升级为环形缓冲区
- 增加发送缓冲区高水位、慢连接治理、背压控制

### 2. 协议与编解码

- 客户端长连接协议从 JSON 升级到 Protobuf
- 协议头补充版本号、请求 ID、序列号、错误码
- 明确协议扩展策略与兼容策略

### 3. 消息可靠性

- ACK 机制
- sequence_id
- 去重窗口
- 重连补拉
- 离线消息持久化
- 消息状态机

### 4. 并发模型

- I/O 线程只负责网络事件
- 业务线程池负责数据库和耗时逻辑
- 通过 `runInLoop()` 回切发送结果

### 5. 数据层

- 引入 MySQL 持久化核心业务数据
- 引入 Redis 管理在线状态和 session

## P1：显著增强项目深度的部分

### 1. 分布式与高可用

- Redis 用户状态中心
- 跨节点会话共享
- 简单的消息路由 / 转发机制
- 节点故障后的重连恢复策略

### 2. 稳定性治理

- 令牌桶限流
- 单接口 QPS 统计
- 熔断与降级
- 心跳超时与空闲连接清理

### 3. 可观测性

- 结构化日志
- Trace ID
- Prometheus 指标
- P99 延迟、在线人数、TPS、失败率监控

### 4. 工程化

- GTest 单测
- clang-format
- clang-tidy
- Google Benchmark

## P2：高亮加分项

- 对象池 / 内存池
- 零拷贝 `sendfile`
- C++17/20 特性升级
- 协程化改造
- 更进一步的分布式解耦（如 Redis Stream / MQ）

## 六、最终建议

如果你的核心目标是**求职**，那么这份清单的正确策略不是“只做最真实的工业方案”，也不是“只做最炫的技术点”，而是：

- 做出一条**稳定可运行的主线**
- 在这条主线之上嵌入几个**高质量的面试亮点**
- 保证每一个写在简历里的点，你都能讲原理、讲权衡、讲落地细节

换句话说，最优解不是“做最多”，而是：

**做那些既能跑起来、又能讲明白、还能经得住追问的升级项。**

对你这份清单来说，我的最终判断是：

- **方向正确，值得继续推进**
- **不需要删很多，但需要重排优先级**
- **必须补齐背压、重连补拉、数据分层、连接治理这几个关键缺口**
- **应当把 `sendfile`、`强制 ET`、`协程重写` 这类炫技项从主线降级为加分项**

如果后续继续推进，建议你就按下面这条主线实施：

`Protobuf 协议升级 -> ACK/去重/补拉 -> MySQL + Redis -> 线程池解耦 -> 背压治理 -> 可观测性 -> 分布式增强 -> 高级性能优化`
