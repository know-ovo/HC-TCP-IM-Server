# 第一层：先看网络底座

这一层不需要先关心 IM 业务。你只要把它当作“一个 Reactor 网络库”去理解就行。

回答 5 个问题：

## 1.监听 socket 在哪里创建，连接在哪里 accept？

**监听socket** 在 **TcpConnection** 的 **构造函数** 中创建

**连接accept**在**TcpServer**的**newConnection()函数**中执行



## 2.一个连接的读事件、写事件、关闭事件分别怎么分发？

读、写、关闭事件 完整分发流程

### 核心原理

**EpollPoller（内核事件监听）→ EventLoop（循环调度）→ Channel（回调分发）→ TcpConnection（业务处理）**

ps:

eventloop先创建，同时创建epollpoller

TcpServer后创建，同时创建channel，然后为channel设置回调函数（设置监听事件）

------

### ① 读事件分发（有数据到达）

1. 内核检测到 socket **有数据可读**

2. **EpollPoller** 通过 `epoll_wait` 捕获读事件，填充对应 Channel

3. EventLoop 拿到活跃 Channel，调用 `channel->handleEvent()`

4. Channel 触发

   读回调，执行你注册的：

   ```
   m_channel->setReadCallback([this](Timestamp t) { handleRead(t); });
   ```

5. 最终进入 **TcpConnection::handleRead()** 处理数据

------

### ② 写事件分发（可发送数据）

1. 调用 `conn->send()` 发送数据，内核缓冲区满→数据存入**输出缓冲**

2. 调用 `m_channel->enableWriting()` 开启可写事件监听

3. 内核通知 socket **可写**，EpollPoller 捕获可写事件

4. EventLoop 调度，Channel 触发

   写回调，执行你注册的：

   ```
   m_channel->setWriteCallback([this]() { handleWrite(); });
   ```

5. 最终进入 **TcpConnection::handleWrite()** 发送缓冲区数据

------

### ③ 关闭事件分发（对端断开 / 异常）

1. 两种触发场景：

   - 对端关闭连接，`handleRead` 读到 **n == 0**，主动触发关闭
   - 网络异常，EpollPoller 捕获关闭 / 错误事件

2. EventLoop 调度，Channel 触发

   关闭回调，执行你注册的：

   ```
   m_channel->setCloseCallback([this]() { handleClose(); });
   ```

3. 最终进入 **TcpConnection::handleClose()** 关闭连接



## 3.`EventLoop` 和 `Channel` 分别负责什么？

### ① EventLoop 负责什么？

它是**Reactor 模型的大脑**，管整个事件流程，不关心具体某个连接：

1. **跑主循环**：`loop()` 里无限调用 `epoll_wait` 等待事件
2. **调度事件**：从 EpollPoller 拿到活跃 Channel，挨个调用 `channel->handleEvent()`
3. **管理 Channel**：提供 `updateChannel/removeChannel`，帮 Channel 注册 / 注销监听
4. **跨线程任务**：`runInLoop/queueInLoop` 支持别的线程往本循环丢任务
5. **线程绑定**：一个 EventLoop 只属于一个线程，保证所有 IO 操作线程安全
6. **唤醒机制**：用 `eventfd` 唤醒阻塞的 epoll_wait，及时执行任务

------

### ② Channel 负责什么？

它是**某个 fd（socket / 唤醒 fd）的专属代理人**，一个 Channel 只管一个 fd：

1. **绑定 fd**：只负责一个文件描述符（如 TCP 连接的 socket、EventLoop 的 wakeupfd）
2. **记录监听事件**：标记要监听**读 / 写 / 关闭 / 错误**哪种事件
3. **保存回调**：存储你注册的 `handleRead/handleWrite/handleClose` 等函数
4. **分发事件**：`handleEvent()` 根据内核触发的事件，调用对应的回调
5. **更新监听**：`enableReading/enableWriting` 等，通知 EventLoop 修改 epoll 监听

------

### ③  极简记忆（面试直接背）

- **EventLoop**：**管循环、管调度、管全局**，驱动整个 Reactor
- **Channel**：**管单个 fd、管事件、管回调**，只做事件分发
- **配合关系**：Loop 负责 “等事件 + 调 Channel”，Channel 负责 “存回调 + 执行回调”



## 4.TcpConnection里输入缓冲和输出缓冲怎么工作？

代码里两个关键成员：

```
Buffer m_inputBuffer;   // 读缓冲区（收数据）
Buffer m_outputBuffer;  // 写缓冲区（发数据）
```

### ① 输入缓冲（收数据）

工作流程：

1. 读事件触发 → `handleRead()`
2. 调用 `m_inputBuffer.readFd()` 把**内核 socket 数据读到内存缓冲区**
3. 一次性读干净，避免多次系统调用
4. 读完后，调用用户设置的 **`m_messageCallback`** 把缓冲区交给业务层处理

作用：

- 解决**粘包分包**
- 让业务层可以**按需读取**，不用关心底层 TCP 流式数据

### ② 输出缓冲（发数据）

工作流程：

1. 用户调用 `conn->send()`
2. 先尝试**直接调用 write/send 发数据**
3. 如果**内核缓冲区满了**，发不完 → **剩余数据放入 m_outputBuffer**
4. 开启**可写事件监听**
5. 等可写事件触发，`handleWrite()` 再从缓冲区慢慢发
6. 发完后关闭可写监听，触发发送完成回调

作用：

- 实现**非阻塞发送**

- 避免发送过快导致数据丢失

- 支持**跨线程安全 send**

  

## 5.高水位、低水位、背压是什么意思，为什么需要它？

### ① 高水位 HighWaterMark

- 定义：**输出缓冲**的**预警上限**（默认 1MB）
- 代码位置：`sendInLoop()`
- 触发：缓冲数据 ≥ 1MB 时
- 作用：**提醒 “发送太快，接收方处理不过来了”**

### ② 低水位 LowWaterMark

- 定义：**输出缓冲**的**恢复安全线**（默认 256KB）
- 代码位置：`handleWrite()`
- 触发：缓冲数据 ≤ 256KB 时
- 作用：**告诉系统 “压力缓解，可以继续正常发送”**

### ③ 背压 BackPressure

- 就是一个**状态**：`m_backPressured = true`
- 表示：**发送方速度 > 接收方处理速度**
- 开启：达到高水位
- 解除：降到低水位

------

### 为什么必须要这三个机制？

#### 不背压会发生什么灾难？

- 客户端网速慢 / 处理慢 **——> **服务端疯狂 send **——> **输出缓冲**无限膨胀** **——>**内存爆了 **——>** 程序崩溃

#### 背压机制的作用

1. **保护服务器内存**，不让缓冲无限增大
2. **告诉业务层放慢发送速度**
3. **防止程序 OOM 崩溃**
   Out Of Memory（内存耗尽）崩溃:你的程序**把系统能给的内存全部用光了**，没有多余内存可用，操作系统为了保护整机不卡死，会**直接强制杀死你的程序**，这就叫 **OOM 崩溃**。
4. 让高并发系统**稳定、不雪崩**

------

### 极简总结（必背）

1. **事件分发**：Loop 监听 → Channel 回调 → TcpConnection 处理
2. **输入缓冲**：收数据，解决粘包，给业务层使用
3. **输出缓冲**：发数据，非阻塞，缓存发不完的内容
4. **高水位**：缓冲太大，预警
5. **低水位**：缓冲变小，恢复
6. **背压**：防止发送过快撑爆内存

## 第二层：看协议和编解码

这一层要解决的问题是：**TCP 只负责传字节流，但 IM 系统需要知道“这是什么消息、是谁发的、该怎么回、失败算什么、重连后从哪里继续”**。  
所以协议层本质上是在给“字节流”补业务语义。

## 1. 协议头里有哪些字段，为什么要有 `command`、`request_id`、`clientSeq`、`serverSeq`、`errorCode`、`checksum`

项目当前协议头定义在 `src/codec/protocol.h` 的 `PacketHeader`，固定 44 字节，字段如下：

```cpp
struct PacketHeader
{
    uint32_t magic;
    uint16_t version;
    uint16_t headerLen;
    uint32_t totalLen;
    uint16_t command;
    uint16_t flags;
    uint32_t requestId;
    uint64_t clientSeq;
    uint64_t serverSeq;
    uint32_t errorCode;
    uint32_t checksum;
};
```

先说完整认知：**协议头不是“为了凑字段”，而是为了让服务端和客户端对一条消息拥有一致的解释权。**

### ① `command` 为什么必须有？

它回答的是：**这包数据到底是什么业务语义。**

- 没有 `command`，服务端拿到 payload 后根本不知道这是登录、心跳、单聊、ACK 还是离线拉取
- 有了 `command`，`server_full.cpp` 就可以按命令字分发到不同逻辑
- 它相当于“业务路由键”

例如：

- `CmdLoginReq` 表示登录请求
- `CmdP2pMsgReq` 表示单聊发消息
- `CmdMessageAckReq` 表示 ACK
- `CmdPullOfflineReq` 表示补拉离线消息

**一句话记忆**：`command` 决定“这包该交给谁处理”。

### ② `request_id` 为什么要有？

它解决的是：**请求和响应如何配对。**

TCP 是流，不保证“你发的第 3 个请求一定第 3 个回来”；而且服务端很多逻辑是线程池异步处理的，响应返回顺序可能变化。

所以必须有 `request_id`：

- 客户端发请求时带一个唯一请求号
- 服务端回包时把同一个 `request_id` 带回去
- 客户端据此知道“这个响应对应我刚才哪一次请求”

这个项目里 `sendPacket()` 回响应时就会把收到的 `requestId` 原样带回。

**一句话记忆**：`request_id` 解决“异步场景下，响应到底是回给谁的”。

### ③ `clientSeq` 为什么要有？

它解决的是：**客户端侧发送顺序、重试幂等、重复提交识别。**

为什么 `request_id` 不够，还要 `clientSeq`？

- `request_id` 更偏“一次请求-一次响应”的调用链语义
- `clientSeq` 更偏“客户端消息流”的顺序语义

它的价值主要在可靠消息链路：

- 客户端重试时，可以用 `clientSeq` 或配合 `client_msg_id` 判断是不是重复消息
- 如果客户端要表达“这是我发出的第 N 条消息”，就需要稳定的客户端侧序号
- 协议升级时，这个字段能为去重、重放保护、消息有序处理预留空间

当前代码里 `Codec::pack()` 的简化打包会先把 `seqId` 同时写进 `requestId` 和 `clientSeq`，本质上说明：**项目已经把“调用标识”和“客户端序列”分开建模了，只是部分链路还在复用同一个值。**

**一句话记忆**：`clientSeq` 解决“客户端这边这条消息在自己的发送历史里是谁”。 

### ④ `serverSeq` 为什么要有？

它解决的是：**服务端投递顺序、ACK 进度、断线补拉起点。**

这是 IM 可靠投递里最关键的字段之一。

你可以把它理解成：**站在接收方视角，服务端给消息发的“投递游标”**。

为什么必须有它？

- 客户端 ACK 时，要告诉服务端“我确认到哪一条了”
- 客户端断线重连后，要告诉服务端“我上次确认到 `serverSeq=x`，后面的你补给我”
- 服务端做离线拉取时，不能只说“给我最近 20 条”，而是要说“给我 `lastAckedSeq` 之后的消息”

这个项目里：

- P2P 发送成功后，服务端会返回 `serverSeq`
- ACK 请求里会带 `serverSeq`
- 离线拉取请求 `PullOfflineReq` 里会带 `lastAckedSeq`

所以后面的 ACK、重试、补拉，全部都是围绕 `serverSeq` 运转的。

**一句话记忆**：`serverSeq` 解决“服务端已经投递到哪里，客户端已经确认到哪里”。 

### ⑤ `errorCode` 为什么要有？

它解决的是：**失败语义标准化。**

如果没有统一错误码，就会出现这些问题：

- 只能看字符串 `"invalid request"`、`"server busy"`，程序不好处理
- 客户端无法根据不同失败类型做不同动作
- 日志、监控、告警没法统一统计

有了 `errorCode` 后：

- `ErrAuthFailed` 可以明确表示鉴权失败
- `ErrServerBusy` 可以明确表示过载拒绝
- `ErrChecksumMismatch` 可以明确表示包损坏

这样客户端就能区分：

- 这是可以重试的错误
- 这是需要重新登录的错误
- 这是协议包本身有问题

**一句话记忆**：`errorCode` 解决“失败不能只靠人读字符串，要让机器也能稳定理解”。 

### ⑥ `checksum` 为什么要有？

它解决的是：**包体完整性校验。**

TCP 能保证“按顺序送达字节流”，但应用层仍然需要知道：

- 我解析出来的 payload 是不是完整的
- 数据有没有损坏
- 我是不是把错误长度的内容当成了一条合法消息

所以项目里在打包时会对 payload 计算校验值，写进 `checksum`；解包时重新计算一遍，如果不一致，就判定为坏包并走错误处理。

它不是安全加密，但它能挡住最基本的“包损坏 / 错误解析”问题。

**一句话记忆**：`checksum` 解决“这包内容是不是完整、是不是被破坏了”。 

### ⑦ 面试时怎么总结这几个字段

可以直接这样讲：

> `command` 解决消息分发，`request_id` 解决请求响应配对，`clientSeq` 解决客户端侧顺序和去重，`serverSeq` 解决服务端投递进度和断线补拉，`errorCode` 解决失败语义标准化，`checksum` 解决包体完整性校验。  
> 这些字段合起来，才让协议从“能传数据”升级到“能表达可靠消息语义”。

## 2. `Codec` 怎么处理粘包拆包

先记一个前提：**TCP 没有消息边界，只有字节流。**

所以会出现两类问题：

- **粘包**：一次读，读到了多个业务包
- **半包**：一次读，只读到一个包的一部分

`Codec` 的任务就是：**从连续字节流里，稳定切出一包一包的完整消息。**

### ① 它靠什么切包？

靠协议头里的两个字段：

- `headerLen`
- `totalLen`

其中真正决定一整包边界的是 `totalLen`。

意思是：

- 先读固定头
- 从头里拿到这包总长度
- 如果缓冲区数据还没攒够，就继续等
- 攒够了再一次性取出完整包

这就是经典的**长度字段帧协议**。

### ② `Codec` 的解析状态机

`codec.h` 里定义了两个状态：

- `kExpectHeader`
- `kExpectPayload`

对应逻辑可以直接理解成两步：

1. 先等头
2. 再等整包

### ③ 实际解析流程

`onMessage()` 会一直 `while (true)` 循环解析缓冲区：

#### 第一步：头不够，就先不动

如果当前可读字节 `< 44`（协议头长度），直接退出，等待下一次网络数据到达。

这就解决了**半包头**问题。

#### 第二步：头够了，就先解析头

`parseHeader()` 会做几件事：

- 读取 `magic`，确认是不是本协议
- 读取 `version`，确认协议版本是否支持
- 读取 `headerLen` / `totalLen`，确认长度是否合法

如果这些检查不过：

- 说明不是合法包，或者长度异常
- 直接清空缓冲区并报错

这一步本质上是在防止：

- 错包
- 脏数据
- 恶意超大包

#### 第三步：知道总长度后，等整包到齐

头解析完后，`m_expectedLength = totalLen`，状态切到 `kExpectPayload`。

如果当前缓冲区还不到 `totalLen`，继续退出等待。

这就解决了**半包 body**问题。

#### 第四步：整包到齐后，再校验并取出

`parsePayload()` 会：

- 根据 `totalLen - headerLen` 算出 payload 长度
- 对 payload 重新计算 `checksum`
- 和头里的 `checksum` 对比

如果校验通过：

- 组装成 `DecodedPacket`
- `buffer->retrieve(m_expectedLength)` 把这一个完整包从缓冲区移走
- 回调上层业务处理

如果校验失败：

- 说明这包内容不可信
- 丢弃这包，恢复到等头状态

### ④ 为什么它能同时解决粘包和拆包？

因为它不是“看到数据就立刻 parse”，而是：

- **不够一包就等**
- **够一包只取一包**
- **缓冲区里还有剩余数据就继续 while 循环**

所以：

- 一次读到半包：先留在缓冲区，下次补齐
- 一次读到 3 包：循环切 3 次
- 一次读到 1.5 包：先切完整的一包，半包留下次再补

### ⑤ 极简总结

`Codec` 处理粘包拆包的核心不是“特殊技巧”，而是三件事：

1. **固定头**
2. **头里带总长度**
3. **缓冲区 + 状态机循环解析**

## 3. JSON 和 Protobuf 在这个项目里的职责分别是什么

这个问题不能只背概念，要分成“设计职责”和“当前落地状态”两层来看。

### ① 设计职责

从设计上，这个项目想走的是**双轨制**：

- **JSON**：用于调试友好、可读性强的链路
- **Protobuf**：用于更正式、更高效、更易演进的长连接协议链路

为什么要这样分？

因为两者解决的问题不一样。

#### JSON 的价值

JSON 的优势是：

- 肉眼可读
- 抓包和日志时容易排查
- Python / shell / Postman / 手写脚本都容易接
- 开发早期调试成本低

所以它特别适合：

- 登录、心跳、P2P 等功能开发初期
- 调试脚本
- 问题排查
- 管理类接口

#### Protobuf 的价值

Protobuf 的优势是：

- 体积更小
- 编解码更快
- 强类型，更不容易字段写错
- 对协议版本演进更友好

所以它更适合：

- 高频主链路
- 长连接正式协议
- 需要明确 schema 的跨端通信

### ② 当前这个项目里实际怎么落地？

**当前实际业务收发链路，主用的还是 JSON。**

证据很直接：

- `server_full.cpp` 里业务消息解析和回包都在调用 `serializer::Serialize/Deserialize`
- `messageSerializer.cpp` 里是基于 `nlohmann/json` 做的序列化
- 消息存储里 `payloadFormat` 默认也还是 `Json`

也就是说，目前项目的现状是：

- **JSON 已经真正接进主流程**
- **Protobuf 已经完成了协议抽象和基础准备，但还没有全面替换主链路**

### ③ Protobuf 在当前项目中承担什么角色？

虽然还没有全面接管收发，但它不是“没用”，而是在承担**协议演进预留层**的角色：

- `protocol.h` 已经定义了 `PayloadFormat`
- `ProtoCodec` 可以解析配置里的 `payload_format`
- `proto/common.proto` 已经定义了 `BaseRequest` / `BaseResponse`

这说明项目的设计思路是：

- 先把“协议头、消息边界、命令字、序列号、错误码”这些骨架搭好
- 再逐步把 payload 从 JSON 迁移到 Protobuf

换句话说，**现在的重点是先把“传什么语义”稳定下来，再把“payload 用什么格式编码”升级掉。**

### ④ 面试里怎么讲最稳？

可以直接这样说：

> 这个项目当前主链路实际还是 JSON，因为 JSON 可读性好、开发和排障效率高；  
> 但协议层已经抽象了 `PayloadFormat`，并预留了 Protobuf 能力，目的是后续把高频长连接消息迁移到更高效、更强类型的编码格式。  
> 所以 JSON 负责“当前可用和调试效率”，Protobuf 负责“未来性能和协议演进”。

## 4. 为什么 `Step_B` 的第一步先升级协议，而不是先做 Redis/MySQL

这个问题是系统设计题，核心答案是：

**因为 Redis/MySQL 解决的是“存”和“共享状态”，协议解决的是“消息语义能不能被正确表达”。**

如果协议层没升级，后面很多能力根本没有可靠落点。

### ① 先做 Redis/MySQL，为什么不够？

假设你一上来就接 MySQL 和 Redis，会马上遇到这些问题：

- 消息入库时，怎么区分这是登录、P2P、ACK 还是补拉？
- 怎么知道一条响应对应哪个请求？
- 怎么表达客户端重试是不是重复发送？
- 怎么知道客户端 ACK 到哪一条了？
- 重连后从哪个位置开始补拉？
- 失败时客户端该重试、重登还是忽略？

这些问题都不是存储本身能解决的，而是**协议字段先要表达得出来**。

也就是说：

- 没有 `request_id`，请求响应配不起来
- 没有 `serverSeq`，ACK 和补拉没锚点
- 没有 `errorCode`，失败语义不统一
- 没有 `command`，消息类型都分不清

存储层再强，也只是把一堆“语义不完整的数据”存下来了。

### ② 为什么协议是“地基”？

因为后面所有高级能力，本质上都依赖协议字段：

- **ACK** 依赖 `serverSeq`
- **重试/幂等** 依赖 `clientSeq`、`client_msg_id`
- **补拉** 依赖 `lastAckedSeq/serverSeq`
- **trace** 依赖 `request_id`
- **统一错误处理** 依赖 `errorCode`
- **版本演进** 依赖 `version`

所以协议层不是“小细节”，而是整个 Step_B 的骨架。

### ③ 为什么升级顺序必须是“先表达，再存储”？

因为工程上有一个基本原则：

**先定义对象语义，再决定怎么存；先定义交互契约，再决定怎么扩。**

不然你后面会出现很痛苦的返工：

- 数据表设计要重改
- ACK 逻辑要重写
- 客户端接口要重改
- 历史数据兼容会很难

而先把协议定好，后面的 Redis/MySQL 才知道自己到底要存什么、同步什么、恢复什么。

### ④ 这个项目里为什么尤其要先做协议？

因为 Step_B 的目标已经不是“能聊天”了，而是：

- 可靠消息
- ACK
- 离线补拉
- 持久化
- 分布式路由
- trace 和 metrics

这些能力全都需要一个**比 Step_A 更强的协议头**。

Step_A 那种“长度 + 命令 + 简单消息体”的协议，够做演示版；
但到了 Step_B，系统已经开始追求：

- 可恢复
- 可追踪
- 可扩展
- 可治理

那协议就必须先升级。

### ⑤ 面试时一句话讲法

> 我们先升级协议，不是因为协议最炫，而是因为它是后续可靠消息、ACK、补拉、存储、分布式路由的语义地基。  
> Redis/MySQL 解决的是“把东西放在哪里”，协议先解决的是“这条消息到底代表什么、系统怎么理解它”。

## 第二层极简总结

1. **协议头的本质**：给字节流补业务语义
2. **`command`**：决定消息分发
3. **`request_id`**：配对请求和响应
4. **`clientSeq`**：标识客户端侧顺序和去重语义
5. **`serverSeq`**：标识服务端投递进度，是 ACK/补拉锚点
6. **`errorCode`**：标准化失败语义
7. **`checksum`**：做完整性校验
8. **`Codec`**：靠“固定头 + 总长度 + 状态机 + 缓冲区”解决粘包拆包
9. **JSON**：当前主用，便于开发调试
10. **Protobuf**：协议演进方向，服务于性能和强类型
11. **先升级协议**：因为后面的存储、ACK、补拉都依赖协议字段先表达清楚
