# M3 系统讲解文档：持久化分层与线程边界收敛

## 1. 阶段目标

M3 不是简单继续“加功能”，而是把 M2 已经跑通的可靠消息链路，从“集中式 store 实现”继续推进成“可扩展的数据层结构 + 明确的线程边界”。

本阶段最终完成了这些目标：

1. 拆出 `messageRepository`
2. 拆出 `deliveryRepository`
3. 拆出 `sessionRepository`
4. 让 `MySqlMessageStore` 委托 repository
5. 让 `AuthService` 接入 MySQL session 持久化
6. 让 ACK 推进回写 `session.last_acked_seq`
7. 让登录、发消息、ACK、补拉、登出回到业务线程池执行
8. 保证最终网络发送仍回切到连接所属 `EventLoop`
9. 收敛 Redis 职责并补齐并发安全保护

这意味着系统已经从“服务类 + SQL 混写”走向“服务编排 + 仓储持久化 + I/O/业务线程分治”的结构。

---

## 2. 为什么 M3 必须跟上

M2 已经把可靠消息语义闭环跑通，但如果继续在 `MySqlMessageStore` 里直接堆 SQL，会越来越难维护：

- 消息主表逻辑和投递表逻辑耦合
- ACK 日志与状态更新混在一起
- `AuthService` 的会话状态仍然主要停留在内存/Redis
- 后续想做连接池、指标、事务边界优化会越来越痛苦

所以 M3 的目标不是“功能翻倍”，而是“把代码结构整理到能继续长下去”。

---

## 3. 本阶段的核心改动

### 3.1 新增 Repository 层

新增文件：

- `src/storage/messageRepository.h`
- `src/storage/messageRepository.cpp`
- `src/storage/deliveryRepository.h`
- `src/storage/deliveryRepository.cpp`
- `src/storage/sessionRepository.h`
- `src/storage/sessionRepository.cpp`

职责拆分如下：

- `MessageRepository`：负责 `im_message`
- `DeliveryRepository`：负责 `im_message_delivery` 与 `im_message_ack_log`
- `SessionRepository`：负责 `im_session`

这样的拆分方式，和表语义是一一对应的，后面扩展会更自然。

### 3.2 `MySqlMessageStore` 从“直接拼 SQL”改为“委托 repository”

本阶段重构后：

- 消息创建交给 `MessageRepository`
- 投递创建/状态更新/ACK 日志交给 `DeliveryRepository`
- `MySqlMessageStore` 自己只保留可靠消息编排相关逻辑

也就是说，`MySqlMessageStore` 不再是“大一统数据访问类”，而更像一个面向 `MessageService` 的领域适配层。

### 3.3 新增 `im_session` 会话表并接入 ACK 恢复点

在 `sql/001_m2_reliable_message.sql` 中新增：

- `im_session`

其核心字段包括：

- `session_id`
- `user_id`
- `device_id`
- `server_node_id`
- `last_acked_seq`
- `created_at_ms`
- `expires_at_ms`

这一版 `im_session` 还不是完整用户中心，但已经具备两个关键价值：

- 会话不再只存在于进程内存
- `last_acked_seq` 开始进入持久化语义

### 3.4 `AuthService` 接入 `SessionRepository`

本阶段新增：

- `AuthService::setSessionRepository()`

并让登录、登出、校验会话三条链路支持 MySQL 会话仓储：

- 登录时写入 `im_session`
- 登出时删除 `im_session`
- `validateSession()` 时优先查询 MySQL
- ACK 成功后回写 `last_acked_seq`

这一步的意义很大：  
从现在开始，认证链路不再只依赖“当前进程记忆”。

### 3.5 线程边界真正落到入口层

本阶段把 `server_full.cpp` 里的重操作真正切到了业务线程池：

- 登录
- 可靠消息发送
- ACK 处理
- 离线补拉
- 广播
- 登出

这样当前线程模型被明确收敛为：

- I/O 线程：收包、解包、连接回调、最终发包
- 业务线程：认证、Redis、MySQL、可靠消息状态推进

虽然 `conn->send()` 本身已经具备回切能力，但 M3 的关键不是“碰巧线程安全”，而是“入口层明确知道哪些逻辑属于业务线程”。

### 3.6 Redis 职责边界与并发安全收敛

M3 对 Redis 做了两层收敛：

1. 职责收敛
   - `AuthService` 只负责 session 相关 Redis key
   - `HeartbeatService` 只负责 online 状态 Redis key

2. 并发安全收敛
   - `RedisClient` 增加互斥锁保护
   - 避免业务线程池与其他线程并发使用同一 hiredis context 时出现数据竞争

这一步不是炫技，而是为了让“Redis 进入业务线程池”后仍然可控。

## 4. 设计取舍

### 4.1 为什么先拆 repository，再补入口层线程边界

因为当前最突出的结构问题在数据层，而不是线程模型。

如果现在就一口气推进：

- 业务线程池职责收敛
- I/O 线程边界梳理
- repository
- connection pool

改动面会太大，风险也会显著上升。

所以 M3 的实际顺序是：

1. 先把 Repository 立起来
2. 再把业务入口切到线程池

这样改动路径更稳。

### 4.2 为什么 `MySqlMessageStore` 还保留一层

因为 `MessageService` 依赖的是 `IMessageStore` 语义，而不是直接依赖 Repository。  
Repository 解决的是“表怎么访问”，Store 解决的是“可靠消息语义怎么组合”。

这两层职责并不冲突。

### 4.3 为什么 `SessionRepository` 先做最小可用版，而不是完整用户域

因为当前目标是先把会话持久化接进主链路，而不是立刻做完整用户域模型。  
先让 session 有表、有查询、有删除，再逐步扩展到更多节点/路由语义，是更自然的演进顺序。

### 4.4 为什么不让业务线程直接写 socket

因为这会破坏 Reactor 的线程模型。  
M3 的原则非常明确：

- 业务线程只做业务计算与状态推进
- 最终连接发送必须回到连接所属的 `EventLoop`

这条边界如果不立住，后面 M4 / M5 做稳定性和观测时会越来越乱。

本阶段已完成验证：

1. `ReadLints` 无新增诊断。

5. 已以 `build_linux_m3/bin/server_full` 启动新版服务。

回归结果说明：

- M3 重构没有破坏 M2 的在线投递链路
- 没有破坏 ACK 链路
- 没有破坏离线补发与主动补拉

也就是说，这一轮是一次“结构升级成功，行为未回退”的完整重构。

## 7. 面试讲解建议

讲 M3 时，不要只说“我又加了几个 repository”。  
更好的说法是：

1. M2 已经跑通，但数据访问逻辑和线程职责开始集中膨胀。
2. 为了支撑后续连接池、路由、可观测和多表扩展，我先按表语义做仓储拆分。
3. Store 保留领域语义，Repository 负责落库细节。
4. 同时我把登录、发消息、ACK、补拉这些重操作切到了业务线程池。
5. 最终发送仍回到 `EventLoop`，保证 Reactor 线程模型不被破坏。
6. 我还做了回归验证，确认 M2 主链路没有回退。

这样能体现你不是在“机械分层”，而是在为后续系统演进做结构治理。

---

## 8. 下一步建议

M3 完成后，最自然的下一阶段就是 M4：

1. Redis 路由表
2. 跨节点消息转发
3. 限流与熔断
4. 连接治理进一步增强

也就是说，M1 解决地基，M2 解决可靠语义，M3 解决结构和线程边界，  
接下来 M4 才开始进入真正的“多节点与稳定性增强”。
