# M2 系统讲解文档：消息可靠性闭环

## 1. 阶段目标

M2 的目标，是把项目从“在线即时转发 demo”升级成“具备基础可靠语义的 IM 服务”。

这一步真正解决的问题不是“能不能发消息”，而是：

1. 消息有没有全局标识。
2. 消息是否可去重。
3. 接收方是否真的收到。
4. 离线用户回来后能不能补到消息。
5. 服务重启后消息状态是否还在。

如果 M1 解决的是“协议和网络能不能承载更复杂系统”，  
那么 M2 解决的就是“消息语义能不能闭环”。

---

## 2. 为什么 M2 是核心升级

一个 IM 服务如果只有在线转发，没有可靠性闭环，就只能算演示样例。

真实系统至少要回答这些问题：

- 客户端重复点击发送怎么办？
- 服务发给接收方后，对方没回 ACK 怎么办？
- 用户离线时消息放哪？
- 用户重新登录时如何继续补发？
- 怎么区分“已持久化”“已投递”“已送达”“已读”“失败”？

M2 的价值就在于把这些问题从“业务层 if/else”提升成一套统一机制。

---

## 3. 核心设计

### 3.1 三个关键标识

M2 引入了三类关键标识：

- `message_id`：服务端消息唯一标识
- `client_msg_id`：客户端幂等消息标识
- `server_seq`：接收方维度的递增序列

它们分别解决不同问题：

- `message_id` 用来标识“这条消息本身”
- `client_msg_id` 用来处理重复提交
- `server_seq` 用来做补拉、ACK 推进和顺序恢复

这三者不能互相替代。

### 3.2 可靠消息状态机

M2 为消息投递引入了明确状态：

- `Created`
- `Persisted`
- `Delivering`
- `Delivered`
- `Failed`
- `Read`

核心思想是：  
不要把“消息有没有发出去”混成一个布尔值，而是用状态机表达消息生命周期。

这样后续才能继续扩：

- 重试
- 超时补偿
- 已读回执
- 指标统计

### 3.3 先落盘，再尝试在线投递

这是 M2 最重要的设计选择。

发送链路不再是：

1. 找连接
2. 直接发
3. 发完就算成功

而是变成：

1. 先写入 store
2. 记录投递状态
3. 如果目标在线，再尝试即时投递
4. 如果不在线，进入待补发队列

这个顺序非常关键。  
因为一旦先发后记，服务在中间任何一步崩掉，消息语义都会断裂。

### 3.4 存储抽象先行

M2 没有把可靠消息直接绑死在 MySQL 上，而是先引入：

- `IMessageStore`
- `InMemoryMessageStore`
- `MySqlMessageStore`

这是一种很典型的工程化做法：

- 内存实现用于早期验证逻辑
- MySQL 实现用于真实持久化
- `MessageService` 只依赖抽象，不关心底层介质

这样后面做 M3 仓储拆分时，重构成本就低很多。

## 4. 本阶段实际改了什么

### 4.1 协议升级

主要文件：

- `src/codec/protocol.h`
- `src/codec/protocol.cpp`
- `src/codec/messageSerializer.*`
- `proto/message.proto`

新增能力：

- `P2PMsgNotify`
- `MessageAckReq / MessageAckResp`
- `PullOfflineReq / PullOfflineResp`
- `client_msg_id`
- `server_seq`
- ACK code
- 离线消息项序列化

这些协议对象让“发消息、回 ACK、主动补拉、登录补发”形成完整闭环。

### 4.2 服务层重构

主要文件：

- `src/service/messageService.h`
- `src/service/messageService.cpp`

核心变化：

- `MessageService` 不再自己保存消息状态
- 消息创建、ACK、补拉、待投递查询统一下沉到 store
- 增加后台重试线程
- 支持 ACK 超时、重试退避、最大重试次数
- 超限后推进到 `FAILED`

这说明 `MessageService` 的职责已经从“消息发送器”升级成“可靠消息编排器”。

### 4.3 MySQL 接入

主要文件：

- `src/storage/mysqlClient.h`

- `src/storage/messageStore.h`
- `src/storage/messageStore.cpp`

- `scripts/init_mysql_schema.ps1`
- `scripts/m2_e2e_verify.py`

数据库层新增：

- `im_message`
- `im_message_delivery`
- `im_message_ack_log`

它们分别负责：

- `im_message`：消息主表
- `im_message_delivery`：每个接收方的投递状态
- `im_message_ack_log`：ACK 审计记录

### 4.4 配置与入口层

主要文件：

新增能力：

- 构建期启用 `HAVE_MYSQL_CLIENT`
- 启动时自动选择 `mysql` 或 `memory` store

## 5. 数据模型为什么这么拆

### 5.1 为什么消息主表和投递表分开

因为“消息内容”和“投递状态”本来就是两类不同语义的数据。

如果把它们塞进一张表里，会出现这些问题：

- 多接收方扩展困难
- ACK / 重试字段越来越臃肿
- 查询待投递消息不够高效

所以 M2 把模型拆成：

- 一条消息主记录
- 一条或多条投递状态记录

这也是后面支持群聊、广播、多端同步的重要前提。

### 5.2 为什么还要单独 ACK 日志

因为 ACK 更新状态和 ACK 审计不是一回事。

- 状态表回答“当前结果是什么”
- ACK 日志回答“历史上发生过什么”

面试时这里是个很好的加分点：  
你不是只做了“状态覆盖更新”，而是保留了可追踪审计记录。

## 6. 重试与补偿策略

M2 当前实现的重试策略包含：

- 定时扫描在线用户未确认消息
- ACK 超时判断
- 退避重试
- 最大重试次数
- 超限后推进 `FAILED`

这套策略不是生产级调度系统，但已经形成了完整的第一版闭环。

当前配置项包括：

- `retry_interval_ms`
- `ack_timeout_ms`
- `retry_backoff_ms`
- `max_retry_count`
- `retry_batch_size`

这意味着可靠消息行为已经不是写死在代码里，而是可以通过配置控制。

## 7. 真实验证过程

### 7.1 编译验证

Linux / WSL 构建链路已识别到 MySQL client 开发库：

- `Found MySQL client: /usr/lib/x86_64-linux-gnu/libmysqlclient.so`

这说明：

- MySQL 相关代码不只是写好了
- 而是已经真正参与编译和链接

### 7.2 建表验证

通过脚本：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\init_mysql_schema.ps1
```

已完成库表初始化，确认存在：

- `im_server`
- `im_message`
- `im_message_delivery`
- `im_message_ack_log`

### 7.3 端到端联调验证

通过脚本：

```bash
python3 scripts/m2_e2e_verify.py
```

已完成以下链路验证：

1. 在线投递
2. 接收端 ACK
3. MySQL 状态推进
4. 离线发送
5. 登录补发
6. 主动 `PullOfflineReq`
7. 离线消息 ACK

实际验证结果：

- `[PASS] 在线投递 + ACK 链路已验证`
- `[PASS] 离线补发 + 主动补拉 + ACK 链路已验证`
- `[PASS] M2 端到端验证完成`

### 7.4 已确认的行为

本轮联调已确认：

- `P2P` 消息会写入 `im_message`
- 投递状态会写入 `im_message_delivery`
- ACK 会推进状态为 `DELIVERED` 或 `READ`
- ACK 审计会写入 `im_message_ack_log`
- 离线用户重连后可收到补发
- 客户端主动补拉也能拉回未确认消息

## 8. 本阶段的设计取舍

### 8.1 为什么不直接上连接池

因为 M2 的重点是先闭合消息语义，而不是一口气做完生产级数据访问层。  
如果一开始同时做连接池、仓储层、调度器、指标系统，复杂度会过高。

### 8.2 为什么先做 store 抽象，不直接 repository

这是为了控制改动半径。  
M2 先让“可靠消息主链路”可用，M3 再进一步把 `MySqlMessageStore` 拆成 repository。

这种分阶段方式更适合项目演进和面试叙述。

### 8.3 为什么当前重试只扫描在线用户

因为这个阶段的目标是“把消息可靠语义跑通”，不是“做全局调度中心”。  
在线用户扫描已经足以验证 ACK、超时、补投、失败终态这些关键逻辑。

## 9. 当前能力边界

M2 已经具备：

- `message_id`
- `client_msg_id`
- `server_seq`
- 幂等去重
- ACK
- 登录补发
- 主动补拉
- 存储抽象
- MySQL 持久化
- 后台重试补投
- ACK 超时控制
- 重试退避
- 最大重试次数
- 失败终态推进
- 自动建表脚本
- 端到端验证脚本

M2 还没有完全做成：

- 生产级连接池
- repository 精细拆层
- 多节点全局序列协调
- 失败熔断
- 独立补偿调度器
- 更细粒度投递指标统计

这些正是 M3 继续推进的内容。

---

## 10. 面试讲解建议

讲 M2 时，建议不要只说“我做了 ACK 和 MySQL”。

更推荐按下面顺序表达：

1. 先讲 IM 可靠性问题有哪些。
2. 再讲你引入了哪三个关键标识。
3. 再讲为什么要先落盘再投递。
4. 再讲状态机如何推进。
5. 最后讲你做了真实 MySQL 联调和端到端验证。

这样会让面试官感受到你不是在“堆功能”，而是在设计一条可靠消息主链路。

---

## 11. 下一阶段

M2 完成后，系统已经具备可靠消息闭环，但数据层还偏集中。

最自然的下一步就是 M3：

- Repository 拆分
- Session 持久化接入
- `MySqlMessageStore` 委托仓储层
- 线程边界继续收敛

也就是说，M2 解决的是“消息能可靠闭环”，  
M3 解决的是“数据层和职责边界能不能继续工程化升级”。
