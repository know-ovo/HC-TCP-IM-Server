# M1 系统讲解文档：协议与网络基础升级

## 1. 阶段目标

M1 的目标不是简单“改个包头”，而是把项目从 demo 风格协议栈，升级成后续可以继续承载可靠消息、持久化、观测和分布式扩展的基础设施层。

这一阶段最终要解决 4 个根问题：

1. 旧协议头信息量不足，无法稳定承载请求/响应、ACK、序列号和错误码。
2. 编解码入口只适合 demo 消息，后面继续扩协议会越来越乱。
3. 网络层缺少背压和输出缓冲治理，高并发下容易失控。
4. 协议格式与传输模型耦合过深，后续很难平滑接入 Protobuf。

***

## 2. 为什么必须先做 M1

如果不先做 M1，后续 M2 / M3 基本都会反复返工：

- 没有 `requestId` / `flags` / `serverSeq`，ACK 链路很难讲清楚。
- 没有统一包头和校验，离线补拉与重试消息无法稳定落地。
- 没有高水位和背压，可靠消息越完善，系统越容易被慢连接拖垮。
- 没有协议抽象层，后面加 JSON/Protobuf 双格式支持会非常痛苦。

所以 M1 本质上是在做“协议地基 + 网络地基”。

***

## 3. 核心设计思路

### 3.1 从“旧消息回调”升级为“完整包回调”

原项目更接近“收到字符串后直接反序列化处理”。\
这种方式在 demo 阶段够用，但一旦协议头变复杂，就必须先把“包”当成一等公民。

因此本阶段做了两层抽象：

- `PacketHeader`：承载协议控制字段
- `DecodedPacket`：承载“包头 + payload”

这样业务层拿到的已经不是一段裸文本，而是一个完整协议单元。

### 3.2 包头字段显式化

新的包头引入了这些关键字段：

- `version`
- `flags`
- `requestId`
- `clientSeq`
- `serverSeq`
- `errorCode`
- `checksum`

这组字段的意义分别对应：

- `version` 解决协议演进问题
- `flags` 解决请求/响应/ACK 语义问题
- `requestId` 解决多请求匹配问题
- `clientSeq` / `serverSeq` 为可靠消息做铺垫
- `errorCode` 解决统一错误表达
- `checksum` 提升包体完整性校验能力

### 3.3 网络层先做“自我保护”，再谈吞吐

M1 没有盲目追求极限性能，而是优先补齐工程上更重要的控制能力：

- ET/LT 模式切换
- 高水位/低水位
- 最大输出缓冲限制
- 读写到 `EAGAIN`

这套能力的意义是：后面即使消息量增大，连接层也有基本治理手段，不会直接退化成“谁发得慢谁拖死服务端”。

### 3.4 Protobuf 采用“可选接入、缺失降级”

M1 没把 Protobuf 强绑成硬依赖，而是做成：

- 找到 Protobuf，则启用
- 找不到，则回退 JSON

这样做的权衡很清楚：

- 优点：构建环境更宽松，不阻塞主线开发
- 代价：运行环境能力不完全一致，需要在文档里说明当前 payload 格式

这个取舍适合当前求职项目阶段。

## 4. 本阶段实际改了什么

### 4.1 协议层

主要文件：

- `src/codec/protocol.h`
- `src/codec/protocol.cpp`
- `src/codec/codec.h`
- `src/codec/codec.cpp`
- `src/codec/messageSerializer.*`

落地内容：

- 新增 `PacketHeader`
- 新增统一 flags 推断逻辑
- 新增错误码体系
- 新增 `DecodedPacket`
- 新增完整包回调 `PacketCallback`
- 保留旧 `MessageCallback` 兼容入口
- 增加 `magic/version/length/checksum` 校验

### 4.2 Protobuf 接线

主要文件：

- `proto/common.proto`
- `proto/auth.proto`
- `proto/message.proto`
- `proto/heartbeat.proto`
- `src/codec/protoCodec.h`
- `src/codec/protoCodec.cpp`

落地内容：

- 新增基础 proto 文件
- 新增 payload format 解析工具
- CMake 支持“找到 Protobuf 就启用，找不到就降级”

### 4.2.1 双轨策略补齐（本次增量）

补齐背景：

- 之前阶段仅完成了 Protobuf 基础设施（`.proto`、CMake、payload 配置解析）
- 业务主链路实际仍以 JSON 编解码为主，尚未形成“长连接 Protobuf + 运维调试 JSON”的可用闭环

本次补齐文件：

- `src/server_full.cpp`
- `src/service/messageService.h`
- `src/service/messageService.cpp`
- `proto/message.proto`

本次补齐内容：

- 在 `server_full` 增加协议双轨编解码分支：登录、心跳、P2P、ACK、离线拉取、广播、踢人
- 新增 `compatible_json` 过渡兼容行为：当 `payload_format=protobuf` 且 payload 非法时，可按开关回退 JSON 解析
- 响应包统一按当前 payload 格式回包，避免“入站 protobuf、出站 json”不一致
- `MessageService` 新增通知编码回调注入能力，确保 `P2PMsgNotify` 与 `BroadcastMsgNotify` 下行同样遵循当前 payload 格式
- 将 `proto/message.proto` 中 `P2PMessageResp.msg_id` 调整为 `uint64`，与协议层 `msgId` 一致

补齐后结论：

- “客户端长连接协议走 Protobuf、管理/调试保留 JSON”已具备可执行实现
- 可通过 `[protocol].payload_format` 在 `json/protobuf` 间切换，并通过 `compatible_json` 控制迁移期兼容策略

### 4.3 网络层治理

主要文件：

- `src/net/eventLoop.*`
- `src/net/epollPoller.*`
- `src/net/tcpConnection.*`

落地内容：

- 新增 ET/LT 开关
- `TcpConnection` 增加高水位/低水位/最大输出缓冲
- 增加背压标记与高水位回调
- 读写逻辑改为持续处理到 `EAGAIN`

### 4.4 入口与配置

主要文件：

- `src/server_full.cpp`
- `conf/server.ini`

落地内容：

- 新增 `[protocol]`
- 新增 `[net]`
- `server_full` 改为走完整 packet 回调
- 启动阶段下发连接层高水位和背压参数

***

## 5. 配置项说明

M1 新增或强化的关键配置项有：

### 5.1 `[protocol]`

- `version`：协议版本号
- `payload_format`：`json` 或 `protobuf`
- `compatible_json`：兼容阶段的回退开关

### 5.2 `[net]`

- `epoll_mode`：`lt` 或 `et`
- `tcp_nodelay`
- `high_water_mark_bytes`
- `low_water_mark_bytes`
- `max_output_buffer_bytes`
- `max_packet_bytes`

这些配置的价值不是“可调着玩”，而是把协议和连接治理从硬编码变成了可控行为。

## 6. 验证过程与结果

本阶段已完成的验证：

1. `ReadLints` 无新增诊断。
2. Linux / WSL 主路径编译通过。
3. `server_full` 已切到完整 packet 回调链路。
4. 缺少 Protobuf 开发环境时，系统会自动回退到 JSON，不阻塞构建。
5. 双轨编解码主链路已补齐：登录、心跳、P2P、ACK、离线拉取、广播下发均支持按配置输出 JSON/Protobuf。

本阶段采用的有效验证基线是 Linux 主路径，而不是 Windows / MSVC。\
原因不是代码无效，而是仓库原本就存在一批更底层的跨平台兼容问题，M1 的验收重点应放在 Linux 服务端主链路。

## 7. 设计取舍

### 7.1 为什么不直接全面切 Protobuf

因为此时最重要的是“协议入口稳定”，不是“序列化方案绝对先进”。\
如果直接全量切 Protobuf，会把构建依赖、调试复杂度、联调成本一起拉高，不适合本阶段。

### 7.2 为什么先补背压，不先补线程池

因为连接治理问题会直接影响所有后续阶段。\
线程池模型可以继续优化，但如果输出缓冲没有边界，M2 的可靠消息越完整，系统越危险。

### 7.3 为什么保留旧回调兼容入口

因为这是渐进式重构，不是推倒重写。\
保留兼容入口可以让业务处理逻辑分阶段迁移，降低升级风险。

***

## 8. 本阶段产出价值

M1 完成后，项目从“可跑 demo”进入“可继续系统化演进”的状态。

它直接为后续阶段提供了这些能力：

- M2 可以安全接入 ACK / 序列号 / 补拉
- M3 可以稳定扩数据层与仓储层
- M4 / M5 可以在更清晰的协议和网络边界上做限流、观测和工程化

***

## 9. 面试讲解建议

如果在面试里讲 M1，建议不要只说“我改了协议头”。\
更好的表达方式是：

1. 先说明旧协议为什么不够用。
2. 再说明新包头字段分别解决什么问题。
3. 再说明网络层为什么必须补背压。
4. 最后强调这是为可靠消息和持久化做前置地基。

这样就能把 M1 从“改代码”讲成“做系统演进设计”。
