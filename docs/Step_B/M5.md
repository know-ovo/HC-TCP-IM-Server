# M5 系统讲解文档：可观测与工程化闭环

## 1. 阶段目标

M5 的目标不是“再加几个功能点”，而是把系统从“能跑、能恢复、能分布式”升级为“可观测、可验证、可持续演进”的工程形态闭环：

1. 结构化日志：日志从“字符串输出”升级为“可检索的事件”
2. Trace ID：端到端链路可串联，跨线程/跨模块透传
3. 指标体系：提供 `/metrics`，度量可靠性、容量、延迟与背压
4. 测试体系：单测 + 集成测试覆盖主链路与关键治理点
5. Benchmark：用数据说明方案取舍（JSON vs Protobuf、LT vs ET、Buffer 等）
6. 配置与构建收口：把新增能力纳入统一配置与构建选项

---

## 2. 为什么 M5 必须做

如果没有 M5，可观测与工程化缺位会带来直接风险：

- 出问题时只有零散日志，很难把“登录-投递-ACK-补拉”串成一条链路
- 无指标，无法回答系统的“容量、可靠性、体验”三类核心问题
- 无测试体系，任何一次重构都没有回归保障
- 无 Benchmark，优化与取舍缺少“证据感”，在面试与实际演进中都说服力不足

M1~M4 解决“能用、能恢复、能扩展”，M5 解决“能观察、能验证、能沉淀”。

---

## 3. 本阶段实际改了什么

### 3.1 结构化日志

- 日志字段统一：
  - `trace_id` `request_id` `user_id` `session_id` `node_id`
  - `command` `message_id` `server_seq` `conn_id`
- 日志分层：
  - INFO：关键状态变化（登录成功、投递启动、ACK 成功、补拉完成）
  - WARN：限流、重试、高水位、慢连接
  - ERROR：数据库异常、协议异常、投递失败
  - DEBUG：开发环境按需开启
- 输出格式（示例）
  - `time=... level=INFO trace_id=... user_id=... command=... message_id=... msg="delivering"`

### 3.2 TraceContext 与透传

- 新增 `TraceContext` 结构，入口生成 `trace_id`
- 跨线程（业务线程池）、跨模块（Auth/Message/Heartbeat）、跨数据层（Redis/MySQL）透传
- 关键操作写入 Trace 相关字段，支持把“收包->解包->落盘->投递->ACK/重试->补拉”完整串联

### 3.3 指标与 `/metrics`

- 新增 `MetricsRegistry` 抽象，支持：
  - 计数器 `incCounter(name)`
  - 直方图/摘要 `observe(name, value)`
  - 仪表盘 `setGauge(name, value)`
- 首批指标（与 P1 计划对齐）：
  - 连接与吞吐：`im_online_connections` `im_login_qps` `im_p2p_qps`
  - 可靠性：`im_message_retry_total` `im_message_ack_timeout_total` `im_messages_failed_total`
  - 背压与并发：`im_threadpool_queue_size` `im_output_buffer_bytes`
  - 体验：`im_message_latency_ms`（从入站到 ACK）
- 轻量 HTTP 端点：
  - 独立端口（默认 9100）
  - 单独线程驱动，返回 Prometheus 文本格式

### 3.4 单元测试

- 新增测试对象（示例集合，与清单一致）：
  - `Codec`：包头解析、长度校验、CRC/Checksum
  - `Buffer`：高水位、回收与扩容行为
  - `TokenBucketRateLimiter`：令牌生成与突发窗口
  - `CircuitBreaker`：关闭->打开->半开状态机
  - `OverloadProtectService`：多维度限流判定

### 3.5 集成测试

- 场景覆盖（与主链路一致）：
  1. 登录 -> 发消息 -> ACK
  2. 离线 -> 登录后补拉
  3. Redis 开关两种模式
  4. 慢客户端触发高水位
  5. ACK 超时触发重试
- 断言维度：
  - 协议正确性（包头/命令字/序列号）
  - MySQL/Repository 状态推进
  - Redis 在线与路由键变化
  - 指标曲线是否随场景变化

### 3.6 Benchmark

- 项目内基准测试建议覆盖：
  - JSON vs Protobuf 编解码吞吐
  - `Buffer` 追加与回收
  - 线程池任务分发吞吐
  - LT vs ET 下 `epoll_wait` 唤醒次数与 CPU 利用率
- 结果呈现：
  - 统一输出 csv/markdown
  - 文档内给出解读维度与面试话术

### 3.7 构建与配置收口

- 构建：
  - 按可选依赖启用：GTest / Prometheus C++ client（若采用）/ Google Benchmark
  - 缺失第三方库时，后备实现不阻塞主链路
- 配置：
  - `[metrics]`：`enable` `bind` `port` `path`
  - `[logging]`：`level` `format` `enable_debug`（示例）
  - `[trace]`：`enable` `sample_ratio`（可选）

---

## 4. 设计取舍

- 轻量 `/metrics` 优先于引入重型观测栈
  - 优点：实现与集成成本低，足够支撑 Prometheus 拉取
  - 代价：指标模型简单，分布式标签维度有限
- 指标“先核心后长尾”
  - 先覆盖“容量（连接数）/可靠性（成功-失败-重试）/并发（队列长度）/背压（输出缓冲）/体验（P99）”
  - 复杂分位数统计与高维标签待后续演进
- 日志“关键即充分”
  - 只在关键节点打点，避免冗余日志挤压 I/O 与磁盘
- Trace 以“能串起来”为第一目标
  - 先解决跨线程与跨模块透传，再讨论与外部追踪系统对接

---

## 5. 配置项

### 5.1 `[metrics]`

- `enable = true`
- `bind = 0.0.0.0`
- `port = 9100`
- `path = /metrics`

### 5.2 `[logging]`（示例）

- `level = info`
- `format = key_value`
- `enable_debug = false`

### 5.3 `[trace]`（示例）

- `enable = true`
- `sample_ratio = 1.0`

以上配置放入 `conf/server.ini`，与现有 `[cluster]` `[rate_limit]` `[circuit_breaker]` `[connection]` 一并管理。

## 6. 验证方法

### 6.1 日志与 Trace 串联

- 触发一条完整链路：登录 -> 发消息 -> ACK
- 检查日志连续性：应存在同一 `trace_id` 下的“收包/落盘/投递/ACK”事件
- 人工或脚本校验关键字段一致性（`request_id/message_id/server_seq/user_id`）

### 6.2 Prometheus 指标

- 打开浏览器或 `curl` 访问：`http://127.0.0.1:9100/metrics`
- 校验首批指标是否存在并随流量变化
  - 例如压测登录是否提升 `im_login_qps`
  - 慢客户端是否提升 `im_output_buffer_bytes`

### 6.3 单测

- 构建并运行单元测试目标
- 关注易回归点：包头解析边界、令牌桶时间推进、断路器状态切换边界

### 6.4 集成测试

- 按场景脚本执行五类流程
- 断言数据库/Redis/日志/指标四视角一致性

### 6.5 Benchmark

- 运行基础基准集
- 输出结果保存为 `bench/` 下的 csv/markdown
- 在文档中给出解读：例如 Protobuf 编解码在消息体规模 X 下的收益区间

## 7. 面试讲解建议

面试讲 M5，避免只说“我加了监控和日志”。推荐叙述路径：

1. 先说 IM 系统排障的真实痛点：跨线程、跨模块、跨数据层难以串联
2. 再说如何通过结构化日志 + Trace 解决“看得见链路”的问题
3. 再说为什么这些指标就是“容量/可靠性/并发/背压/体验”的核心代表
4. 强调你用单测/集成测试/Benchmark 让设计有“证据”
5. 最后说明配置与构建如何把能力“纳管”，避免散落一地

## 8. 当前状态与边界

本阶段完成后，系统具备：

- 结构化日志与 Trace 透传
- 基础指标与 `/metrics` 暴露
- 单测与集成测试骨架
- 基准测试样例与结果沉淀格式
- 配置与构建项收口

仍保留的演进空间：

- 更完善的 Trace 采样与外部 APM 对接
- 更细的指标标签与直方图分桶
- 指标与日志的统一关联（日志转储到集中系统）
- 覆盖更多边界条件的集成测试与故障演练

---

## 9. 与 M1~M4 的承接关系

- 基于 M1 的“协议与网络地基”，日志与指标得以在统一包头/事件模型上打点
- 依赖 M2 的“可靠消息闭环”，Trace 可以串起落盘/投递/ACK/补拉的完整生命周期
- 结合 M3 的“数据层与线程边界收敛”，Trace 才能跨线程、跨仓储安全透传
- 建立在 M4 的“状态中心与自保护能力”之上，指标才能真实反映限流、熔断与连接治理效果

M5 把前四个阶段的“可用性”沉淀为“可观测与可验证”，形成求职与后续演进都能稳定讲述的一整套证据链。
