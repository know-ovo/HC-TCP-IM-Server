# P1 升级计划：分布式、高可用、稳定性、可观测与工程化

## 1. 目标

如果 P0 解决的是“单机主链路闭环”，那么 P1 解决的是：

- 系统在多节点下如何工作
- 出问题时如何保护自己
- 线上问题如何观测与定位
- 如何把项目从“能跑”推进到“可持续维护”

对于求职项目来说，P1 的价值非常大，因为它能体现你不仅会写网络代码，还具备后端系统设计意识。

---

## 2. 分布式与高可用

## 2.1 Redis 状态中心

### 原理

单机内存里的在线状态只能服务当前节点。  
一旦需要多实例部署，就必须引入一个共享状态中心。

当前项目已经有 `RedisClient` 和部分 Session / 在线状态基础，这非常适合作为 P1 的延伸点。

### 建议 Redis 中存放的数据

- `online:{userId}` -> `nodeId`
- `session:{sessionId}` -> 用户会话信息
- `route:{userId}` -> 当前连接所在网关节点
- `presence:{userId}` -> 最后在线时间 / 设备信息

### 权衡

Redis 很适合做在线状态中心，但不适合承担最终可靠存储。  
因此这里的原则仍然是：

- Redis 负责“快状态”
- MySQL 负责“准事实”

---

## 2.2 跨节点路由与转发

### 原理

用户 A 和用户 B 可能连在不同网关节点。  
此时发送消息不能再依赖本机 `userId -> TcpConnection` 映射，而应变成：

1. 查询接收方路由信息
2. 如果在本节点，直接投递
3. 如果在其他节点，转发到目标节点

### 可选方案

#### 方案一：Redis 查询 + 节点间 RPC

特点：

- 结构清楚
- 好讲
- 适合求职项目

#### 方案二：Redis Stream / MQ 中转

特点：

- 解耦更彻底
- 复杂度更高

建议第一版采用：

- **Redis 路由表 + 节点间轻量转发接口**

原因：

- 足以体现分布式意识
- 不会把项目复杂度推得过高

---

## 2.3 节点故障后的恢复策略

### 原理

当某台网关宕机时，用户连接会断开。  
系统必须做到两件事：

- 让客户端尽快重连到其他节点
- 重连后通过 `last_acked_seq` 完成补拉

### 建议策略

- 客户端配置多个接入地址
- 登录后从服务端拿到推荐节点列表
- 连接断开后按顺序重试
- 登录成功后立即触发未确认消息补拉

### 面试表达重点

这里的关键不是“永不掉线”，而是：

- 节点故障后，系统是否具备可恢复性

这比只会说“用了 Redis 所以支持分布式”要强得多。

---

## 3. 稳定性治理

## 3.1 限流

当前项目已有 `OverloadProtectService`，但更偏向简单计数。  
P1 建议升级为更细粒度的限流系统。

### 建议限流维度

- 全局连接数
- 单 IP 建连速率
- 单用户消息速率
- 单命令字 QPS
- 广播类消息单独限流

### 建议算法

- 接入限流：滑动窗口或漏桶
- 业务消息限流：令牌桶

### 权衡

令牌桶的优点：

- 支持一定突发
- 语义直观
- 面试常见，容易讲

因此推荐优先落地令牌桶。

---

## 3.2 熔断与降级

### 原理

当下游服务（例如 MySQL、Redis）抖动时，不能让整个 IM 服务一起雪崩。  
因此需要熔断与降级。

### 建议熔断点

- MySQL 写入超时
- Redis 查询超时
- 线程池排队过长
- 补拉请求过载

### 建议降级策略

- MySQL 异常时：拒绝新消息写入，但允许心跳和基础登录继续
- Redis 异常时：退回本地状态，限制集群特性
- 广播消息在高压时直接降级丢弃

### 面试价值

你要表达的不是“我写了个 Hystrix”，而是：

- 我知道系统在资源紧张时应该优先保住什么

这比单纯堆名词更有说服力。

---

## 3.3 空闲连接与异常连接清理

当前已有 `HeartbeatService`，P1 需要把它从“心跳模块”升级为“连接健康治理中心”。

建议补充：

- 未认证连接超时踢除
- 长时间无业务交互的空闲连接治理
- 写阻塞连接治理
- 心跳正常但业务持续失败的异常连接记录

这样可以把系统从“只要心跳就算活着”升级为“真正理解连接健康”。

## 4. 可观测性

## 4.1 结构化日志

当前项目已有 `spdlog`，这是很好的基础。  
P1 建议把日志从“输出文本”升级为“可检索的结构化事件”。

### 推荐字段

- `trace_id`
- `request_id`
- `user_id`
- `session_id`
- `conn_id`
- `command`
- `message_id`
- `node_id`

### 日志分层

- `INFO`：关键状态变化
- `WARN`：限流、重试、高水位、慢连接
- `ERROR`：数据库异常、协议异常、投递失败
- `DEBUG`：仅开发环境打开

### 权衡

日志不能越多越好，必须围绕“定位问题”来打。  
建议只记录关键链路节点，避免把消息正文全量打进日志。

---

## 4.2 Trace ID 与请求链路

### 原理

线上排查最怕的是：

- 登录日志一条
- 投递日志一条
- ACK 日志一条

但无法判断它们是不是同一次请求。

因此建议：

- 每次请求生成 `trace_id`
- 跨线程、跨模块、跨 Redis/MySQL 操作时一路透传

这样才能把：

- 收包
- 解包
- 落盘
- 投递
- ACK

串成一条完整链路。

---

## 4.3 Prometheus 指标

### 推荐核心指标

- `im_online_connections`
- `im_messages_ingress_total`
- `im_messages_delivered_total`
- `im_messages_retry_total`
- `im_messages_failed_total`
- `im_message_ack_timeout_total`
- `im_threadpool_queue_size`
- `im_tcp_output_buffer_bytes`
- `im_login_qps`
- `im_p2p_qps`
- `im_p99_latency_ms`

### 为什么这些指标重要

- 连接数：看容量
- 成功/失败/重试：看可靠性
- 队列长度：看并发压力
- 输出缓冲：看背压
- P99：看用户体验

建议先做 `/metrics` 暴露，后续再接 Grafana。

## 5. 工程化

## 5.1 单元测试

建议优先为以下模块补单测：

- `Buffer`
- `Codec`
- `ThreadPool`
- `OverloadProtectService`
- `AuthService` 的 Session 序列化逻辑

为什么不是一上来测所有服务：

- 先测基础模块，收益最高
- 服务层测试后续可以逐步补集成测试

---

## 5.2 集成测试

建议新增集成测试场景：

- 登录 -> 发消息 -> ACK -> 断线重连 -> 补拉
- Redis 开启与关闭两种模式
- 目标用户离线 / 在线两种投递路径
- 慢客户端触发高水位
- 节点故障后的重连恢复

对于 IM 项目，集成测试的说服力通常比纯单测更强。

---

## 5.3 静态检查与基准测试

建议接入：

- `clang-format`
- `clang-tidy`
- `Google Benchmark`

基准测试建议覆盖：

- `Buffer` 追加与回收性能
- JSON vs Protobuf 编解码
- 单线程 vs 线程池任务分发
- LT vs ET 下吞吐与 CPU 使用差异

这些数据会让你的项目在简历和面试中更有“证据感”。

---

## 5.4 交付规范

建议后续每完成一个升级点，都补齐以下产物：

- 设计文档
- 配置项说明
- 单测 / 集成测试
- 压测记录
- 面试问答摘要

这样做的意义很大：

- 代码是项目本身
- 文档和数据是你求职时的“证明材料”

---

## 6. 推荐实施顺序

P1 建议按以下顺序推进：

1. Redis 状态中心与路由表
2. 跨节点消息转发
3. 限流、熔断、降级
4. 结构化日志与 Trace ID
5. Prometheus 指标
6. 单测、集成测试、Benchmark

因为：

- 集群与稳定性需要建立在 P0 主链路之上
- 可观测和测试要围绕真实主链路打点，不能空转

---

## 7. 本部分的面试价值

这部分做完后，你能更自然地回答以下问题：

- 如果服务从单机升级到多机，消息怎么路由
- 某个节点挂了怎么办
- Redis 和 MySQL 出问题时，系统怎么保护自己
- 线上问题怎么定位
- 你怎么证明自己的优化真的有效

这会让你的项目从“高性能通信项目”进一步升级为“具备完整后端系统思维的服务端项目”。
