# P0 升级计划（一）：网络层与协议层

## 1. 目标

本部分的目标不是单纯追求“更快”，而是把当前网络与协议骨架升级为：

- 连接模型清晰
- 事件处理边界清晰
- 协议可演进
- 能承载后续 ACK、补拉、持久化、分布式路由

换句话说，这一部分是后续所有升级的地基。

---

## 2. 当前实现现状

从现有代码看：

- `EventLoop` 已具备 `runInLoop()`、`queueInLoop()`、`eventfd` 唤醒能力
- `TcpConnection` 已具备收发缓冲与连接生命周期管理
- `Buffer` 采用 muduo 风格双指针设计，底层仍是 `std::vector<char>`
- 协议层在 `protocol.h` 中定义了固定 16 字节头：长度、命令、序列号、CRC16
- 业务消息体当前通过 `nlohmann/json` 序列化

现阶段的问题主要有：

- 协议头字段仍偏简化，不足以支撑可靠消息和版本演进
- `Buffer` 尚未针对大规模长连接场景加入背压与高水位治理
- ET / LT 的策略还没有明确抽象为可配置能力
- 网络层缺少慢连接、大发包、异常连接的系统化治理机制

---

## 3. 网络层升级方案

## 3.1 非阻塞 I/O 与 LT/ET 事件模式

### 原理

Reactor 模型中，`epoll` 负责监听 fd 上的可读/可写事件。  
事件触发方式分为：

- LT：只要 fd 仍然可读/可写，就持续通知
- ET：状态边沿变化时才通知一次

ET 的优势是减少重复唤醒次数，适合高并发场景；  
ET 的代价是实现复杂度更高，必须在每次回调里持续读/写直到 `EAGAIN`。

### 权衡

本项目是求职导向，`ET` 非常值得做，因为它能显著提升面试亮点。  
但不建议把系统写死为“只能 ET”，更推荐：

- 默认支持 `LT` 和 `ET`
- 文档与面试重点讲 `ET`
- 实际实现保留配置开关，便于回退与对比压测

### 落地实现

建议新增配置项：

- `net.epoll_mode = LT | ET`

建议改造点：

- 在 `Channel` 中支持明确记录事件模式
- 在 `EpollPoller` 封装中统一设置 `EPOLLET`
- 在 `TcpConnection::handleRead()` 中循环读取直到 `EAGAIN`
- 在 `TcpConnection::handleWrite()` 中循环发送直到 `EAGAIN` 或输出缓冲为空

### 验收标准

- LT 与 ET 均可启动并正常通信
- ET 模式下粘包、半包、大包、对端关闭等场景处理正确
- 压测时 ET 模式的 `epoll_wait` 唤醒次数低于 LT

---

## 3.2 Buffer 从“能用”升级为“可治理”

### 原理

当前 `Buffer` 的设计本身并不差，它已经是典型的 muduo 风格实现。  
真正的问题不在于“是不是 `std::vector<char>`”，而在于：

- 当连接写不出去时，缓冲区怎么增长
- 缓冲区增长后何时回收
- 一个慢客户端是否会把内存拖爆

所以这里升级的重点不应该是盲目上内存池，而应该先补：

- 高水位治理
- 背压
- 可观测的缓冲区状态

### 权衡

可选方案有三类：

1. 继续使用连续内存 `Buffer`
2. 升级为环形缓冲区
3. 上对象池 / 内存池

建议顺序是：

- **第一阶段：保留现有 `Buffer`，先加高水位与背压**
- **第二阶段：如果压测暴露频繁扩容/搬移问题，再升级为环形缓冲区**
- **第三阶段：只有在热点非常明确时，再考虑对象池**

原因：

- 当前项目的瓶颈大概率还不在“内存分配器”
- 环形缓冲区的收益比对象池更直观，更容易讲清楚
- 过早做对象池，复杂度会超过求职项目的必要范围

### 落地实现

建议为 `TcpConnection` 增加：

- 输出缓冲区高水位 `highWaterMark`
- 输出缓冲区低水位 `lowWaterMark`
- 高水位回调 `highWaterMarkCallback`
- 写阻塞持续时间统计

建议新增连接级指标：

- `pending_write_bytes`
- `buffer_expand_count`
- `last_write_block_time_ms`

建议治理策略：

- 正常情况下继续缓存待发送数据
- 达到高水位时：
  - 记录 WARN 日志
  - 暂停继续读取该连接上的业务消息，形成背压
- 超过最大写缓冲上限时：
  - 对普通消息丢弃或拒绝
  - 对关键控制消息保留
  - 必要时主动断开慢连接

### 验收标准

- 慢客户端场景下进程内存不随时间无限增长
- 高水位触发时有明确日志与监控
- 输出缓冲区回落后能自动恢复发送

## 3.3 背压、慢连接与异常连接治理

### 原理

长连接服务最常见的问题不是“不会发消息”，而是“某些连接发不出去时怎么办”。  
如果没有背压，服务端只会不断往连接输出缓冲追加数据，最终导致：

- 单连接拖垮内存
- 广播时整体雪崩
- I/O 线程花大量时间服务慢客户端

### 权衡

这里的核心不是“绝不丢消息”，而是“按消息等级与系统状态做策略化处理”。

推荐消息分级：

- Level 1：控制消息，例如登录响应、踢人、错误码
- Level 2：重要业务消息，例如单聊消息
- Level 3：普通通知消息，例如广播类低优先级消息

治理策略：

- 控制消息尽量不丢
- 普通广播在过载时允许丢弃或降级
- 慢连接达到阈值时可以主动断开

### 落地实现

建议新增以下能力：

- 单连接最大输入包长限制
- 单连接最大输出缓冲限制
- 连续写阻塞时间阈值
- 心跳正常但发送长期阻塞的“半死连接”检测

建议在 `TcpConnection` 中增加状态：

- `kNormal`
- `kBackPressured`
- `kClosing`
- `kSlowConsumer`

## 4. 协议层升级方案

## 4.1 从 JSON 升级到 Protobuf

### 原理

JSON 的优点是直观、易调试；  
Protobuf 的优点是：

- 体积更小
- 编解码更快
- 强类型
- 更适合协议版本演进

对于 IM 的长连接主链路，Protobuf 更合适。  
对于管理接口、调试工具，JSON 依然可以保留。

### 权衡

不建议“一刀切彻底消灭 JSON”，而建议采用双轨策略：

- **客户端长连接协议**：Protobuf
- **管理接口 / 运维接口 / 调试脚本**：JSON

这样既保留了工程合理性，也保留了面试亮点。

### 落地实现

建议新增目录：

- `proto/`

建议拆分 `.proto` 文件：

- `common.proto`
- `auth.proto`
- `message.proto`
- `heartbeat.proto`

建议 CMake 集成：

- `find_package(Protobuf REQUIRED)`
- 生成代码后链接到 `server_lib`

### 消息设计建议

统一定义一个基础包体：

- `request_id`
- `client_seq`
- `server_msg_id`
- `timestamp`
- `device_id`

业务消息再按类型挂载。

---

## 4.2 协议头扩展

### 当前头部

当前头部只有：

- magic/header
- length
- cmd
- seqId
- crc16

这能支撑基础通信，但不足以承载完整 IM 主链路。

### 建议升级后的头部字段

- `magic`
- `version`
- `header_len`
- `total_len`
- `command`
- `flag`
- `request_id`
- `client_seq`
- `server_seq`
- `error_code`
- `checksum`

### 字段说明

- `version`：用于协议演进与灰度兼容
- `flag`：标识请求/响应/ACK/重传/压缩/加密
- `request_id`：请求响应配对
- `client_seq`：客户端发送顺序号，用于去重与补拉
- `server_seq`：服务端投递序号，用于重连恢复
- `error_code`：统一失败语义

### 权衡

头部越大，单包固定开销越高；  
但头部过小，后续每升级一次协议都要推翻重来。

因此建议采用：

- 固定头 + 可保留扩展位
- 首版头部适度冗余，但不要过度堆字段

---

## 4.3 协议兼容策略

工业级协议不只看“今天能不能跑”，还要看“明天怎么升级”。

建议兼容策略如下：

- 新服务端优先兼容旧客户端 1 个大版本
- 客户端登录后先做能力协商
- 不支持的命令字返回标准错误码
- 新增字段遵循“可选字段 + 默认值”原则

建议增加一个能力协商消息：

- `CmdHelloReq`
- `CmdHelloResp`

协商内容包括：

- 协议版本
- 是否支持 ACK
- 是否支持补拉
- 是否支持压缩
- 是否支持 TLS

---

## 5. 推荐实施步骤

建议按以下顺序落地：

1. 为网络层增加 LT/ET 配置开关
2. 在 `TcpConnection` 增加高水位与背压治理
3. 为协议头新增 `version/request_id/error_code/flag`
4. 引入 Protobuf，先替换登录、心跳、P2P 三类消息
5. 保留 JSON 调试链路，便于过渡期抓包与对比

这样做的好处是：

- 不会一次性推翻所有消息体
- 能在协议层改完后无缝承接 ACK、补拉、持久化

---

## 6. 本部分的面试价值

这部分做完后，你在面试中可以稳定讲出以下内容：

- 为什么 ET 更高效，但实现更复杂
- 为什么高并发服务必须有背压，而不是无脑缓存
- 为什么 IM 协议要升级到 Protobuf
- 为什么协议头要预留版本、请求 ID、错误码、序号
- 为什么工程上不应该“一刀切消灭 JSON”

这会让你的项目从“写了个聊天程序”上升到“我对长连接协议与高并发接入层有系统设计能力”。
