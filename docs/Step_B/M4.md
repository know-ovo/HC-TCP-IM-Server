# M4 系统讲解文档：分布式与稳定性增强

## 1. 阶段目标

M4 的目标，是让系统从“单节点可用”继续升级到“具备多节点状态共享意识与基础自保护能力”。

这一阶段最终完成了四件事：

1. Redis 路由表与节点状态中心
2. 本地投递 / 远端投递分支
3. 限流与熔断降级
4. 连接治理增强

M1 解决协议地基，M2 解决可靠消息闭环，M3 解决数据层和线程边界，  
M4 解决的就是“系统开始具备分布式形态”和“系统开始知道如何自我保护”。

---

## 2. 为什么 M4 必须单独做

如果系统只有单节点内存状态，那么一旦进入多节点部署，就会立刻出现这些问题：

- 用户到底在线在哪个节点，别的节点不知道
- 目标用户不在本机时，代码没有分支
- 高并发压测下没有细粒度限流
- 线程池积压、慢连接、慢消费者时缺少主动保护

所以 M4 的意义不是“做更多功能”，而是让系统从“功能能跑”走向“系统知道自己什么时候该拒绝、降级、转分支”。

## 3. 本阶段实际改了什么

### 3.1 Redis 状态中心升级

主要文件：

- `src/service/authService.h`
- `src/service/authService.cpp`
- `src/service/heartbeatService.h`
- `src/service/heartbeatService.cpp`
- `src/service/routingService.h`
- `src/service/routingService.cpp`

本阶段统一采用这些 Redis key：

- `session:{sessionId}`
- `user:session:{userId}`
- `user:online:{userId}`
- `route:{userId}`
- `node:alive:{nodeId}`
- `node:connections:{nodeId}`

这意味着 Redis 在 M4 中不再只是“顺手存一下 session”，而是开始承担：

- 会话节点归属
- 用户在线状态
- 用户路由表
- 节点存活心跳
- 节点连接数摘要

### 3.2 新增 `RoutingService`

主要职责：

- `bindUserRoute()`
- `unbindUserRoute()`
- `lookupUserRoute()`

登录成功时：

- `AuthService` 建 session
- `HeartbeatService` 建 online
- `RoutingService` 建 route

连接断开时：

- 清理 route
- 清理 online

这样系统已经形成最基础的“用户 -> 节点”映射能力。

### 3.3 远端分支正式形成

在 `server_full.cpp` 的 `P2P` 处理链路中，M4 增加了显式路由判断：

1. 查 `route:{toUserId}`
2. 如果目标在本节点，继续走现有可靠消息主链路
3. 如果目标在其他节点，进入“远端分支”

当前第一版远端分支策略是：

- 记录日志
- 返回 `remote forwarding not enabled`
- 不假装它是离线用户

这个选择很重要。  
因为它明确地区分了：

- 用户真的离线
- 用户在线但在别的节点

这正是后面 M4.5 / M5 继续补跨节点 RPC 或消息转发通道的前提。

### 3.4 限流系统升级

主要文件：

- `src/infra/rateLimiter.h`
- `src/infra/rateLimiter.cpp`
- `src/service/overloadProtectService.h`
- `src/service/overloadProtectService.cpp`

M4 引入了令牌桶限流器 `TokenBucketRateLimiter`，并把 `OverloadProtectService` 从“只有全局连接数/QPS”升级成：

- `canAcceptConnection(ip)`
- `canProcessCommand(command)`
- `canSendMessage(userId)`

当前已支持的限流维度：

- 单 IP 建连速率
- 登录请求 QPS
- P2P 请求 QPS
- 广播请求 QPS
- 单用户发送速率

这意味着系统已经不再只会“全局粗暴拒绝”，而是开始具备更细粒度的节流能力。

### 3.5 熔断与降级

主要文件：

- `src/infra/circuitBreaker.h`
- `src/infra/circuitBreaker.cpp`
- `src/server_full.cpp`

M4 新增了 `CircuitBreaker`，并把它接入业务线程池提交流程。

当前策略是：

- 线程池任务提交失败累计到阈值后，断路器打开
- 打开后，低优先级任务优先拒绝
- `pullOffline` 在断路器打开时自动缩小批量
- 登录 / ACK 保持高优先级

这符合清单里“优先保登录、心跳、ACK；广播降级；离线补拉缩批”的目标。

### 3.6 连接治理升级

主要文件：

- `src/service/heartbeatService.h`
- `src/service/heartbeatService.cpp`
- `src/server_full.cpp`

M4 为连接新增了更明确的健康状态管理：

- `New`
- `AuthPending`
- `Active`
- `Idle`
- `BackPressured`
- `Closing`

同时增加了这些治理入口：

- `onAuthenticated()`
- `onWriteBlocked()`
- `onBackPressureRecovered()`
- `reportNodeAlive()`

当前治理规则包括：

- 未认证连接超时踢除
- 心跳超时踢除
- 慢消费者长时间写阻塞踢除
- 高水位进入背压状态时打标
- 写完成时回收背压状态

这使得连接层开始从“只管收发”升级成“知道连接是否健康”。

### 3.7 Redis 并发安全补强

主要文件：

- `src/base/redisClient.h`
- `src/base/redisClient.cpp`

由于 M3 已经把 Redis 操作切入业务线程池，M4 继续补了 `RedisClient` 的互斥保护，避免同一个 hiredis context 被多个线程同时使用。

这一步很关键，因为它让：

- session Redis
- online Redis
- route Redis
- node alive Redis

都能在当前线程模型下安全运行。

## 4. 设计取舍

### 4.1 为什么远端分支先返回“未启用”

因为当前阶段目标是先建立正确的结构边界，而不是一次把跨节点链路做完。

如果此时为了“看起来能用”把远端在线用户也当成离线用户处理，会把语义搞混。  
M4 选择显式返回“remote forwarding not enabled”，反而更工程化。

### 4.2 为什么限流和熔断不直接做到特别复杂

因为当前项目还处在单机到多节点过渡阶段。  
M4 第一版要的是：

- 有节流
- 有拒绝
- 有降级
- 有边界

而不是一开始就做特别复杂的动态窗口、多级规则编排平台。

### 4.3 为什么连接治理放在 `HeartbeatService`

因为现有项目里，连接活性、超时检测、在线状态本来就都集中在 `HeartbeatService`。  
M4 沿着现有职责自然扩展，比再新造一个连接治理中心更稳。

### 4.4 为什么 RedisClient 要补互斥

因为 M3/M4 已经明确采用“业务线程处理 Redis/MySQL”的模型。  
如果 Redis 客户端本身还是裸同步上下文，那么越往后做越危险。

## 6. 验证结果

本阶段已完成的验证包括：

### 6.1 编译验证

- `build_linux_m3` 在 M4 改动后重新编译通过
- `ReadLints` 无新增诊断

### 6.2 M2 主链路回归

重新执行：

```bash
python3 scripts/m2_e2e_verify.py
```

结果仍然通过：

- 在线投递 + ACK
- 离线补发 + 主动补拉 + ACK

说明 M4 没有打坏 M2 的可靠消息主链路。

### 6.3 Redis 路由与节点状态验证

在 WSL 本地 Redis 环境中已验证：

- `route:m4_route_user2 = node-1`
- `user:online:m4_route_user2 = 1`
- `node:alive:node-1 = 1`

这说明：

- 登录后 route 会写入
- 在线状态会写入
- 节点存活状态会周期刷新

### 6.4 远端分支验证

手动写入：

- `route:m4_remote_target2 = node-2`

然后从本节点客户端发送 P2P，实际返回：

- `result_msg = remote forwarding not enabled`

这说明远端分支已经真实生效，而不是只存在于代码结构里。

M4 现在可以视为已完成。

已经具备：

- Redis 路由表
- 用户在线状态中心
- 节点存活上报
- 本地 / 远端分支

- 集成测试与 Benchmark 体系

这些属于 M5 及之后的工作。

## 8. 面试讲解建议

讲 M4 时，建议不要只说“我加了 Redis 路由表和限流器”。

更好的表达方式是：

1. 单节点 IM 走向多节点，先要解决用户在线在哪个节点。
2. 所以我先把 Redis 从 session 存储升级成状态中心。
3. 然后在消息入口处显式区分本地投递和远端投递。
4. 再补限流、熔断和连接治理，让系统知道何时该拒绝、何时该降级。
5. 最后通过 Redis key 验证和远端分支验证，证明结构真实生效。

这样能把 M4 讲成一次典型的“从单机可用走向分布式可演进”的升级过程。
