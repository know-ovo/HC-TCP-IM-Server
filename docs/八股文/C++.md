## 一、C++基础语法与核心关键字

### 1. const 关键字（面试100%问）

#### 核心定义

const 是**只读限定符**，用于在编译期强制约束“不可修改”的语义，提升代码安全性、可读性，同时帮助编译器做优化。

#### 全场景用法与底层原理

| 用法场景       | 语法示例                                                                                                        | 核心说明                                                                                                                                                   | 面试考点                                                                                                                                                                                            |
| -------------- | --------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 修饰普通变量   | `const int a = 10;`                                                                                           | 变量只读，初始化后不可修改；编译器会将其放入常量区，而非栈区（除非取地址）                                                                                 | 1. const变量必须初始化 `<br>`2. 能否用 `const int`作为数组长度？C++11后可以，C语言不行 `<br>`3. `#define`和 `const`的区别：宏是预处理替换，无类型检查；const是编译期类型安全，可调试      |
| 修饰指针       | 1. 指向常量的指针：`const int* p;<br>`2. 指针常量：`int* const p;<br>`3. 双重const：`const int* const p;` | 1. 左定值，右定向：`*`左边const → 指向的内容不可改；`*`右边const → 指针本身不可改 `<br>`2. 底层：const修饰的是指针的访问权限，不改变内存本身的属性 | 面试高频手写：区分四种指针写法的含义，90%的人会搞混                                                                                                                                                 |
| 修饰函数参数   | `void func(const std::string& str);`                                                                          | 防止参数在函数内被意外修改；传const引用避免拷贝开销，是C++传参的最佳实践                                                                                   | 1. 为什么传值加const无意义？`<br>`2. 非const引用不能绑定临时对象，const引用可以                                                                                                                   |
| 修饰成员函数   | `class A { void func() const; };`                                                                             | const成员函数内**不能修改任何非静态成员变量**，也不能调用非const成员函数；底层会给this指针加const限定（`const A* this`）                           | 1. const对象只能调用const成员函数 `<br>`2. 成员函数的const重载：`void func();`和 `void func() const;`是合法重载 `<br>`3. mutable关键字：突破const限制，mutable修饰的变量可在const函数内修改 |
| 修饰函数返回值 | `const std::string getStr();`                                                                                 | 防止返回值被修改，多用于返回自定义对象，避免 `getStr() = "xxx"`的非法操作                                                                                | 内置类型返回值加const无意义，编译器会忽略                                                                                                                                                           |

#### 高频补充：const vs constexpr

- const：**运行期只读**，初始化时机可以是运行时（比如 `const int a = rand();`）
- constexpr：**编译期常量**，必须在编译期就能确定值，用于编译期优化、常量表达式、模板参数

### 2. static 关键字

#### 核心定义

static 核心作用是**改变生命周期/作用域**，在C和C++中有不同的扩展用法。

#### 全场景用法

##### （1）C语言原生用法

1. **局部静态变量**：`void func() { static int a = 0; }`

   - 生命周期：程序整个运行期，而非函数栈帧
   - 初始化：**只在第一次进入函数时执行一次**，后续调用不会重复初始化
   - 内存位置：全局/静态存储区，而非栈区
   - 面试考点：局部静态变量的线程安全？C++11后，局部静态变量的初始化是线程安全的（编译器保证加锁），是单例模式的最佳实现（魔法静态）
2. **全局静态变量/静态函数**：`static int g_a = 10;` `static void func() {}`

   - 作用域：**仅限制在当前源文件内**，其他文件无法访问，不会和其他文件的同名符号冲突
   - 面试考点：static全局变量和普通全局变量的区别？普通全局变量是全程序可见，static全局变量仅本文件可见

     (别的文件 #include 也没用，仍然不能直接访问。)(为了避免 **命名冲突** ！)

##### （2）C++面向对象扩展用法

1. **类静态成员变量**

   ```cpp
   class A {
       static int a; // 声明
   };
   int A::a = 10; // 必须在类外定义初始化
   ```

   - 不属于任何对象，属于整个类，所有对象共享同一份数据
   - 生命周期：程序整个运行期
   - 内存位置：全局/静态存储区，不占用对象的内存空间
   - 面试考点：静态成员变量为什么必须类外初始化？因为静态成员属于类，不属于对象，对象构造时不会初始化它，必须在全局作用域单独分配内存
2. **类静态成员函数**

   ```cpp
   class A {
       static void func();
   };
   ```

   - 没有this指针，只能访问类的静态成员变量/静态成员函数，不能访问普通成员
   - 调用方式：`A::func()`，也可以通过对象调用
   - 面试考点：静态成员函数能不能是虚函数？不能，因为虚函数依赖this指针和虚表，静态函数没有this指针

### 3. inline 内联函数

#### 核心定义

inline 是给编译器的**建议**，请求编译器在函数调用点直接展开函数体，消除函数调用的开销（栈帧创建、参数传递、返回地址跳转）。

#### 核心特性

1. 适用场景：短小、频繁调用的函数（比如get/set），函数体过大、有循环/递归的函数，编译器会拒绝内联
2. 必须定义在头文件中：因为内联是编译期展开，编译器必须在调用点看到完整的函数体
3. 允许多重定义：多个源文件包含同一个inline函数的头文件，不会报重定义错误，编译器会保证最终只有一份定义
4. 面试考点：
   - inline函数和宏函数的区别？
     1. 宏是预处理替换，无类型检查，极易出错；inline是编译期处理，有完整的类型安全检查
     2. 宏无法调试，inline函数可以调试
     3. 宏有运算符优先级问题，inline函数不会
   - 内联函数一定展开吗？不一定，只是编译器的建议，debug模式下默认不会内联，编译器会根据优化等级、函数体大小决定是否展开
   - 虚函数可以inline吗？语法上可以写，但运行时多态的场景下，编译器无法确定调用哪个函数，无法展开；只有非多态的直接调用（对象.函数()）才可能展开

### 4. volatile 关键字

#### 核心定义

volatile 告诉编译器：**这个变量的值可能会被意想不到的方式改变**，禁止编译器对这个变量的读写做优化。

#### 核心使用场景

1. 多线程共享的变量（注意：volatile不能保证线程安全，只能防止编译器优化掉读写）
2. 硬件寄存器映射的内存地址（驱动开发）
3. 中断服务程序中修改的变量

#### 面试高频考点

- volatile和const可以同时修饰一个变量吗？可以，比如只读的硬件寄存器，const表示程序不能修改，volatile表示值可能被硬件改变
- volatile指针的用法？和const完全一致，`volatile int* p`表示指向的内容是易变的，`int* volatile p`表示指针本身是易变的
- volatile能保证原子性吗？不能，原子性需要std::atomic，volatile只禁止编译器优化，不保证CPU指令的原子性

### 5. 其他高频关键字

#### （1）extern

- 核心作用：声明外部符号，告诉编译器这个变量/函数在其他源文件中定义，链接时去寻找
- 高频用法：`extern "C"`，告诉编译器按照C语言的规则进行名字修饰（name mangling），解决C++和C代码的混合调用问题

#### （2）sizeof

- 不是函数，是编译期运算符，计算类型/变量占用的内存字节数
- 面试考点：sizeof对空类的结果是1，编译器会给空类分配1字节的占位符，保证不同对象有不同的内存地址
- 数组名做sizeof参数时，返回整个数组的大小；数组名传参退化为指针，sizeof返回指针大小

#### （3）typedef vs using

- typedef：给类型起别名，`typedef std::vector<int> IntVec;`
- using（C++11）：类型别名，比typedef更强大，支持模板别名
  ```cpp
  template<typename T>
  using Vec = std::vector<T>; // 模板别名，typedef做不到
  Vec<int> intVec;
  ```

## 二、C++面向对象编程（OOP）核心原理

### 1. 面向对象四大特性

#### （1）封装

把数据和操作数据的方法绑定在一起，隐藏内部实现细节，只对外暴露必要的接口。

- 实现：通过访问控制符（public/protected/private）
- 核心价值：降低耦合、提升代码安全性、隔离变化

#### （2）继承

子类复用父类的属性和方法，实现代码复用和扩展。

- 访问控制：public继承（is-a关系，最常用）、protected继承、private继承
- 面试考点：
  - 子类的构造/析构顺序：构造先父类→再成员对象→再子类自身；析构完全相反
  - 父类的private成员，子类能访问吗？不能，只能通过父类的public/protected接口访问
  - 同名隐藏：子类的同名函数会屏蔽父类的所有同名函数，哪怕参数不同，不是重载

#### （3）多态（面试深挖重灾区）

同一个行为，不同的对象有不同的表现形式，分为**编译时多态**和**运行时多态**。

##### ① 编译时多态（静态多态）

编译期就确定调用的函数，包括：

- 函数重载
- 模板泛型编程
- 运算符重载

##### ② 运行时多态（动态多态，核心）

**运行时才确定调用的函数版本**，是OOP的核心，面试100%深挖。

###### 实现条件（三要素）

1. 基类必须有**虚函数**（virtual修饰）
2. 子类必须**重写（override）**基类的虚函数
3. 通过**基类的指针/引用**调用虚函数

###### 底层实现原理：虚函数表（vtable）+ 虚指针（vptr）

1. **虚函数表**：

   - 一个类只要有虚函数，编译器就会为这个类生成一张**虚函数表**，是一个函数指针数组，存储了类中所有虚函数的地址
   - 子类重写了哪个虚函数，就会在虚表中用子类的函数地址覆盖基类的地址；没有重写的，保留基类的地址
   - 虚表属于类，所有对象共享同一张虚表，存储在只读数据段
2. **虚指针（vptr）**：

   - 每个有虚函数的类的对象，内存开头都会有一个**虚指针**，指向所属类的虚函数表
   - vptr占用对象的内存空间，32位系统占4字节，64位占8字节
   - 编译器会在对象的构造函数中，完成vptr的初始化，指向正确的虚表
3. **虚函数调用过程**：

   ```cpp
   class Base {
   public:
       virtual void func() { cout << "Base"; }
   };
   class Derived : public Base {
   public:
       void func() override { cout << "Derived"; }
   };
   int main() {
       Base* p = new Derived();
       p->func(); // 调用Derived::func()
   }
   ```

   调用过程：

   1. 取p指向的对象的内存首地址，拿到vptr
   2. 通过vptr找到对应的虚函数表
   3. 从虚表中找到func()的地址
   4. 调用对应的函数

###### 面试高频深挖问题

1. **构造函数可以是虚函数吗？为什么？**
   不可以。因为虚函数调用依赖vptr，而vptr是在构造函数执行过程中才初始化完成的，构造函数执行时，对象还没完全构造好，vptr还没就绪，无法实现虚函数的多态调用。
2. **析构函数为什么建议声明为虚函数？**
   防止内存泄漏。当用基类指针指向子类对象时，如果析构函数不是虚函数，delete基类指针时，只会调用基类的析构函数，不会调用子类的析构函数，导致子类的资源无法释放，造成内存泄漏。
   声明为虚函数后，delete时会触发多态，先调用子类析构，再调用基类析构，完整释放资源。
3. **内联函数、静态函数、构造函数可以是虚函数吗？**
   都不可以。

   - 内联函数需要编译期确定展开，虚函数是运行时确定，矛盾
   - 静态函数没有this指针，而虚函数依赖this指针访问vptr，无法实现
   - 构造函数执行时vptr还没初始化完成
4. **什么是重写（override）、重载、隐藏？**

   | 概念 | 定义                                                                                 | 作用域         |
   | ---- | ------------------------------------------------------------------------------------ | -------------- |
   | 重载 | 同一作用域内，函数名相同，参数列表（个数/类型/顺序）不同，与返回值无关               | 同一个类内     |
   | 重写 | 子类重写基类的**虚函数**，函数名、参数列表、返回值必须完全一致（协变返回除外） | 基类和子类之间 |
   | 隐藏 | 子类的同名函数屏蔽了基类的所有同名函数，无论参数是否相同，无论是否是虚函数           | 基类和子类之间 |
5. **C++11 override 和 final 关键字的作用？**

   - override：显式声明这个函数是重写基类的虚函数，如果函数签名和基类不匹配，编译器会直接报错，避免手滑写错函数名/参数导致的隐藏问题
   - final：修饰类时，这个类不能被继承；修饰虚函数时，这个函数不能被子类重写

#### （4）抽象

隐藏复杂的实现细节，只对外暴露抽象的接口，通过纯虚函数实现。

- 纯虚函数：`virtual void func() = 0;`
- 抽象类：包含纯虚函数的类，不能实例化，只能被继承，子类必须重写所有纯虚函数才能实例化

### 2. 继承的高级问题

#### （1）菱形继承（钻石继承）的问题

```cpp
class A { public: int a; };
class B : public A {};
class C : public A {};
class D : public B, public C {};
```

问题：

1. 数据冗余：D的对象中会有两份A的成员变量（B继承的和C继承的）
2. 二义性：`D d; d.a = 10;` 编译器不知道访问的是B::a还是C::a，直接报错

#### （2）解决方案：虚继承

```cpp
class B : virtual public A {};
class C : virtual public A {};
```

- 原理：虚继承让共享的基类A在最终子类D中只保留一份实例，解决冗余和二义性
- 底层实现：编译器会给B和C添加一个虚基表指针，指向虚基表，虚基表中记录了共享基类A的内存偏移量
- 面试考点：虚继承的开销？增加了指针的内存占用，访问成员变量需要通过偏移量计算，有性能开销，非必要不使用

## 三、C++内存管理（面试核心重灾区）

### 1. C++内存分区（6大区域）

| 内存分区        | 存储内容                     | 生命周期                         | 管理方式                               | 面试考点                                                                                                 |
| --------------- | ---------------------------- | -------------------------------- | -------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| 栈区（Stack）   | 局部变量、函数参数、返回值   | 函数调用时创建，函数结束自动释放 | 编译器自动管理                         | 栈空间大小有限（Windows默认1MB，Linux默认8MB），栈溢出（Stack Overflow）的原因：递归过深、大数组局部变量 |
| 堆区（Heap）    | 动态分配的内存（new/malloc） | 手动分配，手动释放               | 程序员手动管理，忘记释放会造成内存泄漏 | 堆是向上增长，栈是向下增长；malloc分配的内存，底层有额外的空间存储内存块大小、校验信息                   |
| 自由存储区      | new/delete分配的内存         | 同堆区                           | 底层是基于堆区实现的，C++专属          | 自由存储区是C++的抽象概念，堆是操作系统的概念，new底层调用malloc时，自由存储区就是堆区                   |
| 全局/静态存储区 | 全局变量、静态变量           | 程序整个运行期                   | 编译器自动管理，程序结束释放           | 初始化的全局/静态变量和未初始化的分开存储，未初始化的会被默认初始化为0                                   |
| 常量存储区      | 字符串常量、const全局常量    | 程序整个运行期                   | 只读，不可修改，修改会触发段错误       | 面试坑：`char* p = "abc"; *p = 'd';` 会崩溃，因为"abc"在常量区，只读                                   |
| 代码区          | 程序的二进制可执行代码       | 程序运行期                       | 只读，不可修改                         | 存放CPU执行的机器指令                                                                                    |

### 2. new/delete 与 malloc/free 深度对比

#### （1）malloc/free 基础

- malloc/free是C标准库函数，只负责**内存的分配和释放**，不负责对象的构造和析构
- 用法：`void* p = malloc(100);` 分配100字节内存，失败返回NULL；`free(p)` 释放内存
- 底层：操作系统的系统调用（Linux下是brk/mmap）

#### （2）new/delete 底层实现

new/delete是C++运算符，不是函数，分为两步操作：

1. **new的底层**：
   1. 调用 `operator new` 标准库函数，分配内存（底层调用malloc）
   2. 调用对象的**构造函数**，初始化内存
   3. 返回对象的类型安全指针
2. **delete的底层**：
   1. 调用对象的**析构函数**，清理资源
   2. 调用 `operator delete` 标准库函数，释放内存（底层调用free）

#### （3）new[]/delete[]

- new[]：先分配内存，再对数组中的每个元素依次调用构造函数
- delete[]：先对数组中的每个元素依次调用析构函数，再释放内存
- 面试坑：`new int[10]` 用 `delete` 释放没问题，因为内置类型没有析构函数；但 `new A[10]` 必须用 `delete[]` 释放，否则只会调用第一个元素的析构函数，造成内存泄漏

#### （4）核心区别全对比

| 对比维度     | new/delete                           | malloc/free             |
| ------------ | ------------------------------------ | ----------------------- |
| 类型         | C++运算符                            | C标准库函数             |
| 构造/析构    | 自动调用                             | 不处理，只分配释放内存  |
| 返回值       | 类型安全的对应类型指针               | void*，需要强制类型转换 |
| 失败处理     | 抛出std::bad_alloc异常               | 失败返回NULL            |
| 分配大小     | 编译器根据类型自动计算               | 需要手动指定字节数      |
| 重载         | 支持重载operator new/operator delete | 不可重载                |
| 内存分配位置 | 自由存储区                           | 堆区                    |
| 数组处理     | new[]/delete[] 专门处理数组          | 数组需要手动计算大小    |

#### （5）placement new 定位new

- 作用：在已经分配好的内存上构造对象，不分配新的内存
- 用法：`new (p) A();` 在p指向的内存上构造A对象
- 场景：内存池、高频对象创建，避免频繁分配释放内存的开销
- 注意：必须手动调用析构函数 `p->~A();`，不能用delete释放内存，因为内存不是new分配的

### 3. 内存问题核心考点

#### （1）内存泄漏

- 定义：动态分配的内存，使用完后没有释放，导致程序运行期内存占用持续上涨，最终耗尽内存
- 常见原因：
  1. new/malloc分配的内存，忘记delete/free
  2. 基类析构函数不是虚函数，delete基类指针时子类析构没调用
  3. 异常安全问题，函数抛异常后，delete语句没执行
  4. 智能指针循环引用
- 排查工具：Linux下Valgrind、AddressSanitizer（ASAN），Windows下DebugDiag

#### （2）野指针 vs 悬空指针

- 野指针：指向**随机、非法、未初始化**内存的指针，比如未初始化的局部指针变量
- 悬空指针：指向**已经释放的内存**的指针，比如delete后的指针，内存已经还给系统，但指针还指向原来的地址
- 避免方法：
  1. 指针初始化时置为NULL/nullptr
  2. 内存释放后，立即将指针置为nullptr
  3. 不要返回局部变量的地址
  4. 优先使用智能指针

#### （3）内存碎片

- 定义：内存中出现大量不连续的小空闲块，无法被分配使用，导致内存利用率下降
- 分类：
  1. 内部碎片：分配的内存块中，未被使用的部分（比如内存池的固定块分配）
  2. 外部碎片：频繁分配释放不同大小的内存，导致空闲内存被分割成小碎片
- 解决方法：内存池、对象池、使用连续内存的容器（vector）、减少频繁的小内存分配释放

## 四、现代C++核心特性（C++11~C++20，面试必问）

### 1. 智能指针（面试100%问，项目高频使用）

C++11引入的RAII风格的智能指针，用于自动管理动态内存，解决内存泄漏、野指针问题。

#### （1）四种智能指针全解析

##### ① std::unique_ptr 独占智能指针

- 核心特性：**独占所有权**，同一时间只能有一个unique_ptr指向同一个对象，不支持拷贝，只支持移动
- 底层实现：一个裸指针的封装，无额外开销，和裸指针性能几乎一致
- 核心用法：
  ```cpp
  std::unique_ptr<A> p1 = std::make_unique<A>(); // C++14推荐，异常安全
  std::unique_ptr<A> p2 = std::move(p1); // 所有权转移，p1变为空
  ```
- 面试考点：
  1. 为什么unique_ptr不支持拷贝？因为独占所有权，拷贝会导致两个unique_ptr指向同一个对象，析构时重复释放
  2. unique_ptr可以作为函数返回值吗？可以，编译器会做返回值优化（RVO/NRVO），或者自动触发移动构造
  3. 自定义删除器：unique_ptr支持自定义删除器，比如管理文件指针、socket句柄
     ```cpp
     auto deleteFunc = [](sslocal://flow/file_open?url=FILE%2A+fp&flow_extra=eyJsaW5rX3R5cGUiOiJjb2RlX2ludGVycHJldGVyIn0=) { fclose(fp); };
     std::unique_ptr<FILE, decltype(deleteFunc)> fp(fopen("test.txt", "r"), deleteFunc);
     ```

##### ② std::shared_ptr 共享智能指针

- 核心特性：**共享所有权**，多个shared_ptr可以指向同一个对象，通过**引用计数**管理生命周期，最后一个shared_ptr析构时，自动释放对象
- 底层实现：包含两个指针，一个指向对象，一个指向**控制块**
  控制块中存储：引用计数、弱引用计数、删除器、分配器
- 引用计数：
  - 线程安全：引用计数的增减是**原子操作**，无锁，多线程下安全
  - 线程不安全：shared_ptr对象本身的读写不是线程安全的，多线程同时修改同一个shared_ptr需要加锁
- 核心用法：
  ```cpp
  std::shared_ptr<A> p1 = std::make_shared<A>(); // 推荐，一次分配内存，性能更好
  std::shared_ptr<A> p2 = p1; // 拷贝，引用计数+1
  std::cout << p1.use_count() << std::endl; // 输出2，引用计数
  ```
- 面试高频考点：
  1. make_shared 相比直接new的优势？
     - 一次内存分配：对象和控制块在同一块连续内存中分配，而new会分两次分配，减少内存碎片，提升性能
     - 异常安全：避免函数参数求值时，new之后抛异常导致的内存泄漏
  2. shared_ptr的线程安全？
     引用计数的增减是线程安全的；但多个线程同时修改同一个shared_ptr对象（比如赋值、reset）不是线程安全的，需要加锁；多个线程读同一个shared_ptr是安全的
  3. 为什么不能用同一个裸指针初始化多个shared_ptr？
     会生成多个独立的控制块，每个都有自己的引用计数，最后会导致重复释放，内存崩溃

##### ③ std::weak_ptr 弱智能指针

- 核心特性：**不拥有所有权**，不会增加引用计数，用来辅助shared_ptr，解决循环引用问题
- 核心用法：

  ```cpp
  std::shared_ptr<A> p1 = std::make_shared<A>();
  std::weak_ptr<A> wp = p1; // 引用计数不变
  std::shared_ptr<A> p2 = wp.lock(); // 升级为shared_ptr，成功则引用计数+1，失败返回空
  wp.expired(); // 判断指向的对象是否已经被释放
  ```
- 核心作用：解决shared_ptr的循环引用问题
  循环引用场景：

  ```cpp
  class A { public: std::shared_ptr<B> b; };
  class B { public: std::shared_ptr<A> a; };
  int main() {
      auto pA = std::make_shared<A>();
      auto pB = std::make_shared<B>();
      pA->b = pB;
      pB->a = pA;
      // 函数结束，pA和pB析构，引用计数都是1，永远不会减到0，内存泄漏
  }
  ```

  解决方案：把其中一个shared_ptr换成weak_ptr，不会增加引用计数，函数结束后引用计数正常减到0，内存释放。

##### ④ std::auto_ptr

- C++98的智能指针，C++11废弃，C++17移除
- 核心问题：拷贝构造时会转移所有权，导致原对象变为空，极易出错，比如容器中存储auto_ptr会导致未定义行为

#### （2）enable_shared_from_this

- 核心作用：在类的成员函数中，安全获取指向自身的shared_ptr
- 用法：类继承 `std::enable_shared_from_this<类名>`，成员函数中调用 `shared_from_this()`获取自身的shared_ptr
- 底层原理：类内部有一个weak_ptr，指向自身的控制块，shared_from_this()会通过weak_ptr升级为shared_ptr
- 面试坑：不能在构造函数中调用shared_from_this()，因为构造函数执行时，对象还没被shared_ptr接管，weak_ptr还没初始化，会抛异常

### 2. 右值引用、移动语义、完美转发（面试深挖重灾区）

#### （1）左值 vs 右值

- **左值**：可以取地址、有名字、生命周期长的表达式，比如变量、函数返回的左值引用
- **右值**：不能取地址、没有名字、临时对象，生命周期只在当前表达式，比如字面量、函数返回的临时对象、std::move的结果
- C++11细分：
  - 纯右值：字面量、临时对象
  - 将亡值：即将被销毁、可以被移动的对象，比如std::move的返回值

#### （2）右值引用

- 定义：用 `&&`表示，绑定到右值上的引用，延长右值的生命周期
- 语法：`int&& rref = 10;` 把临时值10绑定到右值引用rref上，生命周期延长到rref的作用域结束
- 核心作用：实现**移动语义**和**完美转发**

#### （3）移动语义

- 核心定义：**转移对象的资源所有权**，而不是拷贝资源，把一个对象的内存资源“偷”过来给另一个对象，避免昂贵的内存拷贝，大幅提升性能
- 实现：移动构造函数、移动赋值运算符
  ```cpp
  class String {
  private:
      char* data;
      size_t len;
  public:
      // 拷贝构造：深拷贝，开销大
      String(const String& other) {
          len = other.len;
          data = new char[len + 1];
          memcpy(data, other.data, len + 1);
      }
      // 移动构造：转移资源，O(1)开销
      String(String&& other) noexcept {
          data = other.data; // 偷指针
          len = other.len;
          // 把原对象置空，防止析构时释放资源
          other.data = nullptr;
          other.len = 0;
      }
      ~String() { delete[] data; }
  };
  ```
- std::move：本质是**类型转换**，把左值强制转换为右值引用，触发移动语义，本身不移动任何东西
- 面试考点：
  1. 移动构造函数为什么要加noexcept？
     告诉编译器这个移动构造不会抛异常，容器（比如vector）扩容时会优先使用移动构造，而不是拷贝构造，提升性能；如果不加noexcept，编译器会认为移动构造可能抛异常，扩容时会用拷贝构造保证异常安全
  2. 哪些场景会触发移动语义？
     函数返回局部对象、用std::move转换后的对象、临时对象赋值/初始化

#### （4）完美转发

- 核心定义：在函数模板中，把参数**保持原有的值类型（左值/右值）**不变，转发给另一个函数，不会因为转发变成左值
- 实现基础：
  1. 引用折叠规则：模板参数推导时，`T&&` 可以接受左值和右值，左值传入会推导为左值引用，右值传入会推导为右值引用
  2. std::forward `<T>`()：根据模板参数T的类型，把参数转换为对应的左值/右值引用
- 用法示例：
  ```cpp
  // 目标函数
  void func(int& a) { cout << "左值引用"; }
  void func(int&& a) { cout << "右值引用"; }

  // 完美转发模板
  template<typename T>
  void forwardFunc(T&& t) {
      func(std::forward<T>(t)); // 保持t的左值/右值属性不变
  }

  int main() {
      int a = 10;
      forwardFunc(a); // 传入左值，调用左值版本
      forwardFunc(10); // 传入右值，调用右值版本
  }
  ```
- 核心应用场景：可变参数模板、线程池任务提交、emplace_back、智能指针的make_shared等

### 3. RAII 资源管理（C++核心编程思想）

#### 核心定义

**RAII（Resource Acquisition Is Initialization）资源获取即初始化**，是C++管理资源的核心思想。

- 原理：利用对象的生命周期管理资源，在对象的构造函数中获取资源，在对象的析构函数中释放资源
- 核心价值：自动管理资源，无论程序正常返回还是异常退出，对象都会被析构，资源一定会被释放，避免资源泄漏
- 典型应用：智能指针、std::lock_guard/std::unique_lock、文件句柄封装、socket封装

### 4. lambda 表达式

C++11引入的匿名函数，用于简化代码，实现闭包。

#### 语法

`[捕获列表](sslocal://flow/file_open?url=%E5%8F%82%E6%95%B0%E5%88%97%E8%A1%A8&flow_extra=eyJsaW5rX3R5cGUiOiJjb2RlX2ludGVycHJldGVyIn0=) mutable 异常说明 -> 返回值类型 { 函数体 }`

#### 核心组成

1. **捕获列表**：捕获外部作用域的变量，供lambda内部使用
   - `[]`：不捕获任何变量

   - `[&]`：引用捕获所有外部变量

2. **参数列表**：和普通函数的参数列表一致
3. **mutable**：值捕获的变量，默认在lambda内是只读的，加mutable可以修改
4. **返回值类型**：可以省略，编译器自动推导

#### 面试考点

- lambda的底层实现：编译器会为lambda生成一个唯一的匿名类，重载 `operator()`，捕获的变量会作为这个类的成员变量
- 无捕获的lambda可以隐式转换为函数指针，有捕获的不行

### 5. 其他高频现代C++特性

#### （1）auto 类型推导

- 编译器在编译期自动推导变量的类型，简化代码，尤其是复杂的迭代器类型
- 面试考点：auto推导的规则
  1. auto不会推导出引用，除非显式指定 `auto&`
  2. auto会忽略顶层const，保留底层const，比如 `const int a = 10; auto b = a;` b的类型是int，不是const int
  3. 函数返回值auto：C++14支持，编译器根据return语句推导返回值类型

#### （2）decltype

- 推导表达式的类型，常用于模板编程、类型萃取
- 用法：`decltype(a) b = 10;` 推导a的类型，定义b
- 面试考点：decltype和auto的区别？auto用于定义变量，decltype用于推导表达式类型，常用于模板元编程

#### （3）constexpr

- 编译期常量/编译期函数，在编译期就能计算出结果，提升性能，用于模板参数、数组长度等需要编译期常量的场景
- 和const的区别：const是运行期只读，constexpr是编译期常量

#### （4）nullptr

- C++11引入的空指针常量，替代NULL
- 解决NULL的二义性问题：NULL本质是0，函数重载时 `void func(int)`和 `void func(int*)`，传NULL会匹配到int版本，传nullptr会匹配到指针版本

## 五、STL容器与算法底层实现

### 1. 序列式容器

#### （1）std::vector 动态数组

- 底层结构：**连续的动态数组**，内存完全连续
- 扩容机制：
  - GCC/Linux：2倍扩容，当前容量满了之后，重新分配2倍大小的内存，把原数据拷贝过去，释放旧内存
  - MSVC/Windows：1.5倍扩容
- 核心特性：
  - 随机访问O(1)，尾部插入删除O(1)，中间插入删除O(n)
  - 内存连续，CPU缓存命中率高，性能好
- 面试高频考点：
  1. reserve() 和 resize() 的区别？
     - reserve()：只扩容，改变容量（capacity），不改变大小（size），不构造对象，提前分配内存避免频繁扩容
     - resize()：改变大小（size），如果新size大于capacity会扩容，同时会构造/析构对象
  2. emplace_back() 和 push_back() 的区别？
     - push_back()：先构造临时对象，再拷贝/移动到容器末尾
     - emplace_back()：直接在容器末尾的内存上构造对象，避免临时对象的拷贝/移动，性能更好，C++11推荐
  3. vector迭代器失效的场景？
     - 扩容：所有迭代器全部失效，因为内存地址变了
     - 中间插入/删除：插入/删除位置之后的所有迭代器失效
     - 尾部插入：如果扩容则全部失效，不扩容则尾后迭代器失效
     - 尾部删除：只有尾迭代器失效

#### （2）std::list 双向链表

- 底层结构：**双向循环链表**，每个节点包含数据、前驱指针、后继指针，节点内存不连续
- 核心特性：
  - 不支持随机访问，访问元素O(n)
  - 任意位置插入删除O(1)，只需要修改指针
  - 不会迭代器失效，只有被删除的节点的迭代器失效，其他都有效
- 面试考点：list和vector的选型？
  - 频繁随机访问，用vector
  - 频繁中间插入删除，用list

#### （3）std::deque 双端队列

- 底层结构：**分段连续的数组 + 中控器**，中控器是一个指针数组，每个指针指向一段固定大小的缓冲区（数组）
- 核心特性：
  - 头尾插入删除O(1)，中间插入删除O(n)
  - 支持随机访问，但性能比vector差
  - 不会像vector那样全量扩容，只需要新增缓冲区，修改中控器，不会拷贝原数据
- 应用场景：std::stack和std::queue的默认底层容器

### 2. 关联式容器

#### （1）std::map / std::set

- 底层结构：**红黑树**（平衡二叉搜索树）
- 核心特性：
  - map是键值对，set只有键，键唯一，不可修改
  - 元素按键自动排序，默认升序
  - 插入、删除、查找都是O(logn)
- 面试考点：
  1. 为什么用红黑树，不用AVL树？
     AVL树是严格平衡二叉树，左右子树高度差不超过1，插入删除需要频繁旋转，性能差；红黑树是弱平衡，旋转次数更少，插入删除性能更好，更适合频繁修改的场景，查找性能和AVL树差距极小
  2. 迭代器失效场景？只有被删除的节点的迭代器失效，其他都有效，因为红黑树的节点内存不连续，修改不会影响其他节点

#### （2）std::unordered_map / std::unordered_set

- 底层结构：**哈希表（开链法）**，数组+链表，C++11后链表长度超过8会转为红黑树
- 核心特性：
  - 无序，键唯一
  - 平均插入、删除、查找O(1)，最坏情况O(n)（哈希冲突严重）
- 面试高频考点：
  1. 哈希冲突的解决方法？
     - 开链法（拉链法）：STL用的方案，哈希值相同的元素放在同一个链表中
     - 开放地址法：线性探测、二次探测、再哈希法
  2. 负载因子和rehash？
     - 负载因子 = 元素个数 / 桶的个数，默认最大负载因子是1
     - 当负载因子超过阈值时，会触发rehash：重新分配更大的桶数组，把所有元素重新哈希到新桶中，开销很大
  3. 迭代器失效场景？
     - 插入：rehash后所有迭代器失效，不rehash则有效
     - 删除：只有被删除的元素的迭代器失效，其他有效
  4. unordered_map和map的选型？
     - 不需要排序，追求极致查找性能，用unordered_map
     - 需要有序，保证稳定的O(logn)性能，用map

### 4. STL算法

- 拷贝：copy、copy_if
- 替换：replace、replace_if

- 面试考点：sort的底层实现？
  GCC的std::sort是**内省排序（Introsort）**：
  1. 数据量大时，用快速排序
  2. 递归深度超过阈值时，转为堆排序，避免快排最坏情况O(n²)
  3. 数据量小于阈值（16个元素）时，转为插入排序，小数组插入排序性能更好

## 六、C++多线程与并发编程

### 1. std::thread 线程

- C++11引入的跨平台线程库，底层封装了操作系统的线程API（Linux下pthread，Windows下CreateThread）
- 核心用法：
  ```cpp
  void func(int a) { cout << a; }
  int main() {
      std::thread t(func, 10); // 创建线程，执行func
      t.join(); // 等待线程执行完毕，阻塞主线程
      // t.detach(); // 分离线程，主线程不等待，线程在后台运行
  }
  ```
- 面试考点：
  1. join() 和 detach() 的区别？
     - join()：主线程阻塞，等待子线程执行完毕，回收线程资源，一个线程只能join一次
     - detach()：子线程和主线程分离，主线程不等待，子线程在后台运行，资源由系统回收，detach后不能再join
  2. 线程函数的参数传递？
     - 线程的参数会拷贝到线程的栈空间，即使是引用，也要用std::ref包裹，否则是值拷贝
     - 不能传递局部变量的引用给detach的线程，否则主线程结束后，局部变量被销毁，线程访问悬空指针

---

### 2. 互斥锁与锁管理

#### （1）std::mutex 互斥锁

- 保护临界区，同一时间只有一个线程能持有锁，保证线程安全
- 核心方法：lock() 加锁、unlock() 解锁、try_lock() 尝试加锁，失败立即返回
- 其他锁类型：
  - std::recursive_mutex：递归锁，同一个线程可以多次加锁，不会死锁
  - std::timed_mutex：带超时的锁，try_lock_for() 等待指定时间
  - std::recursive_timed_mutex：递归+超时锁

#### （2）RAII锁管理

##### ① std::lock_guard

- 构造时加锁，析构时自动解锁，无论函数正常返回还是抛异常，都会解锁，避免死锁
- 缺点：不能手动解锁，只能在析构时解锁，不支持超时

##### ② std::unique_lock

- 比lock_guard更灵活，支持手动lock/unlock、延迟加锁、超时加锁、转移锁所有权
- 缺点：比lock_guard有一点点额外开销
- 面试考点：lock_guard和unique_lock的选型？
  简单场景，只需要自动加解锁，用lock_guard，性能更好；需要灵活控制锁的场景，用unique_lock，比如和条件变量配合使用

---

### 3. std::condition_variable 条件变量

- 用于线程间的同步，一个线程等待条件成立，另一个线程通知条件成立，避免空轮询消耗CPU
- 核心方法：
  - wait()：阻塞等待，直到被通知，需要配合unique_lock使用
  - wait_for()/wait_until()：带超时的等待
  - notify_one()：唤醒一个等待的线程
  - notify_all()：唤醒所有等待的线程
- 面试高频考点：
  1. wait()为什么要配合unique_lock？
     wait()会在阻塞时自动解锁，被唤醒时自动加锁，需要锁的控制权，所以必须用unique_lock
  2. 什么是虚假唤醒？
     wait()可能在没有被notify的情况下返回，所以wait()必须放在循环里，每次醒来都要重新检查条件是否成立
     正确用法：
     ```cpp
     std::unique_lock<std::mutex> lock(mtx);
     cv.wait(lock,  { return !queue.empty(); }); // 循环检查条件
     ```

### 4. 原子操作 std::atomic

- 无锁的线程安全操作，CPU保证对原子变量的操作是不可分割的，不会被线程调度打断，比互斥锁轻量高效，适合计数器、标志位等简单场景
- 支持的类型：bool、int、long、指针等内置类型
- 核心用法：
  ```cpp
  std::atomic<int> count = 0;
  count++; // 原子自增，线程安全，无锁
  ```
- 面试考点：volatile和atomic的区别？
  volatile只禁止编译器优化，不保证原子性；atomic保证操作的原子性，线程安全

### 5. 内存序（memory_order）

C++11定义了6种内存序，用于控制多线程下的内存可见性和指令重排序，面试高频深挖。

| 内存序               | 作用                                                                                        | 场景                         |
| -------------------- | ------------------------------------------------------------------------------------------- | ---------------------------- |
| memory_order_relaxed | 松散序，只保证操作的原子性，不保证内存可见性和指令重排序                                    | 计数器，不需要同步的场景     |
| memory_order_acquire | 获取序，本线程中，acquire之后的读写指令不能重排到acquire之前；保证看到release线程的所有写入 | 配合release实现线程同步      |
| memory_order_release | 释放序，本线程中，release之前的读写指令不能重排到release之后；写入对acquire线程可见         | 配合acquire实现线程同步      |
| memory_order_acq_rel | acquire+release，同时满足两者的约束                                                         | 读写都需要同步的场景         |
| memory_order_seq_cst | 顺序一致性，最强约束，所有线程看到的所有操作顺序完全一致，默认内存序                        | 多线程复杂同步，保证绝对顺序 |
| memory_order_consume | 消费序，和数据依赖相关，极少使用                                                            | 无                           |

### 6. 死锁

#### 死锁四个必要条件

1. 互斥条件：资源同一时间只能被一个线程持有
2. 持有并等待：线程持有至少一个资源，同时请求其他被持有的资源
3. 不可剥夺：资源只能被持有者主动释放，不能被其他线程强行剥夺
4. 循环等待：线程之间形成循环等待资源的链

#### 避免死锁的方法

1. 按固定顺序加锁，所有线程都按同一个顺序申请锁
2. 避免嵌套加锁，一个线程最多持有一个锁
3. 使用超时锁，超过时间自动释放锁
4. RAII管理锁，自动解锁
5. std::lock() 一次性加多个锁，避免循环等待

### 7. 异步任务：std::future / std::promise / std::async

- std::async：启动一个异步任务，返回std::future，用于获取任务的结果
- std::promise：用于在一个线程中设置结果，另一个线程通过future获取结果
- std::future：获取异步任务的结果，get()会阻塞等待结果返回

## 七、C++模板与泛型编程

### 1. 模板基础

模板是C++实现泛型编程的核心，编译期多态，代码复用，类型安全。

#### （1）函数模板

```cpp
template<typename T>
T add(T a, T b) {
    return a + b;
}
```

编译器会根据传入的参数类型，实例化对应类型的函数。

#### （2）类模板

```cpp
template<typename T>
class Stack {
private:
    std::vector<T> data;
public:
    void push(const T& val) { data.push_back(val); }
    T pop() { T val = data.back(); data.pop_back(); return val; }
};
```

类模板必须显式指定模板参数，`Stack<int> s;`

### 2. 模板特化

#### （1）全特化

对特定的类型，提供特殊的实现

```cpp
template<>
class Stack<std::string> {
    // 针对string类型的特殊实现
};
```

#### （2）偏特化（部分特化）

对模板参数的一部分做限制，比如指针类型、引用类型

```cpp
template<typename T>
class Stack<T*> {
    // 针对所有指针类型的特殊实现
};
```

### 3. SFINAE

**替换失败不是错误（Substitution Failure Is Not An Error）**
模板参数推导时，如果替换失败，编译器不会报错，只会忽略这个模板重载，选择其他匹配的重载，是C++模板元编程的核心基础。

### 4. 可变参数模板

C++11引入，支持模板参数数量可变，用于泛型编程、完美转发、tuple等场景

```cpp
// 递归终止
void print() {}
// 可变参数模板
template<typename T, typename... Args>
void print(const T& first, const Args&... args) {
    cout << first << endl;
    print(args...); // 递归展开参数包
}
```

## 八、C++编译与链接过程

### 1. 程序从源码到可执行文件的四个阶段

#### （1）预处理阶段（.cpp → .i）

- 执行预处理器指令：#include 头文件展开、#define宏替换、条件编译处理
- 删除注释
- 生成预处理后的.i文件

#### （2）编译阶段（.i → .s）

- 编译器对预处理后的代码进行词法分析、语法分析、语义分析、优化
- 生成汇编代码.s文件

#### （3）汇编阶段（.s → .o）

- 汇编器把汇编代码翻译成机器指令，生成目标文件.o（Windows下.obj）
- 目标文件包含二进制机器码、符号表、重定位表、字符串表等

#### （4）链接阶段（.o → 可执行文件）

- 链接器把多个目标文件、静态库链接在一起，解决符号引用、重定位，生成最终的可执行文件
- 分为静态链接和动态链接

### 2. 静态链接 vs 动态链接

| 对比维度       | 静态链接                                     | 动态链接                                               |
| -------------- | -------------------------------------------- | ------------------------------------------------------ |
| 链接时机       | 编译期链接，把库的代码直接嵌入到可执行文件中 | 运行期链接，可执行文件只记录库的信息，运行时加载到内存 |
| 可执行文件大小 | 大，包含库的代码                             | 小，不包含库代码                                       |
| 运行依赖       | 无依赖，可独立运行                           | 依赖系统中存在对应的动态库                             |
| 升级           | 库升级需要重新编译程序                       | 库升级不需要重新编译，替换库文件即可                   |
| 性能           | 运行时无加载开销，性能好                     | 运行时需要加载库，有一点点开销                         |

### 3. 名字修饰（name mangling）

C++支持函数重载，编译器会对函数名进行修饰，把函数名、参数类型、命名空间等信息编码到最终的符号名中，区分不同的重载函数。

- C语言没有函数重载，名字修饰很简单，只在函数名前加下划线
- `extern "C"`的作用就是告诉编译器，按照C语言的规则进行名字修饰，实现C和C++的混合调用

### 4. 强符号与弱符号

- 强符号：函数、初始化的全局变量
- 弱符号：未初始化的全局变量、__attribute__((weak))修饰的符号
- 链接规则：
  1. 同名的强符号只能有一个，否则报重定义错误
  2. 有一个强符号和多个弱符号，选择强符号
  3. 多个弱符号，选择占用内存最大的那个

## 九、C++异常处理

### 1. 基本语法

```cpp
try {
    // 可能抛异常的代码
    throw std::runtime_error("出错了");
} catch (const std::exception& e) {
    // 捕获异常，处理
    cout << e.what() << endl;
} catch (...) {
    // 捕获所有异常
}
```

### 2. 异常安全

- 基本保证：异常发生后，程序资源不泄漏，对象处于有效状态
- 强保证：异常发生后，程序状态回滚到异常发生之前，要么成功，要么不变
- 不抛保证：函数绝对不会抛异常，用noexcept修饰

### 3. noexcept 关键字

- 告诉编译器，这个函数不会抛异常，编译器可以做优化
- 如果noexcept函数内抛了异常，程序会直接调用std::terminate()终止
- 移动构造函数加noexcept，容器扩容时会优先使用移动构造

### 4. RAII与异常

RAII是C++中保证异常安全的核心手段，无论函数是否抛异常，对象都会被析构，资源一定会被释放，避免资源泄漏。
