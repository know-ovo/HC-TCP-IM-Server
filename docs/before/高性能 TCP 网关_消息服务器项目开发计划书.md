# 一、项目概述

## 1.1 项目名称

HighConcurrency TCP Gateway / Message Server（高性能 TCP 网关 & 消息服务器）

## 1.2 项目定位

本项目是一套基于 C++17 开发的企业级高并发 TCP 网关/消息服务器，定位为「通用接入层网关 + 轻量级消息推送服务器」，可作为后端服务的统一接入入口，负责客户端连接管理、消息路由、鉴权校验、流量控制，同时支持点对点消息、广播消息等基础消息服务，适用于游戏后台、物联网设备接入、即时通讯等场景。

项目核心目标：打造一套「高并发、高稳定、可扩展、可部署」的工业级 TCP 网关，满足 C++ 后端求职项目的核心要求（可写进简历、可面试讲解、可量化性能）。

## 1.3 项目价值

- 技术价值：掌握 Reactor 主从多线程模型、epoll 高性能网络编程、自定义协议设计、线程池异步处理等 C++ 后端核心技术，贴合大厂实际开发场景。
- 求职价值：作为完整的企业级项目，可直接写入简历，面试时可详细讲解架构设计、技术难点、性能优化，区别于普通 Demo 项目，提升求职竞争力。
- 实用价值：可直接部署使用，支持万级并发连接，提供完整的接入层能力，可扩展为游戏网关、物联网网关等具体场景。

# 二、需求分析

## 2.1 核心需求（必须实现）

### 2.1.1 网络通信需求

- 支持 TCP 协议，基于 Reactor 主从多线程模型，实现高并发 IO 处理。
- 支持非阻塞 IO + epoll ET 边缘触发模式，提升网络 IO 性能。
- 支持万级并发客户端连接，单个客户端连接长时间保持（长连接）。
- 解决 TCP 粘包、半包问题，实现可靠的消息收发。

### 2.1.2 连接管理需求

- 支持客户端连接的建立、断开、异常处理，维护连接生命周期。
- 实现心跳保活机制，自动检测异常连接（超时、断开）并清理，避免资源泄漏。
- 支持连接状态监控（在线、离线、异常），可打印连接统计信息。
- 实现连接踢人机制，支持手动/自动踢除指定客户端。

### 2.1.3 协议与消息需求

- 设计自定义二进制应用层协议，包含长度、命令号、序列号、校验和、消息体，支持协议扩展。
- 实现协议编解码模块，支持消息的打包、解包，处理粘包、半包。
- 支持消息路由：点对点消息（客户端A→客户端B）、广播消息（服务器→所有在线客户端）。
- 支持消息重发（可选），确保消息可靠送达（针对重要消息）。

### 2.1.4 业务逻辑需求

- 实现客户端登录鉴权（简单 token 鉴权），拒绝非法连接。
- 实现基础消息转发逻辑，支持客户端之间的消息互通。
- 实现过载保护：当并发连接数/消息量超过阈值时，拒绝新连接或丢弃非重要消息，避免服务器崩溃。

### 2.1.5 工程化需求

- 采用 CMake 构建工程，支持跨平台（Linux 为主）。
- 集成结构化日志（spdlog），支持不同日志级别（DEBUG/INFO/WARN/ERROR），便于问题排查。
- 支持配置文件（ini/json），可配置端口、线程数、心跳超时时间、限流阈值等参数，无需重新编译。
- 实现优雅启动、优雅关闭，避免强制退出导致的连接泄漏、消息丢失。

## 2.2 扩展需求（可选实现，提升项目亮点）

- 支持 Protobuf 序列化，替换部分二进制协议，提升协议扩展性。
- 实现简单 RPC 调用能力，支持客户端调用服务器接口。
- 添加黑白名单功能，限制指定 IP/客户端的连接。
- 实现运行时监控：统计并发连接数、QPS、消息延迟、错误率等指标。
- 支持 Docker 部署，提供 Dockerfile，便于快速部署。

## 2.3 性能需求

- 并发连接数：支持 ≥ 10000 个长连接。
- 消息处理能力：QPS ≥ 100000（单条消息 128 字节）。
- 消息延迟：平均延迟 ≤ 10ms，峰值延迟 ≤ 50ms。
- 资源占用：CPU 占用 ≤ 30%（4 核 8G 服务器），内存占用 ≤ 500MB（10000 连接）。
- 稳定性：连续运行 72 小时无崩溃、无内存泄漏、无连接异常。

## 2.4 非功能需求

- 可扩展性：模块解耦，新增功能（如新增协议、新增业务逻辑）无需修改核心代码。
- 可维护性：代码规范、注释完整、结构清晰，便于后续修改和扩展。
- 可靠性：具备异常处理机制（如连接异常、消息解析失败、线程崩溃），避免服务器整体崩溃。
- 可测试性：提供测试用例、压测工具，便于验证性能和功能。

# 三、技术选型

## 3.1 开发语言与标准

语言：C++17（现代 C++ 特性，如智能指针、Lambda、std::thread、std::atomic 等，提升开发效率和代码安全性）。

## 3.2 网络编程技术

- IO 模型：Reactor 主从多线程模型（one loop per thread），主 Reactor 负责监听新连接，子 Reactor 负责处理已连接客户端的 IO 事件。
- IO 多路复用：epoll（Linux 平台，高性能，支持百万级文件描述符，ET 边缘触发模式）。
- IO 类型：非阻塞 IO（避免线程阻塞，提升并发处理能力）。
- 网络编程接口：原生 Linux socket API（不依赖第三方网络库，深入理解底层原理，面试更有优势）。

## 4.3 核心模块详细设计

### 4.3.1 基础工具模块（base）

- 日志模块（Log.h/Log.cpp）：封装 spdlog，提供 DEBUG/INFO/WARN/ERROR 四个级别日志，支持输出到控制台和文件，可通过配置文件设置日志级别和输出路径。
- 配置模块（Config.h/Config.cpp）：解析 ini 配置文件，提供端口、线程数、心跳超时、限流阈值等参数的读取接口。
- 线程池模块（ThreadPool.h/ThreadPool.cpp）：自定义实现，支持固定数量线程，线程安全队列传递任务，支持任务异步执行，优雅关闭。
- 线程安全队列（BlockingQueue.h）：模板类，支持多线程安全的入队、出队操作，基于 mutex + condition_variable 实现阻塞等待。
- 工具函数（Util.h/Util.cpp）：提供字节序转换、字符串处理、时间戳获取等常用工具函数。

### 4.3.2 网络核心模块（net）

- Channel 类（Channel.h/Channel.cpp）：封装文件描述符（fd）、IO 事件（可读、可写）和事件回调函数（读回调、写回调、错误回调），用于注册到 Epoll 中，处理 IO 事件。
- EpollPoller 类（EpollPoller.h/EpollPoller.cpp）：封装 epoll 相关系统调用（epoll_create、epoll_ctl、epoll_wait），负责管理所有 Channel，监听 IO 事件，将触发的事件分发给对应 Channel 处理。
- EventLoop 类（EventLoop.h/EventLoop.cpp）：Reactor 核心，包含一个 EpollPoller，负责循环等待 IO 事件，处理事件回调，管理 Channel 的生命周期，支持单线程循环（one loop per thread）。
- Acceptor 类（Acceptor.h/Acceptor.cpp）：负责监听新连接，封装 socket、bind、listen、accept 等操作，当有新连接到来时，创建 TcpConnection 对象，并将其注册到子 Reactor 中。
- TcpConnection 类（TcpConnection.h/TcpConnection.cpp）：封装客户端连接，管理连接的生命周期（连接建立、数据收发、连接断开），关联 Channel 和 EventLoop，处理读事件（接收数据）、写事件（发送数据）。
- TcpServer 类（TcpServer.h/TcpServer.cpp）：服务器总入口，管理主 Reactor 和子 Reactor 线程池，创建 Acceptor，管理所有 TcpConnection 对象，提供服务器启动、关闭接口。
- ConnectionManager 类（ConnectionManager.h/ConnectionManager.cpp）：管理所有客户端连接，提供连接的添加、删除、查询、遍历接口，用于心跳检查、广播消息等场景。

### 4.3.3 协议编解码模块（codec）

- 协议定义（Protocol.h）：定义自定义二进制协议结构，包含消息头（长度、命令号、序列号、校验和）和消息体。
- Codec 类（Codec.h/Codec.cpp）：实现协议编解码逻辑，包括消息打包（将消息体转换为二进制流）、消息解包（将二进制流转换为消息体）、粘包半包处理（基于消息头长度字段，缓存未完整接收的消息）。

### 4.3.4 业务逻辑模块（service）

- AuthService 类（AuthService.h/AuthService.cpp）：实现客户端登录鉴权逻辑，验证客户端发送的 token，拒绝非法连接。
- HeartbeatService 类（HeartbeatService.h/HeartbeatService.cpp）：实现心跳保活机制，定期检查客户端心跳消息，超时未发送心跳的客户端自动断开连接。
- MessageService 类（MessageService.h/MessageService.cpp）：实现消息路由、点对点转发、广播消息逻辑，处理客户端发送的消息，转发给目标客户端。
- OverloadProtectService 类（OverloadProtectService.h/OverloadProtectService.cpp）：实现过载保护，监控并发连接数和消息量，超过阈值时拒绝新连接或丢弃非重要消息。

### 4.3.5 应用入口模块

- server.cpp：项目入口函数，负责初始化日志、配置、线程池、TcpServer，启动服务器，处理信号（如 SIGINT、SIGTERM），实现优雅关闭。

## 4.4 核心流程设计

### 4.4.1 服务器启动流程

1. 初始化日志模块（读取配置文件，设置日志级别和输出路径）。
2. 初始化配置模块（读取端口、线程数、心跳超时等参数）。
3. 初始化线程池（根据配置的线程数创建工作线程）。
4. 创建 TcpServer 对象，初始化主 Reactor 和子 Reactor 线程池。
5. 启动 TcpServer，开始监听新连接。
6. 注册信号处理函数，监听 SIGINT、SIGTERM 信号，用于优雅关闭。
7. 阻塞等待信号，或进入主循环，直到收到关闭信号。

### 4.4.2 客户端连接流程

1. 客户端发起 TCP 连接请求。
2. 主 Reactor 中的 Acceptor 监听到底层 socket 可读事件，调用 accept 接收新连接，创建 client_fd。
3. 创建 TcpConnection 对象，封装 client_fd，设置读、写、错误回调。
4. 将 TcpConnection 关联的 Channel 注册到子 Reactor 的 Epoll 中，监听可读事件（等待客户端发送数据）。
5. ConnectionManager 添加该 TcpConnection，记录连接状态为“在线”。

### 4.4.3 消息收发流程

1. 客户端发送消息（二进制协议格式）。
2. 子 Reactor 的 Epoll 监听到底层 client_fd 可读事件，触发 Channel 的读回调。
3. TcpConnection 的读回调读取数据，将数据交给 Codec 模块进行解包，处理粘包、半包。
4. Codec 解包成功后，得到完整的消息体，将消息抛给线程池，异步执行业务逻辑。
5. 业务线程处理消息（如鉴权、消息转发），处理完成后，生成响应消息，交给 Codec 模块打包。
6. TcpConnection 将打包后的消息写入客户端，完成消息发送。

### 4.4.4 优雅关闭流程

1. 服务器收到 SIGINT 或 SIGTERM 信号。
2. 停止监听新连接，拒绝所有新的连接请求。
3. 关闭所有子 Reactor 的 Epoll，停止处理 IO 事件。
4. 关闭所有客户端连接，发送断开通知（可选）。
5. 停止线程池，等待所有正在执行的业务任务完成。
6. 关闭日志模块，释放所有资源。
7. 程序退出。

# 六、测试方案

## 6.1 测试目标

验证项目功能完整性、稳定性、性能达标情况，确保项目符合需求分析中的各项要求，为上线和求职展示提供可靠依据。

## 6.2 测试类型

### 6.2.1 功能测试

- 测试范围：所有核心功能（连接管理、消息收发、鉴权、心跳、消息路由、广播、过载保护）。
- 测试方法：编写测试用例，手动或自动执行，验证功能是否符合预期。
- 核心测试用例：
  - 客户端连接/断开测试：多个客户端同时连接、断开，验证连接管理正常。
  - 消息收发测试：客户端发送消息，服务器回显/转发，验证消息无丢失、无乱序。
  - 粘包半包测试：发送大量短消息/长消息，验证编解码模块能正确处理。
  - 鉴权测试：发送合法/非法 token，验证鉴权功能正常，非法连接被拒绝。
  - 心跳测试：客户端不发送心跳，验证服务器能自动断开超时连接。
  - 消息路由测试：客户端A发送消息给客户端B，验证客户端B能收到，其他客户端收不到。
  - 广播测试：服务器发送广播消息，验证所有在线客户端都能收到。
  - 过载保护测试：模拟超过阈值的并发连接/消息量，验证服务器能拒绝新连接或丢弃消息。

### 6.2.2 性能测试

- 测试范围：并发连接数、QPS、消息延迟、CPU/内存占用、稳定性。
- 测试工具：自定义压测客户端（模拟多客户端并发连接和消息发送）、wrk（辅助测试）。
- 测试场景：
  - 并发连接测试：逐步增加客户端连接数，直到达到 10000 连接，验证服务器稳定运行。
  - QPS 测试：10000 连接下，客户端持续发送消息，测试服务器 QPS 是否达到 100000+。
  - 延迟测试：记录消息从客户端发送到接收响应的平均延迟和峰值延迟。
  - 资源占用测试：监控服务器 CPU、内存占用，验证是否符合性能需求。
  - 稳定性测试：服务器连续运行 72 小时，监控是否有崩溃、内存泄漏、连接异常。

### 6.2.3 异常测试

- 测试范围：异常连接、异常消息、服务器异常退出、网络中断等场景。
- 核心测试用例：
  - 客户端异常断开：客户端强制退出，验证服务器能正确清理连接，无资源泄漏。
  - 异常消息测试：发送非法格式消息，验证服务器能正确处理，不崩溃。
  - 服务器异常退出：发送 SIGINT 信号，验证服务器能优雅关闭，无消息丢失、连接泄漏。
  - 网络中断测试：模拟网络中断，恢复后验证连接能否正常恢复。

## 6.3 测试环境

- 服务器环境：Ubuntu 22.04，4 核 8G，g++ 11.4.0，CMake 3.22。
- 客户端环境：Ubuntu 22.04（或 Windows WSL2），用于运行压测客户端和测试程序。
- 测试工具：gdb、strace、perf、tcpdump、wrk、自定义压测客户端。

## 6.4 测试结果验收标准

- 功能测试：所有核心功能测试用例通过率 100%，无功能异常。
- 性能测试：并发连接数 ≥ 10000，QPS ≥ 100000，平均延迟 ≤ 10ms，CPU 占用 ≤ 30%，内存占用 ≤ 500MB，连续运行 72 小时无崩溃。
- 异常测试：所有异常场景下，服务器能正常处理，不崩溃、无资源泄漏。
